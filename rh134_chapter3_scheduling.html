ℹ️Why Task Scheduling Matters

System administrators rarely sit at a terminal waiting to run commands at 3 AM. Task scheduling lets you automate maintenance windows, generate nightly reports, run backup jobs, and enforce security checks without manual intervention. Mastering both one-time (at) and recurring (cron) scheduling gives you full control over your system’s automated behavior.

2

Scheduling Deferred User Jobs

Sometimes you need to run one or more commands at a specific future time. A deferred job is a task scheduled to execute once, at a particular point in the future. Unlike recurring cron jobs, deferred jobs run exactly once and are automatically removed from the queue after execution.

A job that is queued to run one time at a specified future date and time. After it executes, it is removed from the job queue automatically.

Common Use Cases

Consider these real-world scenarios where a one-time future job is the right tool:

  • Overnight maintenance: A user queues a long-running database optimization script to run at 2 AM so it does not affect daytime performance.
  • Firewall safety net: A system administrator applies experimental firewall rules, then schedules a job to revert them in 10 minutes. If the new rules work correctly, the administrator cancels the revert job. If the connection is lost, the revert runs automatically.
  • One-time report generation: A report must be delivered at a precise time, but the administrator will not be available at that moment.
  • Staged deployments: A configuration change is queued for a future maintenance window approved by change management.

The at Package and atd Daemon

Red Hat Enterprise Linux provides the at package, which is installed and enabled by default in RHEL 10. This package supplies:

Component Type Description
atd System daemon Runs in the background, checks for pending jobs, and executes them at the correct time.
at Command Queues a new job and reads the commands to run from standard input or a script file.
atq Command Lists all pending jobs for the current user (equivalent to at -l).
atrm Command Removes a pending job by its job number (equivalent to at -d).

Job Queues

The atd daemon supports multiple job queues, labeled a through z and A through Z. Jobs are placed in the default queue a unless the -q option specifies otherwise. Queues with higher letters run at a higher nice value (lower CPU priority), meaning they run more politely in the background.

A labeled group of scheduled at jobs. The atd daemon supports queues a-z and A-Z, with higher-letter queues receiving lower scheduling priority. A process scheduling priority value (-20 to 19). Higher nice values mean the process yields CPU time to others and runs at lower priority. "Being nice" means using fewer resources.
🔴Privilege Boundary

Unprivileged users can view and manage only their own queued jobs. The root user can view and manage all jobs for all users. This is an important distinction for multi-user environments.

3

Using the at Command

Scheduling a New Job

The basic syntax for queueing a deferred job is:

Basic at Command Syntax
# Queue a job from a script file (most common method) user@host:~$ at TIMESPEC < script_file # Queue a job interactively (type commands, finish with Ctrl+D) user@host:~$ at TIMESPEC at> echo "Job running" > /tmp/job.out at> job 5 at Thu Mar 12 23:00:00 2026

The at command reads the commands to execute from standard input (STDIN). When you redirect from a script file (< myscript), the script’s contents become the job. When entering commands interactively, press Ctrl+D on an empty line to signal end of input.

Standard input — the default source of data for a command, typically the keyboard in interactive use. Can be redirected from a file using the < operator.
Scheduling a Script to Run in 5 Minutes
user@host:~$ at now +5min < myfile warning: commands will be executed using /bin/sh job 2 at Wed May 14 20:40:00 2025

Time Specification (TIMESPEC)

The at command accepts flexible, natural language time specifications. The general rule is: provide time first, then optionally a date or offset.

💡Time vs. Date Defaults

If you specify only a time (no date), the job runs today if that time hasn’t passed yet, or tomorrow if it has. If you specify only a date, the time defaults to the current time of day.

now +5min

Five minutes from this moment.

now +2hours

Two hours from now.

teatime tomorrow

4:00 PM tomorrow (teatime = 16:00).

noon +4 days

12:00 PM four days from today.

midnight

Midnight tonight (00:00).

02:00pm

2:00 PM today or tomorrow.

15:59

3:59 PM in 24-hour format.

5pm august 3 2026

5:00 PM on a specific date.

The full list of valid specifications is documented in /usr/share/doc/at/timespec on your RHEL system.

Inspecting the Job Queue with atq

Use atq (or the equivalent at -l) to list pending jobs:

Viewing Pending Jobs with atq
user@host:~$ atq 28 Mon May 19 05:13:00 2025 a user 29 Tue May 20 16:00:00 2025 h user 30 Wed May 21 12:00:00 2025 a user

Each line of atq output contains four fields:

FieldExampleMeaning
Job number28Unique identifier for this job. Used with atrm to remove it.
Execution date/timeMon May 19 05:13:00 2025When the job will run.
Queue letteraWhich queue the job is in (a–z, A–Z).
OwneruserThe user account that created the job.

Inspecting Job Contents with at -c

To see what commands a specific job will run, use at -c JOBNUMBER. This displays the full shell environment that was captured when the job was created, followed by the commands themselves:

Inspecting Job #28 Contents
user@host:~$ at -c 28 #!/bin/sh # atrun uid=1000 gid=1000 # mail user 0 umask 22 SHELL=/bin/bash; export SHELL HOME=/home/user; export HOME PATH=/usr/local/bin:/usr/bin:/bin; export PATH cd /home/user || { echo 'Execution directory inaccessible'; exit 1; } /usr/local/bin/daily_report.sh

Removing Jobs with atrm

To cancel a pending job before it runs, use atrm JOBNUMBER:

Removing a Pending Job
# Remove job number 29 from the queue user@host:~$ atrm 29 # Verify it was removed user@host:~$ atq 28 Mon May 19 05:13:00 2025 a user 30 Wed May 21 12:00:00 2025 a user
⚠️Jobs Execute in the Background

Jobs run by atd do not have a controlling terminal. Any output (stdout or stderr) that is not redirected to a file will be emailed to the job owner. Ensure your system has a mail transfer agent (MTA) configured, or always redirect output explicitly in your scripts.

A mail transfer agent (MTA) is software that routes email between systems. Examples include Postfix and Sendmail. Without one configured, at job output is silently lost.

Scheduling Recurring User Jobs

Many system administration tasks need to run not once, but repeatedly on a regular schedule — every day, every week, or every few minutes. Recurring jobs are managed by the crond daemon, which is enabled and started by default in RHEL 10.

A scheduled task that runs automatically at repeated intervals according to a defined schedule, such as daily, weekly, or every N minutes.

How crond Works

The crond daemon wakes up every minute, reads its configuration files, and launches any jobs whose schedule matches the current time. It reads from two sources:

  • Per-user crontab files: Each user has a personal crontab file edited with the crontab -e command. These files are stored in a system directory but are owned and managed by the respective user.
  • System-wide cron directories: Files in /etc/cron.d/, /etc/cron.daily/, /etc/cron.hourly/, etc. These are for system-level recurring tasks and are managed by the root user or packages.
🔴Do Not Use crontab as root for Personal Jobs

Using crontab as the root user is discouraged because personal jobs configured to run as root can be exploited by privilege escalation attacks. System-wide scheduled jobs should instead use the /etc/cron.d/ drop-in directory with explicit user fields. This chapter covers user crontab management only.

Crontab Command Options

CommandAction
crontab -l List all scheduled jobs for the current user.
crontab -e Open the user’s crontab in the default editor (vim unless $EDITOR is set).
crontab -r Remove all jobs for the current user. Use with caution — no confirmation prompt.
crontab filename Replace the current crontab with the contents of filename. Reads from stdin if no file given.
crontab -u username -l List jobs for another user (requires privilege, typically root).

Output and Email Behavior

When a cron job produces output to stdout or stderr that is not redirected, the crond daemon captures it and emails it to the job owner. Two environment variables in the crontab file control this behavior:

A named variable stored in the shell environment that affects process behavior. In a crontab, variables like SHELL and MAILTO configure how jobs run and where output is sent.
VariablePurposeExample
SHELL Sets the shell used to execute cron commands. Default is /bin/sh. SHELL=/bin/bash
MAILTO Email address to receive job output. Set to empty (MAILTO="") to suppress all email. MAILTO=admin@example.com
HOME Home directory used by the cron job. Defaults to the user’s home directory. HOME=/home/student
💡Suppress Email Output

To prevent email noise from cron jobs, either redirect all output explicitly in the command (>> /var/log/myjob.log 2>&1) or add MAILTO="" at the top of your crontab file.

Crontab Syntax

Each job entry in a crontab file consists of exactly six fields separated by whitespace. The first five fields define the schedule; the sixth is the command:

Crontab Field Order
# minute hour day-of-month month day-of-week command * * * * * /path/to/command # | | | | | # 0-59 0-23 1-31 1-12 0-7 # (or name)(0=Sun, 7=Sun)

Field Value Syntax

Each of the five time fields uses the same set of syntax rules:

Wildcard: *

The asterisk means “every possible value” for that field. For example, * in the Hours field means every hour (0–23).

# Run every minute, every hour, every day * * * * * /usr/local/bin/heartbeat.sh # Run at minute 0 of every hour, every day 0 * * * * /usr/local/bin/hourly_check.sh

Specific Number

A single integer selects exactly one value for the field. Days of the week use 0–7 where both 0 and 7 represent Sunday. Month names (Jan–Dec) and day names (Mon–Sun) are also accepted as three-letter abbreviations.

# Run at 9:00 AM every Friday 0 9 * * 5 /usr/local/bin/weekly_report.sh # Same job using day abbreviation 0 9 * * Fri /usr/local/bin/weekly_report.sh # Run at 9:00 AM on the 1st of every February 0 9 1 2 * /usr/local/bin/feb_audit.sh

Range: x-y

A hyphen between two values selects all integers from x to y inclusive. The first number must always be less than or equal to the second.

# Run at 9:00 AM Monday through Friday 0 9 * * 1-5 /usr/local/bin/daily_report.sh # Run every minute during business hours (9AM-5PM) * 9-17 * * 1-5 /usr/local/bin/monitor.sh # WRONG: Fri-Tue is invalid because 5 > 2 # CORRECT: Use a list of two ranges instead 0 9 * * 0-2,5-6 /usr/local/bin/job.sh
🔴Range Direction Rule

In a range x-y, x must be less than or equal to y. Tue-Fri (2-5) is valid. Fri-Tue (5-2) is not valid and will cause an error. Express wrap-around schedules as a comma-separated list of two ranges.

List: x,y

Comma-separated values match each individual value in the list. Lists may also contain ranges.

# Run at minutes 0, 15, 30, and 45 of every hour 0,15,30,45 * * * * /usr/local/bin/quarter_check.sh # Run at minutes 5, 10-13, and 17 5,10-13,17 * * * * /usr/local/bin/complex_check.sh # Run on January, March, July (months 1, 3, 7) 0 0 1 1,3,7 * /usr/local/bin/quarterly.sh

Step Interval: */x

The slash operator defines a step interval. */x matches every x-th value in the full range of the field. Can also be combined with a range: 10-50/5 means every 5th value from 10 to 50.

# Run every 7 minutes */7 * * * * /usr/local/bin/poll.sh # Run every 2 hours 0 */2 * * * /usr/local/bin/bi_hourly.sh # Run every 5 minutes between :10 and :50 10-50/5 * * * * /usr/local/bin/windowed_check.sh

The Percentage Sign (%) in Commands

The crontab format has one quirk: an unescaped percent sign (%) in the command field is treated as a newline character, and everything after it is passed to the command as standard input.

In crontab command fields, an unescaped % is treated as a newline. Text after % is piped to the command as stdin. Escape literal % characters with a backslash: \%.
Percent Sign as Newline / stdin in Crontab
# Send email with body text using % as newline/stdin separator 0 9 * * 1-5 mutt -s "Checking in" developer@example.com % Hi there, just checking in. # The % sends "Hi there, just checking in." as stdin to mutt # To use a literal % in a command, escape it with backslash */5 * * * * printf "Load: \%s\n" "$(cat /proc/loadavg)" >> /var/log/load.log

Examples of Recurring User Jobs

Let’s walk through several real-world crontab examples from the course material, analyzing the schedule expressed by each entry.

Example 1: Annual Backup

Run on February 3rd at 9:00 AM, Every Year
# Minute Hour DayOfMonth Month DayOfWeek Command 0 9 3 2 * /usr/local/bin/yearly_backup

Reading this entry: Run at minute 0 of hour 9 (9:00 AM), on the 3rd day of the month, only in month 2 (February), on any day of the week. The combination of specific day-of-month (3) and specific month (2) means this job runs once per year: February 3rd at 9:00 AM.

Example 2: Friday Chime in July

Every 5 Minutes, Fridays in July, 9AM–4:55PM
# Minute Hour DayOfMonth Month DayOfWeek Command */5 9-16 * Jul 5 echo "Chime"

Reading this entry: At every 5th minute (0, 5, 10, …, 55), during hours 9 through 16 (9:00 AM through 4:59 PM), on any day of the month, only in July, only on Fridays (day 5). The range 9–16 for hours means the job starts at 9:00 and the last execution occurs at 16:55, because the next interval (17:00) falls outside the range.

💡Email Output Behavior

Because this job uses echo "Chime" without output redirection, the crond daemon will capture that output and email it to the job owner. This runs in a background environment with no controlling terminal (no screen), so crond buffers output and sends it via email. Set MAILTO="" in your crontab to suppress this.

Example 3: Weekday Midnight Report

Two Minutes Before Midnight, Monday–Friday
# Minute Hour DayOfMonth Month DayOfWeek Command 58 23 * * 1-5 /usr/local/bin/daily_report

Reading this entry: At minute 58 of hour 23 (11:58 PM), every day of the month, every month, Monday through Friday (days 1–5). This job runs just before midnight on every weekday, a common pattern for “end of business day” reporting.

Example 4: Email Check-In with stdin

Daily Email Check-In at 9:00 AM Weekdays
# Minute Hour DayOfMonth Month DayOfWeek Command 0 9 * * 1-5 mutt -s "Checking in" developer@example.com % Hi there, just checking in.

Reading this entry: At 9:00 AM every weekday, run mutt to send an email. The % character separates the command from the message body — text after the first % is passed to mutt as standard input, which becomes the email body.

Chapter Summary

In this chapter, you learned how to schedule both one-time deferred jobs and recurring automated tasks on RHEL 10 as a regular user, without requiring root access or system-level configuration.

Key Concepts Reviewed

  • The at command queues a job to run exactly once at a future time specified in natural language. The atd daemon monitors the queue and executes jobs at the scheduled time.
  • The atq command (or at -l) lists pending jobs; atrm (or at -d) cancels them by job number.
  • Time specifications for at are flexible natural language expressions: now +5min, teatime tomorrow, noon +4 days, 5pm august 3 2026.
  • The safety-net pattern uses at to schedule a revert job when making risky configuration changes, then cancels it if the change succeeds.
  • The crond daemon manages recurring jobs from per-user crontab files, which are edited with crontab -e.
  • Crontab entries use five time fields (minute, hour, day-of-month, month, day-of-week) followed by the command, supporting * wildcards, specific values, ranges (x-y), lists (x,y), and step intervals (*/x).
  • An unescaped % in a crontab command field acts as a newline and passes subsequent text to the command as standard input.
  • Always back up a crontab with crontab -l > backup.txt before running crontab -r, which deletes all entries with no confirmation.
