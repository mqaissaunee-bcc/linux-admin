NFS Overview

The Network File System (NFS)NFS is an open internet-standard protocol originally developed by Sun Microsystems that allows file sharing across a network. It supports native Linux file permissions and attributes. is an internet-standard protocol that Linux, UNIX, and similar operating systems use as their native network file system. NFS is an open standard that supports native Linux permissions and file-system attributes, making it a first-class file system for RHEL environments.

NFS Protocol Versions

By default, Red Hat Enterprise Linux 10 uses NFS version 4.2. RHEL 10 fully supports both NFSv3 and NFSv4 protocols. Understanding the differences between these versions is important when working in heterogeneous environments.

NFSv3 — Legacy RPC-Based Protocol

  • Uses the Remote Procedure Call (RPC)RPC is a protocol that allows a program on one computer to execute code on another computer over the network without the programmer explicitly managing network details. protocol for all NFS transactions.
  • Requires the rpcbind service to run on the NFS server (TCP/UDP port 111).
  • Supports both TCP and UDP transport protocols.
  • Clients use the showmount command to discover available exports.
  • Stateless by design — server does not track open files.
⚠️Legacy Warning

NFSv3 is considered legacy. New deployments should use NFSv4 or NFSv4.2 unless compatibility with older systems specifically requires NFSv3.

NFSv4 / NFSv4.2 — Modern Stateful Protocol

  • Eliminates the use of the legacy RPC protocol and rpcbind service.
  • Uses TCP only (port 2049) — no UDP support.
  • Introduces an export treeThe NFSv4 export tree is a browseable pseudo-filesystem rooted at / on the NFS server that contains all paths of exported directories, allowing clients to discover available exports. containing all paths for the server’s exported directories.
  • Supports Kerberos security for encrypted, authenticated mounts.
  • NFSv4.2 (RHEL 10 default) adds: server-side copy, sparse file support, labelled NFS with SELinux.
  • Stateful protocol — server tracks open files and client state.
💡Best Practice

Always use NFSv4.2 in new RHEL 10 deployments. The RHCSA exam tests NFSv4 knowledge. Use nfsvers=4.2 or fstype=nfs4 mount options to explicitly request NFSv4.

SMB / CIFS — Windows File Sharing

  • RHEL 10 supports mounting shared directories from Microsoft Windows systems.
  • Uses the Server Message Block (SMB)SMB is a network file-sharing protocol developed by Microsoft. The Common Internet File System (CIFS) is an older dialect of SMB. Linux can access SMB shares using the cifs-utils package. or Common Internet File System (CIFS) protocols.
  • Requires the cifs-utils package; uses -t cifs mount type.
  • Mounting options are protocol-specific and depend on your Windows Server or Samba configuration.
  • Mounting SMB/CIFS shares from Microsoft Windows systems is beyond the scope of this course, but the same general mounting methods apply.

NFS Architecture: Server and Client Roles

NFS follows a classic client-server model. NFS servers export directories, making them available to the network. NFS clients mount exported directories to an existing local mount point directory on the client system.

Three Ways to Mount NFS Exports

NFS clients can mount exported directories using one of three methods. Each has trade-offs in terms of persistence, fault tolerance, and user permissions:

Method Command / File Persists on Reboot Root Required Best Use Case
Manual mount No Yes Temporary access, testing
Persistent /etc/fstab Yes Yes Always-available storage
Automounter autofs Yes No* On-demand, user homedirs

* Users do not need root to access automounted directories; the autofs service handles mounting as root in the background.

Querying NFS Exports

Before mounting an NFS export, you need to know what directories the server is making available. The method for discovering these exports differs between NFSv3 and NFSv4 because of fundamental differences in how the two protocol versions operate.

Querying NFSv3 Exports with showmount

NFSv3 uses the rpcbindrpcbind is a service that maps RPC program numbers to network ports. NFSv3 servers require rpcbind to run on TCP/UDP port 111 so clients can discover which port the NFS service is using. service as a port mapper. Clients connect to port 111 on the server, learn which port NFS is using, then connect to that port. The showmount command handles this discovery automatically.

Query NFSv3 Exports with showmount
[root@host ~]# showmount --exports server Export list for server: /shares/test1 /shares/test2

Querying NFSv4 Exports by Browsing the Export Tree

NFSv4 eliminates the need for rpcbind and introduces an export tree. To discover what a server exports, you mount the root (/) of the server’s export tree. This gives you a browseable view of all available exports without actually mounting any of them.

Browse NFSv4 Export Tree
# Mount the server's export tree root to a temporary mount point [root@host ~]# mount server:/ /mnt # Browse available exports (no actual data mounted yet) [root@host ~]# ls /mnt docs home shared # Navigate into an export to trigger a real mount [root@host ~]# cd /mnt/docs
🔴Kerberos-Protected Exports

Exported directories that use KerberosKerberos is a network authentication protocol that uses tickets to allow nodes to prove their identity securely over a non-secure network. NFS can use Kerberos (sec=krb5) for strong authentication and encryption. security (sec=krb5) will appear in the export tree but cannot be accessed without valid Kerberos credentials. Their path names are visible but their content is not accessible during export tree browsing.

Manually Mounting NFS Exports

Manual mounting provides temporary access to an NFS export. This is useful for testing connectivity, verifying permissions, or providing short-term access without modifying system configuration files. Manual mounts do not survive a reboot.

Step 1: Create a Mount Point

A mount pointA mount point is a directory on the local file system where a remote file system is attached. The directory must exist before you can mount something to it. While /mnt exists by default, creating dedicated directories is best practice. must exist before you can mount to it. Although /mnt is available as a temporary mount point, the recommended practice is to create a dedicated directory for long-term or persistent mounting.

Create Mount Point
[root@host ~]# mkdir -p /mnt/nfsdocs

Step 2: Mount the NFS Export

Use the mount command with the -t nfs type flag and mount options. The server:/export syntax tells the mount command this is an NFS share.

Mount NFS Export
# Full syntax: mount -t nfs -o options server:/export mountpoint [root@host ~]# mount -t nfs -o rw,sync serverb:/shares/docs /mnt/nfsdocs # Verify the mount [root@host ~]# mount | grep nfs serverb:/shares/docs on /mnt/nfsdocs type nfs4 (rw,relatime,sync,vers=4.2,...)

Key Mount Options Explained

OptionMeaningWhen to Use
rwRead/write accessWhen clients need to write data
roRead-only accessShared config/data that should not be modified
syncWrites complete before returning to callerRecommended for all production mounts — prevents data loss
asyncWrites may be bufferedHigher performance, but risk of data loss on failure
nfsvers=4.2Force specific NFS versionExplicitly require NFSv4.2 features
_netdevNetwork device required/etc/fstab entries — delays mount until network is available
softReturn errors if server unreachableNon-critical mounts; prevents hangs
hardRetry indefinitely if server unreachableCritical data mounts (default behavior)
💡Why Use sync?

The sync option forces all write operations to complete on the server before returning success to the calling process. This is strongly recommended for all production network mounts where transactions must be completed or returned as failed. Without sync, a server crash could cause silent data loss.

Unmounting NFS Exports

Unmount an NFS export with the umount command (note: no “n” in umount). You must not be inside the mounted directory when unmounting.

Unmount NFS Export
[root@host ~]# umount /mnt/nfsdocs

Handling “Device is Busy” Errors

Sometimes umount fails with a “device is busy” error. This means a process has an open file or a shell has its current working directory inside the mounted file system. Use lsof to find the culprit:

Find Open Files Preventing Unmount
[root@host ~]# lsof /mnt/nfsdocs COMMAND PID USER FD TYPE DEVICE SIZE/OFF NODE NAME program 5534 user txt REG 252,4 910704 128 /mnt/nfsdocs/program # After closing the process, retry unmount [root@host ~]# umount /mnt/nfsdocs # Last resort: force unmount (RISK OF DATA LOSS) [root@host ~]# umount -f /mnt/nfsdocs
⚠️Warning: Force Unmount

The -f force option can cause loss of unwritten data for all open files on the mount. Use it only as a last resort in critical scenarios when an application cannot be closed gracefully.

Persistently Mounting NFS Exports

To have an NFS export mounted automatically every time the system boots, add an entry to the /etc/fstab/etc/fstab is the file system table — a configuration file that defines how disk partitions and remote file systems should be automatically mounted during system startup. file. This file uses syntax similar to local file system entries, with the addition of NFS-specific options.

/etc/fstab Entry Format for NFS

/etc/fstab — NFS Entry Syntax
# Format: server:/export mountpoint fstype options dump pass serverb:/shares/docs /mnt/nfsdocs nfs rw,sync,_netdev 0 0 # After editing /etc/fstab, mount the new entry: [root@host ~]# mount /mnt/nfsdocs

Understanding the /etc/fstab Fields

FieldExample ValueExplanation
Device/Sourceserverb:/shares/docsNFS server hostname and exported path
Mount Point/mnt/nfsdocsLocal directory to mount to (must exist)
File System TypenfsNFS file system type (or nfs4 for NFSv4 only)
Optionsrw,sync,_netdevMount options; _netdev waits for network before mounting
Dump0Legacy backup flag; always 0 for NFS
Pass0fsck order; always 0 for NFS (no fsck on network FS)
🔴Important: Boot-Time Risks with /etc/fstab

Mounting NFS exports via /etc/fstab can cause problems if the network is unavailable at boot time or if the NFS server is unreachable. Storage timeouts may increase significantly, potentially causing very slow boots or boot failures. Always include the _netdev option and consider whether the automounter is a better choice for network file systems.

💡Removing a Persistent Mount

Unmounting with umount does not remove the entry from /etc/fstab. The mount will return on the next reboot. To permanently remove a persistent NFS mount, delete or comment out its line in /etc/fstab, then run umount.

The Automounter Service (autofs)

The automounter service (autofs)autofs is a kernel and user-space service that mounts file systems on demand when accessed and unmounts them automatically after a configurable idle timeout. It replaces the need for static /etc/fstab entries for network file systems. is a service that automatically mounts file systems and NFS exports on demand, and automatically unmounts them when the mounted resources are no longer in use. Understanding why the automounter exists — not just how to use it — is essential to appreciating its role in system administration.

Why Does the Automounter Exist?

The automounter was created to solve two related problems:

  1. Non-privileged user access: Regular users do not have permission to use the mount command. Without the automounter, non-privileged users cannot access removable media or unmounted network shares — even if they have permissions on the remote file system itself.
  2. On-demand mounting: File systems mounted in /etc/fstab are mounted at boot and remain mounted until shutdown. If the NFS server is unavailable at boot, the system may hang. The automounter only mounts when access is actually needed.

How the Automounter Works

The autofs service uses configuration files populated with mount information, similar to /etc/fstab. The key difference is when mounting occurs:

Key Advantages of the Automounter

Protection from Unexpected Corruption

By unmounting a file system each time it is no longer in use, the automounter protects it from unexpected corruption when open files are kept open unnecessarily. A file system that is not mounted cannot be corrupted by crashes or power failures — the kernel has no open buffers for it.

Always Uses Current Mount Configuration

When the automounter remounts a file system, it reads the current mount configuration. This is in contrast to /etc/fstab, which may still be using a configuration from months ago at the last system boot. Configuration changes take effect on the next mount cycle without a reboot.

Non-Privileged User Access

File systems mounted through the automounter are available by default to all users without requiring root privileges. The autofs service performs the actual mount and umount calls as root in the background, transparently to the accessing user. This is essential for user home directories stored on NFS servers.

Redundant Server Support

If your NFS server configuration includes redundant servers and paths, the automounter can select the fastest available connection each time a new file system mount is requested. This provides basic load balancing and fault tolerance that static /etc/fstab entries cannot achieve.

Direct vs. Indirect Mounts

The autofs service supports two types of mount-point mapping to handle the two common use cases for automounting:

TypeMount PointWhen CreatedUse Case
Direct Map Absolute path (e.g., /mnt/docs) Permanent directory; exists even when unmounted Known, fixed mount locations
Indirect Map Relative name under a base (e.g., /shares/docs) Created by autofs on demand; removed when unmounted Dynamic sets of mounts (user homedirs, project dirs)

Configuring the Automounter Service

Configuring the automounter requires installing packages, editing the master map file, and creating one or more map files. The configuration uses a two-level structure: a master map that identifies base directories, and map files that specify individual mounts.

Step 1: Install Required Packages

Install autofs and nfs-utils
[root@host ~]# dnf install autofs nfs-utils ...output omitted... Is this ok [y/N]: y ...output omitted... Complete!

Step 2: The Master Map File

The master map fileThe autofs master map file ties together base mount directories with their corresponding map files. Each line associates a base mount point directory with a map file that contains the detailed mount information for that directory. is the top-level configuration for autofs. By default, autofs uses /etc/auto.master. The recommended location for custom entries is the /etc/auto.master.d/ directory, which autofs reads automatically.

💡Naming Convention

Files in /etc/auto.master.d/ must have the .autofs extension to be recognized by autofs. The name itself is mostly arbitrary (e.g., shares.autofs, home.autofs), but should be meaningful.

/etc/auto.master.d/demo.autofs — Master Map Format
# Format: mountpoint map-file [options] /shares /etc/auto.shares # mountpoint: Base directory for this group of automounts # map-file: File that contains the individual mount details

Step 3: The Map File

The map fileAn autofs map file specifies the individual mount points, their mount options, and source locations. It is the equivalent of individual /etc/fstab lines, but only for the automounted directory tree defined in the master map. specifies the individual mount points within the base directory defined in the master map. The automounter creates directories automatically if they do not exist.

/etc/auto.shares — Map File Format
# Format: mountpoint -options source-location work -rw,sync hosta:/shares/work docs -rw,sync hosta:/shares/docs # Mount options start with a dash (-) and are comma-separated # Source location uses the host:/path format for NFS
📖Naming Convention for Map Files

The naming convention for a map file is /etc/auto.name, where name reflects the content of the map. For example, /etc/auto.shares for share mounts, /etc/auto.home for home directories.

Step 4: Start and Enable the autofs Service

After creating the master map and map files, start and enable the autofs service using systemctl.

Enable and Start autofs
[root@host ~]# systemctl enable --now autofs Created symlink /etc/systemd/system/multi-user.target.wants/autofs.service → /usr/lib/systemd/system/autofs.service. # Verify autofs is running [root@host ~]# systemctl status autofs ● autofs.service - Automounts filesystems on demand Loaded: loaded (/usr/lib/systemd/system/autofs.service; enabled) Active: active (running) since ...

Alternative: systemd.automount

The systemd.automountsystemd.automount is a systemd unit type that creates on-demand mount points from /etc/fstab entries. It is simpler to configure than autofs but only supports absolute path (direct) mounts. unit type provides an alternative automount method by using /etc/fstab entries with the x-systemd.automount option. This can be simpler than the full autofs configuration.

/etc/fstab — systemd.automount Method
# Add x-systemd.automount to the options in /etc/fstab: serverb:/remote/finance /remote/finance nfs rw,sync,x-systemd.automount 0 0 # Reload systemd to pick up the new fstab entry [root@host ~]# systemctl daemon-reload # Enable the generated automount unit # Unit name is based on mount path: / becomes - and path separators become - [root@host ~]# systemctl start remote-finance.automount
💡systemd.automount Limitation

The systemd.automount method supports only absolute path mount points, similar to autofs direct maps. It cannot create dynamic indirect mounts or use wildcards. For home directories or large sets of dynamic mounts, use the full autofs service.

Direct and Indirect Map Files

Mastering the difference between direct and indirect map files is essential for the RHCSA exam and for real-world autofs administration. The two types serve different use cases and use different configuration syntax.

Direct Map Files

A direct mapA direct map mounts an NFS export at a specific absolute path that is always visible in the file system, even when unmounted. The actual content appears when accessed. Direct maps use /- as the base directory in the master map. maps an NFS export to an absolute, known mount point location. The directory appears to exist permanently, even when the file system is not mounted. The content becomes visible when accessed.

To use direct maps, use /- as the base directory in the master map file:

Direct Map — Master Map Entry
# /etc/auto.master.d/direct.autofs # Base directory /- means: use absolute paths from the map file /- /etc/auto.direct
/etc/auto.direct — Direct Map File
# Format: /absolute/path -options server:/export /mnt/docs -rw,sync hosta:/shares/docs /mnt/apps -ro,sync hosta:/shares/apps # /mnt must exist; autofs creates/removes /mnt/docs and /mnt/apps

Indirect Map Files

An indirect mapAn indirect map mounts NFS exports as subdirectories under a specified base directory. The subdirectories do not exist until accessed, at which point autofs creates them, mounts the export, and removes them again when the export is unmounted. uses a base directory under which subdirectories appear dynamically when accessed. The directories appear to exist when you ls the base directory, but the actual NFS mount happens only when you enter one of them.

Indirect Map — Master Map Entry
# /etc/auto.master.d/indirect.autofs /shares /etc/auto.indirect # /shares is the base; autofs creates /shares/work, /shares/docs, etc.
/etc/auto.indirect — Indirect Map File
# Format: relative-name -options server:/export work -rw,sync hosta:/shares/work docs -rw,sync hosta:/shares/docs # Results in: /shares/work and /shares/docs # autofs creates and removes these directories automatically

Using Wildcards in Indirect Maps

When an NFS server exports multiple subdirectories within a parent directory, you can use a wildcard entryAn autofs wildcard entry uses * as the mount point key and & as a placeholder in the source location. When a user accesses /base/anything, the * matches "anything" and & substitutes it into the source path, so autofs mounts server:/export/anything automatically. to handle all subdirectories with a single line:

/etc/auto.indirect — Wildcard Entry
# Format: * -options server:/export/& # * = any subdirectory name # & = substituted with the actual name used by the user * -rw,sync hosta:/shares/& # User accesses: /shares/work # autofs mounts: hosta:/shares/work -> /shares/work # User accesses: /shares/reports # autofs mounts: hosta:/shares/reports -> /shares/reports
🔴Wildcard Requirement

For a wildcard entry to work, the directory accessed must actually exist as an export on the NFS server. If a user accesses /shares/nonexistent and the server does not export that path, the mount will fail. Wildcards do not automatically create server-side exports.

Chapter Summary

In this chapter, you learned about the Network File System protocol and how to mount and manage NFS exports on RHEL 10, both manually and through the automounter service.

Key Concepts Review

  • NFS is an open-standard network file system protocol. RHEL 10 defaults to NFSv4.2, which uses TCP only (port 2049) and includes an export tree for discovering available shares.
  • NFSv3 uses the RPC protocol and rpcbind (port 111). Use showmount --exports server to query NFSv3 exports.
  • NFS exports can be mounted manually (mount command, temporary), persistently (/etc/fstab, survives reboot), or on-demand (autofs service).
  • The sync mount option ensures write operations complete before returning to the caller and is strongly recommended for all production NFS mounts.
  • Use lsof mountpoint to identify processes preventing an unmount. Use umount -f only as a last resort.
  • The autofs service mounts file systems on demand and unmounts them after an idle timeout, protecting file systems and requiring no root permissions from users.
  • Direct maps use /- as the base in the master map and absolute paths in the map file. Indirect maps use a base directory and relative names.
  • Wildcard entries (* -options host:/path/&) mount any subdirectory from a server automatically.
  • The systemd.automount alternative uses /etc/fstab with the x-systemd.automount option but supports only absolute paths.

Command Reference

dnf install nfs-utils

Install NFS client tools and kernel modules required for mounting

dnf install autofs nfs-utils

Install the automounter service and NFS utilities

Query available NFS exports on an NFSv3 server

Manually mount an NFS export with read-write and sync options

Unmount a mounted NFS export (must not be in use)
