SELinux Architecture

Security Enhanced Linux (SELinux) A kernel-level security module developed by the NSA and integrated into RHEL. It enforces mandatory access control policies on top of standard Linux file permissions. is a critical security layer built into the Linux kernel. While traditional Linux permissions control who can read, write, or execute a file, SELinux controls how each process can use the resources it accesses. This distinction is fundamental: a malicious process running as an authorized user can still be blocked by SELinux from performing unauthorized actions.

The Problem SELinux Solves

Consider a web server with a vulnerability. An attacker who exploits that vulnerability gains the same system privileges as the apache user. With only standard Linux permissions, that attacker could read any file accessible to apache, write to /tmp, or even attempt to escalate privileges. SELinux prevents this by confining the web server process to only the specific file types and ports it is designed to use.

DAC vs. MAC: Two Security Layers

Linux systems enforce two layers of access control simultaneously:

👤

Discretionary Access Control (DAC)

The traditional Linux permission model. Owners and administrators set permissions using chmod, chown, and ACLs. Users can discretionarily grant or restrict access. SELinux decisions happen after DAC checks pass.

🔒

Mandatory Access Control (MAC)

SELinux's model. Policies are system-wide and cannot be bypassed by individual users. Even root cannot override SELinux denials without modifying the policy itself. MAC adds an extra gate that DAC cannot influence.

🔴Key Principle

SELinux does not replace file permissions. Both must allow an action for it to succeed. A file could be world-readable (DAC allows) but if the process's SELinux type has no policy rule for that file's context (MAC denies), the access fails.

Targeted Policy and Confined Domains

RHEL uses a targeted policy The default SELinux policy in RHEL. Only specific targeted processes (like Apache, SSH, DNS) are confined. Other processes run unconfined and receive no SELinux protection. by default. In a targeted policy, only specific daemons and services (like httpd, sshd, named) are confined. A confined process runs within an SELinux domain and is subject to all applicable MAC policy rules. An unconfined process has no SELinux restrictions beyond standard DAC permissions. Applications without a specific SELinux policy run unconfined and receive no MAC protection. This approach protects the most critical services while minimizing interference with other software.

SELinux Context Labels

Every process, file, directory, and network port in an SELinux system carries a security context label. A string of metadata attached to every SELinux-managed resource. It has four fields: user, role, type, and security level. Policy rules match context labels to determine allowed actions. Policy rules match a process's context label to a resource's context label to decide whether access is allowed.

A full SELinux context label has four colon-separated fields:

system_u:object_r:httpd_sys_content_t:s0   /var/www/html/index.html
system_u = SELinux User object_r = SELinux Role httpd_sys_content_t = Type (most important) s0 = Sensitivity Level
📋Type Context is Key

In the targeted policy, the type field drives all policy decisions. Type context names always end in _t (e.g., httpd_t for the web server process, httpd_sys_content_t for web content files). When troubleshooting SELinux, focus on type labels.

SELinux Operating Modes

SELinux operates in one of three modes at any given time. Understanding when to use each mode is essential for both daily administration and troubleshooting.

🛡

Enforcing (Default & Recommended)

SELinux actively enforces all policy rules. Any access not explicitly allowed is denied and logged. This is the production-ready mode and the default in RHEL 10.

📝

Permissive (Diagnostic)

SELinux loads policies and logs all violations but does not block any access. Used for testing new applications and troubleshooting policy issues without breaking services.

⛔

Disabled (Strongly Discouraged)

SELinux is completely inactive. No policies are loaded and no file labeling occurs. Re-enabling SELinux after disabling it requires a full filesystem relabel, which is slow and risky. RHEL 10 no longer supports this option.

⚠RHEL 10 Change

RHEL 10 no longer supports setting SELINUX=disabled in /etc/selinux/config. To boot without SELinux enforcement, use the enforcing=0 or selinux=0 kernel boot parameters temporarily instead.

Checking the Current Mode

getenforce — View Current Mode
[root@host ~]# getenforce Enforcing

Changing the Mode Temporarily

Use setenforce to toggle between enforcing and permissive mode at runtime. This change does not survive a reboot.

setenforce — Change Mode at Runtime
# Switch to permissive (0 = permissive, 1 = enforcing) [root@host ~]# setenforce 0 [root@host ~]# getenforce Permissive # Switch back to enforcing [root@host ~]# setenforce 1 [root@host ~]# getenforce Enforcing
💡Troubleshooting Tip

When a service suddenly stops working, temporarily switching to permissive mode is a quick way to determine if SELinux is the cause. If the service works in permissive mode but not enforcing mode, you have an SELinux context or Boolean issue to resolve.

Setting the Persistent Default Mode

Edit /etc/selinux/config to make a mode change survive reboots. This file is read at boot time:

/etc/selinux/config
# This file controls the state of SELinux on the system. # SELINUX= can take one of these three values: # enforcing - SELinux security policy is enforced. # permissive - SELinux prints warnings instead of enforcing. # disabled - No SELinux policy is loaded. (Not supported in RHEL 10) SELINUX=enforcing # SELINUXTYPE= can take one of these three values: # targeted - Targeted processes are protected. # minimum - Modification of targeted policy; only selected processes protected. # mls - Multi Level Security protection. SELINUXTYPE=targeted

Kernel Boot Parameters

SELinux mode can also be controlled at boot time via kernel parameters, which override /etc/selinux/config:

ParameterEffect
enforcing=1Boot into enforcing mode
enforcing=0Boot into permissive mode (useful for troubleshooting)
selinux=1Enable SELinux at boot
selinux=0Disable SELinux at boot (temporary only; not persistent)
⚠Reboot Advisory

Red Hat recommends rebooting after switching from permissive to enforcing mode. Services that started in permissive mode may lack proper context labels, causing unexpected denials in enforcing mode after a reboot restarts them with full confinement.

Controlling SELinux File Contexts

The most common SELinux problem you will encounter is a file with the wrong context label. Understanding how files get their contexts, and how to change them correctly, is the most practical SELinux skill for a system administrator.

How Files Get Their Initial Context

SELinux maintains a file-labeling policy database in /etc/selinux/targeted/contexts/files/. When a file is created:

  1. Policy match: If the file's path matches a labeling policy rule, the policy's specified context is applied.
  2. Directory inheritance: If no rule matches, the file inherits the SELinux context of its parent directory.

This means files are always labeled when created, but the label may not always be correct if the file is placed in an unexpected location.

Copy vs. Move: A Critical Difference

⚠Copy vs. Move Behavior

Copying a file always creates a new inode. The new file's context is determined by the destination directory's policy — the original context is not preserved unless you use cp --preserve=context.

Moving a file within the same filesystem moves the existing inode. The file keeps its original SELinux context. This can cause problems when moving files to a location where a different context is expected.

Demonstrating Copy vs. Move Context Behavior
# Create two test files in /tmp (inherit tmp_t context) [root@host ~]# touch /tmp/file1 /tmp/file2 [root@host ~]# ls -Z /tmp/file* unconfined_u:object_r:user_tmp_t:s0 /tmp/file1 unconfined_u:object_r:user_tmp_t:s0 /tmp/file2 # Move file1 and copy file2 to web content directory [root@host ~]# mv /tmp/file1 /var/www/html/ [root@host ~]# cp /tmp/file2 /var/www/html/ # Result: moved file KEPT its original wrong context! [root@host ~]# ls -Z /var/www/html/file* unconfined_u:object_r:user_tmp_t:s0 /var/www/html/file1 <-- WRONG unconfined_u:object_r:httpd_sys_content_t:s0 /var/www/html/file2 <-- Correct

Three Methods to Change SELinux Contexts

The recommended approach: create a persistent policy rule for a directory, then apply it. This survives filesystem relabels and can be re-applied at any time.

semanage fcontext — Persistent Context Policy
# Install required packages [root@host ~]# dnf install -y policycoreutils policycoreutils-python-utils # Step 1: Create a policy rule for /virtual and all files in it [root@host ~]# semanage fcontext -a -t httpd_sys_content_t '/virtual(/.*)?' # Step 2: Apply the policy rule to the actual files [root@host ~]# restorecon -RFvv /virtual Relabeled /virtual from ...default_t:s0 to ...httpd_sys_content_t:s0 Relabeled /virtual/index.html from ...default_t:s0 to ...httpd_sys_content_t:s0 # List all file context policy rules [root@host ~]# semanage fcontext -l | grep /virtual /virtual(/.*)? all files system_u:object_r:httpd_sys_content_t:s0 # Show only custom (locally added) policies [root@host ~]# semanage fcontext -l -C SELinux fcontext type Context /virtual(/.*)? all files system_u:object_r:httpd_sys_content_t:s0
OptionDescription
-a, --addAdd a new file context policy rule
-d, --deleteDelete an existing policy rule
-l, --listList all file context policy rules
-l -CList only local customizations (not default policy)

The restorecon command resets files to the context defined in the SELinux policy database. It is used after semanage fcontext creates or updates a rule, or to fix incorrectly labeled files.

restorecon — Apply Policy Contexts
# Fix context on a single file [root@host ~]# restorecon -v /var/www/html/mypage Relabeled /var/www/html/mypage from admin_home_t:s0 to httpd_sys_content_t:s0 # Fix contexts on all files recursively (-R) with verbose output (-v) [root@host ~]# restorecon -Rv /var/www/html/ Relabeled /var/www/html/file1 from user_tmp_t:s0 to httpd_sys_content_t:s0 # Force relabel, resetting all contexts including those already correct (-F) [root@host ~]# restorecon -RFvv /virtual
📋Note

restorecon relies on the policy rules in the SELinux database. If no rule exists for a path, it uses the parent directory's context. Always create the policy rule with semanage fcontext before running restorecon on a new custom directory.

The chcon command changes a file's SELinux context directly, without creating a policy rule. Changes made with chcon are temporary — they will be overwritten if restorecon is run or the filesystem is relabeled.

chcon — Temporary Context Change (Testing Only)
# Temporarily change context on /virtual directory [root@host ~]# chcon -t httpd_sys_content_t /virtual [root@host ~]# ls -Zd /virtual unconfined_u:object_r:httpd_sys_content_t:s0 /virtual # Running restorecon will REVERT this change! [root@host ~]# restorecon -v /virtual Relabeled /virtual from httpd_sys_content_t:s0 to default_t:s0
⚠Warning

Use chcon only for testing and debugging. In production, always use semanage fcontext + restorecon to ensure your changes survive relabeling operations.

The "Pirate Expression" in Policy Rules

When viewing semanage fcontext -l output, you will frequently see the pattern (/.*)? appended to directory names. This is called the pirate expression (it looks like a face with an eye patch and a hook) and means "this directory itself, and optionally any file path inside it."

Understanding the Pirate Expression (/.*)?
# This rule covers /var/www/html AND everything inside it /var/www/html(/.*)? all files system_u:object_r:httpd_sys_content_t:s0 # The regex breakdown: # /var/www/html - match the directory itself # ( - start of optional group # /.* - slash + any characters (any file path within) # )? - end of optional group - the whole group can exist or not # Always use quotes around the pattern with semanage! [root@host ~]# semanage fcontext -a -t httpd_sys_content_t '/virtual(/.*)?'

Tuning SELinux Policy with Booleans

SELinux Booleans On/off switches that enable or disable optional behaviors in an SELinux targeted policy. Booleans allow fine-tuning without modifying the policy source code. Each targeted application has its own set of Booleans. are named on/off switches that enable or disable optional features in an SELinux targeted policy. They allow administrators to customize application behavior without writing custom policy code. Each application with a targeted policy documents its available Booleans in its _selinux(8) man page.

📚Finding Boolean Documentation

Every targeted service has a dedicated SELinux man page. For example, man httpd_selinux documents all httpd Booleans, file contexts, and port contexts. Install the documentation with: dnf install selinux-policy-doc

Listing and Viewing Booleans

getsebool and semanage boolean — View Boolean Status
# List all Booleans and their current state [root@host ~]# getsebool -a abrt_anon_write --> off abrt_handle_event --> off ... httpd_enable_homedirs --> off httpd_can_network_connect --> off ... # Check a specific Boolean [root@host ~]# getsebool httpd_enable_homedirs httpd_enable_homedirs --> off # Detailed view: persistent value, current value, and description [root@host ~]# semanage boolean -l | grep httpd_enable_homedirs httpd_enable_homedirs (off , off) Allow httpd to enable homedirs # ^ ^ # current persistent (default)

Setting Booleans

Use setsebool to change a Boolean's state. Without -P, the change is temporary and reverts on reboot:

setsebool — Enable and Disable Booleans
# Temporarily enable httpd home directory sharing [root@host ~]# setsebool httpd_enable_homedirs on [root@host ~]# semanage boolean -l | grep httpd_enable_homedirs httpd_enable_homedirs (on , off) Allow httpd to enable homedirs # ^ ^ # current persistent (still off - reverts on reboot!) # Make the change PERSISTENT with -P flag [root@host ~]# setsebool -P httpd_enable_homedirs on [root@host ~]# semanage boolean -l | grep httpd_enable_homedirs httpd_enable_homedirs (on , on) Allow httpd to enable homedirs # ^ ^ # current persistent (both on - survives reboot) # List only customized Booleans (different from policy defaults) [root@host ~]# semanage boolean -l -C SELinux boolean State Default Description httpd_enable_homedirs (on , on) Allow httpd to enable homedirs

Common SELinux Booleans for the httpd service. Use the toggles to explore their state (simulation only):

Investigating and Resolving SELinux Issues

SELinux denials are logged and can be analyzed with several tools. Knowing how to read and act on SELinux audit messages is essential for day-to-day administration.

Troubleshooting Mindset

Before changing SELinux settings, consider these guiding principles:

  • Most denials mean SELinux is working correctly. A denial is often evidence of misconfiguration elsewhere, not an SELinux problem to be disabled.
  • The most common cause is a wrong file context. A file placed in the wrong location, or moved rather than copied, is responsible for most SELinux issues.
  • Never disable SELinux as a "fix." Putting SELinux into permissive mode for production is trading security for convenience.
  • Check the service's _selinux man page before modifying any Boolean or context.

Tools for Investigating SELinux Issues

The audit log contains raw AVC (Access Vector Cache) The kernel component that caches and logs SELinux access decisions. AVC messages in the audit log record the exact process, resource, and action involved in each denial. messages. Each entry shows the source context (the process), the target context (the resource), and the denied action.

Anatomy of an AVC Audit Log Entry
[root@host ~]# tail /var/log/audit/audit.log type=AVC msg=audit(1749508506.271:159): avc: denied { getattr } for pid=1984 comm="httpd" path="/var/www/html/mypage" scontext=system_u:system_r:httpd_t:s0 tcontext=unconfined_u:object_r:admin_home_t:s0 tclass=file permissive=0 # Key fields to note: # denied { getattr } - The denied action # comm="httpd" - The process that was denied # path="..." - The file that was accessed # scontext=...httpd_t:s0 - Source (process) context # tcontext=...admin_home_t - Target (file) context <-- PROBLEM: wrong type! # permissive=0 - Enforcing mode (1 = permissive, not blocking)

The setroubleshoot-server daemon processes AVC events and writes human-readable summaries to /var/log/messages, including a sealert command with a UUID for deeper analysis.

/var/log/messages — sealert Summary
[root@host ~]# tail /var/log/messages Jun 9 22:35:06 servera setroubleshoot[2175]: SELinux is preventing /usr/sbin/httpd from getattr access on the file /var/www/html/mypage. For complete SELinux messages run: sealert -l a0bca4b2-46a9-4252-b0cd-6de0dfcdd454

The sealert command provides a full analysis report including confidence-rated suggestions for resolution. Always evaluate the root cause before applying any suggestion.

sealert — Detailed Denial Analysis
[root@host ~]# sealert -l a0bca4b2-46a9-4252-b0cd-6de0dfcdd454 SELinux is preventing /usr/sbin/httpd from getattr access on the file /var/www/html/mypage. **** Plugin restorecon (99.5 confidence) suggests **** If you want to fix the label. /var/www/html/mypage default label should be httpd_sys_content_t. Then you can run restorecon. Do: # /sbin/restorecon -v /var/www/html/mypage **** Plugin catchall (1.49 confidence) suggests **** If you believe that httpd should be allowed getattr access on the mypage file by default. Then you should report this as a bug. # ausearch -c 'httpd' --raw | audit2allow -M my-httpd # semodule -X 300 -i my-httpd.pp Additional Information: Source Context: system_u:system_r:httpd_t:s0 Target Context: unconfined_u:object_r:admin_home_t:s0 Target Objects: /var/www/html/mypage [ file ]
⚠Confidence Ratings Are Not Always Right

The sealert confidence rating indicates how often that solution resolves similar events — not whether it is correct for your situation. In this example, the 99.5% confidence suggestion (restorecon) is correct because the file was moved from /root (which has admin_home_t) to the web content directory. The audit2allow suggestion (1.49%) would create a custom policy that permanently allows httpd to access home-type files — the wrong fix for the wrong reason.

The ausearch command queries the audit log with powerful filters. It is useful for finding AVC events programmatically or when you want raw data without the sealert analysis.

ausearch — Query Audit Log
# Search for AVC events from the recent past [root@host ~]# ausearch -m AVC -ts recent # Search for all AVC events today [root@host ~]# ausearch -m AVC -ts today # Search for AVC events from a specific process [root@host ~]# ausearch -m AVC -c httpd -ts recent # Run all events through sealert for analysis [root@host ~]# sealert -a /var/log/audit/audit.log

The RHEL web console (cockpit) provides a graphical interface for viewing and resolving SELinux issues. Navigate to SELinux in the left menu to see the current enforcement state and a list of active access control errors.

Each error entry shows the affected file and process, and includes a Solution details expansion. You can click Apply this solution to automatically run the recommended fix command. After correcting all issues, the error list should show "No SELinux alerts."

💡Tip

The web console's SELinux troubleshooter uses the same sealert data as the command-line tool. It is particularly useful during initial setup of new services when multiple context issues may appear simultaneously.

Common SELinux Scenarios and Solutions

SymptomLikely CauseSolution
Web server returns 403 Forbidden for a file in /var/www/html File was moved (not copied) from another location; wrong context inherited restorecon -v /var/www/html/filename
Service works with a custom port but SELinux blocks it Port not labeled with the correct SELinux port type semanage port -a -t http_port_t -p tcp 8888

💡Pattern to Remember

Process types end in _t and run in domains (e.g., httpd_t). File types end in _t and have contexts (e.g., httpd_sys_content_t). The policy contains rules of the form: allow [domain] [file_context]:file { read write };

Chapter Summary

In this chapter, you learned how SELinux enhances Linux security with Mandatory Access Control policies that operate alongside standard file permissions.

Key Concepts Reviewed

  • SELinux Architecture: SELinux adds MAC on top of DAC. It confines targeted processes into domains and enforces access rules using context labels. If no allow rule exists, access is denied.
  • Context Labels: Every resource has a four-field label (user:role:type:level). The type field drives all decisions in the targeted policy. View contexts with -Z on ps, ls, cp, and mkdir.
  • Modes: Enforcing (default; blocks and logs), Permissive (logs only; used for troubleshooting), Disabled (not supported in RHEL 10).
  • File Contexts: Moving a file preserves its context; copying assigns the destination's context. The recommended fix workflow is semanage fcontext -a then restorecon -Rv.
  • Booleans: Toggle optional application behavior with setsebool. Use -P to make changes persistent. Document Booleans are in each app's _selinux(8) man page.
  • Troubleshooting: Raw AVC events go to audit.log; summaries with UUIDs go to /var/log/messages. Use sealert -l UUID for analysis. Always fix the root cause, not just apply the suggested workaround.
