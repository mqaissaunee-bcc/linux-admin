Understanding RPM Packages

Red Hat Enterprise Linux uses the Originally "Red Hat Package Manager," RPM is now a recursive acronym standing for "RPM Package Manager" and is used by many Linux distributions. format for distributing software. Each RPM package is an archive containing the files to install, metadata about the package, and scripts that run during installation.

RPM Package Components

Each RPM package is an archive containing the following components:

üì¶ Package Files

The actual files (binaries, libraries, configuration files, documentation) that the package installs on your system.

üìã Metadata

Information about the package including name, version, release, architecture, description, dependencies, and licensing.

‚öôÔ∏è Scripts

Pre-install, post-install, pre-uninstall, and post-uninstall scripts that run during package transactions.

üîê GPG Signature

A cryptographic signature that verifies the package's authenticity and integrity. Red Hat signs all packages it releases.

RPM Package Naming Convention

RPM packages follow a standardized naming convention that provides important information about the package:

Package Name Structure
httpd - 2.4.63 - 1.el10 . x86_64 . rpm

Name The name of the software package httpd

Architecture The CPU architecture the package was built for x86_64, aarch64, noarch
  • x86_64: 64-bit Intel/AMD processors (most common)
  • aarch64: 64-bit ARM processors
  • ppc64le: IBM POWER processors (little-endian)
  • s390x: IBM Z mainframe systems
  • noarch: Architecture-independent packages (scripts, documentation)
  • i686: 32-bit Intel/AMD processors (legacy)
üí° GPG Signature Verification

Red Hat digitally signs all packages with GNU Privacy Guard - A cryptographic tool used to sign and verify software packages, ensuring they come from a trusted source and haven't been tampered with. keys. The RPM system verifies package integrity by confirming that the package is signed with the appropriate GPG key. The RPM system fails to install a package if the GPG signature does not match.

Installing, Updating, and Removing Software

The dnf command is the primary tool for managing software packages on RHEL 10. It automatically resolves Dependencies are other packages that a software package requires to function properly. DNF automatically identifies and installs all required dependencies. and handles complex package transactions.

Installing Packages

Use dnf install to download and install a software package along with all its dependencies:

# Install a single package [root@host ~]# dnf install httpd Dependencies resolved. ================================================================================ Package Arch Version Repository Size ================================================================================ Installing: httpd x86_64 2.4.63-1.el10 rhel-10-appstream-rpms 53 k Installing dependencies: apr x86_64 1.7.5-2.el10 rhel-10-appstream-rpms 132 k apr-util x86_64 1.6.3-21.el10 rhel-10-appstream-rpms 101 k ... # Install multiple packages [root@host ~]# dnf install httpd mariadb-server php # Install from a local RPM file [root@host ~]# dnf install ./package-1.0-1.el10.x86_64.rpm

Updating Packages

Use dnf update to update installed packages to newer versions:

# Update a specific package [root@host ~]# dnf update httpd # Update all installed packages [root@host ~]# dnf update # Check for available updates without installing [root@host ~]# dnf check-update # Update and remove obsolete packages [root@host ~]# dnf upgrade

Note: dnf update and dnf upgrade are now equivalent in DNF. Both will update packages and remove obsolete packages when necessary.

Removing Packages

Use dnf remove to uninstall a package and its unused dependencies:

# Remove a package [root@host ~]# dnf remove httpd Dependencies resolved. ================================================================================ Package Arch Version Repository Size ================================================================================ Removing: httpd x86_64 2.4.63-1.el10 @rhel-10-appstream-rpms 4.7 M Removing unused dependencies: httpd-core x86_64 2.4.63-1.el10 @rhel-10-appstream-rpms 40 k ... # Remove a package and keep its dependencies [root@host ~]# dnf remove --noautoremove httpd

Searching for Packages

Use dnf search to find packages by name or description:

# Search for packages by keyword [student@workstation ~]$ dnf search web server ========================= Name Matched: web, server ========================== httpd.x86_64 : Apache HTTP Server nginx.x86_64 : A high performance web server and reverse proxy server # Search in package names only [student@workstation ~]$ dnf search --all httpd # Find which package provides a file [student@workstation ~]$ dnf provides /etc/httpd/conf/httpd.conf httpd-2.4.63-1.el10.x86_64 : Apache HTTP Server Repo : rhel-10-for-x86_64-appstream-rpms Matched from: Filename : /etc/httpd/conf/httpd.conf

Use dnf info to display detailed information about a package:

Managing Software Repositories

Repository configuration files are stored in /etc/yum.repos.d/. You can enable, disable, and add repositories using DNF commands or by editing these configuration files directly.

Repository Commands

Terminal # Enable a repository [root@host ~]# dnf config-manager --enable rhel-10-for-x86_64-supplementary-rpms # Disable a repository [root@host ~]# dnf config-manager --disable rhel-10-for-x86_64-supplementary-rpms # Add a new repository from a URL [root@host ~]# dnf config-manager --add-repo https://example.com/repo/myrepo.repo # View repository information [student@workstation ~]$ dnf repoinfo rhel-10-for-x86_64-baseos-rpms

Repository Configuration File Format

Each repository is defined in a .repo file with the following structure:

/etc/yum.repos.d/example.repo [example-repo] name=Example Repository baseurl=https://example.com/repo/$releasever/$basearch/ enabled=1 gpgcheck=1 gpgkey=https://example.com/RPM-GPG-KEY-example
Parameter Description
[repo-id] Unique identifier for the repository (used in commands)
name Human-readable name for the repository
baseurl URL to the repository directory containing packages
enabled 1 = enabled, 0 = disabled
gpgcheck 1 = verify GPG signatures, 0 = skip verification
gpgkey URL or path to the GPG public key for signature verification
üí° Variable Substitution

Repository URLs support variables like $releasever (RHEL version, e.g., "10") and $basearch (system architecture, e.g., "x86_64"). DNF automatically substitutes these with the appropriate values for your system.

Application Streams and Modules

Application Streams (AppStream) is a feature that allows multiple versions of user-space components to be delivered and updated more frequently than the core platform packages. allow RHEL to provide multiple versions of the same software simultaneously. This means you can choose which version of a programming language, database, or web server to install based on your application requirements.

Understanding Modules

A A module is a collection of RPM packages representing a component and its dependencies. Each module can have multiple streams (versions) available. is a set of RPM packages that belong together. Each module can have multiple streams (versions) available, and each stream can have different profiles (installation variants).

‚ÑπÔ∏è Module Stream Locks

Once you enable a module stream, the system is locked to that stream. To switch to a different stream, you must first reset the module with dnf module reset, then enable and install the new stream. This ensures consistent dependencies across your installed packages.

Package Groups and Environments

A package group is a collection of related packages that can be installed together. Groups simplify the installation of complete software stacks or development environments. are collections of packages that serve a common purpose. Installing a group installs all the packages in that group, making it easy to set up complete environments.

Working with Package Groups

Terminal # List all available groups [student@workstation ~]$ dnf group list Available Environment Groups: Server with GUI Server Minimal Install Workstation Available Groups: Container Management Development Tools Headless Management Network Servers System Tools # View packages in a group [student@workstation ~]$ dnf group info "Development Tools" Group: Development Tools Description: Basic development tools Mandatory Packages: autoconf automake gcc gcc-c++ make Default Packages: binutils ... # Install a package group [root@host ~]# dnf group install "Development Tools" # Remove a package group [root@host ~]# dnf group remove "Development Tools"

Transaction History

DNF maintains a history of all package transactions, allowing you to view past operations and even undo or redo them:
