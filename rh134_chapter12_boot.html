The Boot Loader: GRUB2

Modern computer systems require many hardware and software components to cooperate before a login prompt appears. Understanding this sequence is essential for diagnosing boot failures and configuring the system for specific startup behaviors.

Firmware: UEFI and BIOS

When a computer is powered on, it loads a firmwareFirmware is low-level software stored in ROM on the motherboard that initializes hardware and locates a boot loader on disk before handing off control to the operating system. boot system from a ROM chip on the motherboard. In modern x86 machines, this is typically either:

  • UEFI (Unified Extensible Firmware Interface)UEFI is the modern replacement for BIOS. It uses an EFI System Partition on disk and stores settings in NVRAM, allowing faster boot times and support for disks larger than 2 TB. — The standard in all machines manufactured since approximately 2020.
  • BIOS (Basic Input/Output System)BIOS is the older firmware standard that reads the first 512-byte sector (MBR) of a disk to find the boot loader. Common in machines manufactured before 2020. — The legacy firmware found in older hardware.

GRUB2 with UEFI

In a UEFI system, the boot firmware reads settings from NVRAMNon-Volatile RAM (NVRAM) is persistent memory on the motherboard that stores UEFI boot settings even when the system is powered off. to determine which disk and EFI applicationAn EFI application is a special executable stored on the EFI System Partition that UEFI firmware can load directly — GRUB2 generates an EFI application as its boot entry. to load.

The disk contains an EFI System Partition (ESP), typically mounted at /boot/efi. GRUB2 generates EFI-compatible files in this partition. After GRUB2 loads, it presents the boot menu.

💡Tip

To see the EFI System Partition on a running RHEL system: ls /boot/efi/EFI/redhat/. You will find grubx64.efi — the GRUB2 UEFI boot application.

GRUB2 with BIOS

In a BIOS system, the boot firmware reads the first sector of a disk — the Master Boot Record (MBR)The Master Boot Record is the first 512-byte sector of a disk. It contains a small boot loader (boot.img) and the partition table. GRUB2 installs boot.img here in BIOS systems. — which contains the boot.img image.

This boot.img loads core.img from the unpartitioned MBR gap space before the first partition. The core.img image loads additional modules to read the filesystem and locate the GRUB2 configuration. GRUB2 installs and manages these images automatically.

⚠️Warning

Never write raw data to the beginning of a BIOS-boot disk without backing up the MBR first. Overwriting boot.img will prevent the system from booting.

The GRUB2 Boot Menu

After GRUB2 initializes and loads disks, it presents a boot menu. From this menu you can:

  • Select which kernel version to boot (multiple entries if several kernels are installed).
  • Edit the kernel command-line arguments for a single boot by pressing E.
  • Press C to access the GRUB2 command line for advanced troubleshooting.
💡Tip

Changes made in the GRUB2 editor (via E) affect only that one boot instance. Use grubby to make persistent changes, as described in the next section.

Managing GRUB2 with grubby and Kernel Arguments

The grubbygrubby is a command-line tool for reading and modifying GRUB2 boot-menu entries. It can change the default boot entry, add or remove kernel arguments, and display entry details. command is the standard RHEL 10 tool for making persistent changes to GRUB2 configuration. It operates on the boot-loader entries in /boot/loader/entries/.

Viewing Boot Entry Information

To view all boot entries and their details:

Viewing a specific boot entry
[root@host ~]# grubby --info 1 index=1 kernel="/boot/vmlinuz-6.12.0-55.12.1.el10_0.x86_64" args="ro crashkernel=2G-64G:256M,64G-:512M $tuned_params" root="UUID=15507695-22bb-4c65-94e6-a438e095983f" initrd="/boot/initramfs-6.12.0-55.12.1.el10_0.x86_64.img $tuned_initrd" title="Red Hat Enterprise Linux (6.12.0-55.12.1.el10_0.x86_64) 10.0" id="b70d796f931842cb9776f658cd068435-6.12.0-55.12.1.el10_0.x86_64"

The key fields in this output are:

  • index — The boot-menu position number, starting at 0.
  • kernel — Path to the vmlinuzvmlinuz is the compressed Linux kernel executable. The "z" suffix indicates it is compressed with gzip. GRUB2 decompresses and loads it into memory during boot. kernel image GRUB2 will load.
  • args — Kernel command-line arguments passed at boot.
  • root — The block device (by UUID) containing the kernel and initramfs.
  • initrd — Path to the initramfsThe initramfs (initial RAM filesystem) is a compressed archive loaded into memory by GRUB2. It contains drivers, scripts, and a minimal root filesystem needed to mount the real root partition before systemd takes over. image.
  • title — The display name shown in the GRUB2 boot menu.

Changing the Default Boot Entry

Setting and verifying the default boot entry
# Set entry index 0 as the default [root@host ~]# grubby --set-default-index 0 The default is /boot/loader/entries/ffffffffffffffffffffffff-6.12.0-55.9.1.el10_0.x86_64.conf with index 0 and kernel /boot/vmlinuz-6.12.0-55.9.1.el10_0.x86_64 # Verify the change [root@host ~]# grubby --default-index 0 [root@host ~]# grubby --default-kernel /boot/vmlinuz-6.12.0-55.9.1.el10_0.x86_64

Adding and Removing Kernel Arguments

Kernel command-line argumentsKernel command-line arguments are options passed to the Linux kernel at boot time that control hardware detection, debugging output, and system behavior. They are stored in GRUB2 boot entries and can be modified with grubby. are passed to the kernel at boot and can control behavior such as whether a graphical splash screen appears, how the kernel handles crashes, and which init system to use.

Managing kernel arguments with grubby
# Add rhgb (graphical boot) and quiet (suppress messages) to a kernel [root@host ~]# grubby --update-kernel /boot/vmlinuz-6.12.0-55.9.1.el10_0.x86_64 \ --args="rhgb quiet" # Remove those arguments from the same kernel [root@host ~]# grubby --update-kernel /boot/vmlinuz-6.12.0-55.9.1.el10_0.x86_64 \ --remove-args="rhgb quiet" # Apply a kernel argument to ALL installed kernels [root@host ~]# grubby --update-kernel ALL --args="crashkernel=512M"
📝Note

The rhgb (Red Hat Graphical Boot) and quiet arguments suppress kernel boot messages. During troubleshooting, removing these arguments lets you see detailed boot messages that reveal the cause of failures.

The RHEL 10 System Initialization Sequence

Understanding the complete boot sequence lets you identify exactly where a failure is occurring. The RHEL 10 initialization process moves through several distinct stages, each responsible for progressively more of the running system.

Complete Boot Sequence

1

Firmware (UEFI / BIOS)

Power-on self-test (POST), hardware initialization, firmware reads NVRAM (UEFI) or MBR (BIOS) to locate the boot loader.

↓
2

GRUB2 Boot Loader

Loads vmlinuz kernel and initramfs image from disk into memory. Presents the boot menu. Passes kernel command-line arguments and the initramfs location to the kernel.

↓
3

Kernel Initialization

The kernel decompresses itself, detects and initializes hardware using drivers from initramfs, then executes /sbin/init from the initramfs as PID 1PID 1 is the first process started by the kernel. It is the ancestor of all other processes. In RHEL 10, /sbin/init is a symbolic link to systemd, making systemd PID 1.. In RHEL 10, /sbin/init is a symlink to systemd.

↓
4

initramfs systemd (initrd.target)

systemd from the initramfs runs all units for initrd.target, which includes mounting the real root filesystem from disk onto /sysroot. The kernel then pivots (switches) to the real root filesystem.

↓
5

systemd (Disk) — Default Target

systemd re-executes itself from the installed copy on disk, then activates the default target (e.g., multi-user.target or graphical.target), starting all required services and resolving dependencies automatically.

↓
6

Login Prompt

The system presents a text-based or graphical login screen, completing the boot process.

💡Tip

You can inspect the contents of the initramfs image with lsinitrd. This shows exactly which kernel modules and scripts are loaded into memory before the real root filesystem is mounted:

lsinitrd /boot/initramfs-$(uname -r).img | head -40

Where Failures Occur

Knowing the sequence above makes failure diagnosis much easier. Different symptoms indicate failures at different stages:

  • Black screen, no POST — Hardware or power failure before firmware even runs.
  • GRUB2 menu does not appear — Boot loader is missing or corrupted (MBR or EFI entries damaged).
  • "kernel panic" — Kernel stage failure; often due to missing driver or corrupted initramfs.
  • System drops to emergency shell during boot — systemd failed to mount a filesystem listed in /etc/fstab — the most common exam scenario.
  • Services fail but login works — Individual systemd unit failure, not a boot-loader issue.

systemd Targets: Selecting the System State

A systemd target is a named collection of units that together define a specific operating state. Targets replace the older concept of runlevelsRunlevels were a SysVinit concept (0–6) that defined discrete system states. systemd targets are more flexible and can have dependencies on other targets, unlike the strictly numbered runlevel system. from SysVinit.

Common systemd Targets

Click a target card to learn more about each target and how to use it:

🖥️
graphical.target

Multiple users, graphical & text login

💻
multi-user.target

Multiple users, text-only login

🛠️
rescue.target

Single-user, basic services running

🚨
emergency.target

Minimal system, root FS read-only

graphical.target

This is the standard desktop target. It includes multi-user.target plus a display manager service (like GNOME Display Manager). Most workstations and systems with a graphical interface default to this target.

To switch at runtime: systemctl isolate graphical.target

To set as default: systemctl set-default graphical.target

Supports AllowIsolate=yes — can be used with systemctl isolate.

multi-user.target

This is the standard server target. All system services run, but there is no graphical display manager. SSH, networking, and other services are fully available. Most headless servers default to this target.

To switch at runtime: systemctl isolate multi-user.target

To set as default: systemctl set-default multi-user.target

rescue.target

The rescue target waits for sysinit.target to complete before opening a root shell. This means logging services are running and most filesystems are mounted read/write. Use rescue mode when you need to fix configuration issues that don't prevent filesystem mounting.

To boot into rescue: Append systemd.unit=rescue.target to the kernel command line in GRUB2.

Requires root password to access the shell.

emergency.target

The emergency target starts the absolute minimal system — the root filesystem is mounted read-only. Use emergency mode when the system cannot reach rescue.target (for example, a damaged filesystem prevents sysinit.target from completing).

To boot into emergency: Append systemd.unit=emergency.target to the kernel command line in GRUB2.

To remount root read/write: mount -o remount,rw /

Requires root password to access the shell.

Managing the Default Target

Getting and setting the default systemd target
# Check the current default target [root@host ~]# systemctl get-default multi-user.target # Set graphical.target as the new default [root@host ~]# systemctl set-default graphical.target Removed '/etc/systemd/system/default.target'. Created symlink '/etc/systemd/system/default.target' → '/usr/lib/systemd/system/graphical.target'. # Verify the change [root@host ~]# systemctl get-default graphical.target
💡Tip

systemctl set-default works by creating a symbolic link at /etc/systemd/system/default.target. You can verify this with ls -l /etc/systemd/system/default.target.

Switching Targets at Runtime

The systemctl isolate command switches the running system to a different target immediately. It stops all services the new target does not need and starts any required services not yet running. Only targets where AllowIsolate=yes can be used with isolate.

Runtime target switching with systemctl isolate
# Switch to text-only mode (stops graphical services) [root@host ~]# systemctl isolate multi-user.target # Switch back to graphical mode [root@host ~]# systemctl isolate graphical.target # List all targets and their state [user@host ~]$ systemctl list-units --type=target --all # View target dependencies [user@host ~]$ systemctl list-dependencies graphical.target | grep target

Booting to a Different Target at Boot Time

To override the default target for a single boot — for example, to enter rescue mode after a misconfiguration — append the target to the kernel command line in GRUB2:

  1. Boot or reboot the system.
  2. At the GRUB2 menu, press any key (except Enter) to interrupt the countdown.
  3. Select the kernel entry to modify, then press E to edit it.
  4. Move the cursor to the line beginning with linux (the kernel command line).
  5. Append: systemd.unit=rescue.target (or emergency.target).
  6. Press Ctrl+X to boot with these changes.
🔴Important

This change is temporary — it only affects the current boot. On the next reboot, the system returns to its configured default target.

Powering Off and Rebooting

System power and reboot commands
# Graceful shutdown (stops services, unmounts filesystems, powers off) [root@host ~]# systemctl poweroff # Equivalent shortcut: [root@host ~]# poweroff # Graceful reboot [root@host ~]# systemctl reboot # Equivalent shortcut: [root@host ~]# reboot # Halt (stops system but does NOT power off — for manual power-off) [root@host ~]# systemctl halt

Repairing Damaged File Systems at Boot Time

File-system issues are one of the most common causes of boot failures in RHEL. During boot, systemd reads /etc/fstab and attempts to mount all listed filesystems. If a mount fails, systemd may drop to an emergency shell. Knowing how to diagnose and fix these issues is essential for the RHCSA exam.

Common Boot-Time File-System Failures

Corrupted File System

If systemd detects file-system corruptionFile-system corruption occurs when the on-disk data structures become inconsistent, often due to improper shutdown, hardware failure, or power loss. Journal-based filesystems like ext4 and XFS can often recover automatically by replaying the journal., it attempts an automatic repair. For journal-based filesystems (XFS, ext4), this is usually a fast journal replay. For severe corruption, the system opens an emergency shell.

Tool for XFS: xfs_repair /dev/sdXN

Tool for ext4: fsck.ext4 -p /dev/sdXN

Nonexistent Device, UUID, or Mount Point

If an /etc/fstab entry references a UUID or device that does not exist, systemd waits (times out) then opens an emergency shell with a message like: Timed out waiting for device /dev/sda2.

This is the most common exam scenario. The fix is to either correct the UUID/device path in /etc/fstab, or comment out the offending entry, then run systemctl daemon-reload and mount --all to verify.

Missing Mount Point Directory

If the directory listed as the mount point in /etc/fstab does not exist, mount --all will report: mount point does not exist.

The fix is simple: create the missing directory with mkdir -p /path/to/mountpoint, then retry mount --all.

File-System Repair Tools

⚠️Warning

Always unmount a filesystem before running repair tools. Running xfs_repair or e2fsck on a mounted filesystem can cause data loss or further corruption.

Checking and repairing XFS and ext4 filesystems
# Check and repair an XFS filesystem (must be unmounted first) [root@host ~]# xfs_repair /dev/sdb1 Phase 1 - find and verify superblock... Phase 2 - using internal log Phase 3 - for each AG... ... done # Check and repair ext4 filesystem (-p = auto-fix minor issues) [root@host ~]# fsck.ext4 -p /dev/sdb2 /dev/sdb2: clean, 11/131072 files, 26929/524288 blocks # Note: fsck.xfs only satisfies boot requirements but does nothing # The real XFS repair tool is xfs_repair [root@host ~]# fsck.xfs /dev/sdb1 If you wish to check the consistency of an XFS filesystem or repair a damaged filesystem, see xfs_repair(8).

Step-by-Step Emergency Shell Recovery

When a system drops to an emergency shell due to an /etc/fstab error, follow this recovery procedure:

1

Log In to the Emergency Shell

At the prompt "Give root password for maintenance", enter the root password.

2

Check If Root Is Read-Only

Check root filesystem mount mode
[root@host ~]# mount | grep "on / " /dev/sda3 on / type xfs (ro,relatime,seclabel,...) # "ro" means read-only — remount it: [root@host ~]# mount -o remount,rw /
3

Try Mounting All Filesystems

Attempt to mount all /etc/fstab entries
[root@host ~]# mount --all mount: /olddata: can't find UUID=4d5c85a5-8921-4a06-8aff-80567e9689bc.

This shows exactly which entry is failing.

4

Edit /etc/fstab to Fix the Error

Comment out or fix the bad entry in /etc/fstab
[root@host ~]# vi /etc/fstab # Comment out the offending line: # UUID=4d5c85a5-8921-4a06-8aff-80567e9689bc /olddata xfs defaults 0 0
5

Reload systemd and Verify

Reload systemd configuration and verify mounts
[root@host ~]# systemctl daemon-reload [root@host ~]# mount --all # No errors = success
6

Reboot to Verify Full Boot

Reboot the system
[root@host ~]# systemctl reboot
📝Note

The nofail mount option in /etc/fstab allows the system to continue booting even if that filesystem cannot be mounted. This is useful for non-critical removable media but should never be used for production filesystems that applications depend on.

Example: UUID=xxxx /backup xfs defaults,nofail 0 0

Chapter Summary

This chapter covered the complete RHEL 10 boot process, from firmware initialization through GRUB2 and the systemd initialization sequence, with a focus on control, customization, and troubleshooting.

Key Concepts Recap

  • GRUB2 is the default boot loader for RHEL 10, supporting both UEFI (EFI System Partition) and BIOS (MBR) firmware.
  • The grubby command makes persistent changes to GRUB2 boot entries: view entries (--info), set the default (--set-default-index), and manage kernel arguments (--args, --remove-args).
  • Kernel arguments modified at the GRUB2 editor (E) are temporary (one boot only); use grubby --update-kernel for permanent changes.
  • The RHEL 10 boot sequence: Firmware → GRUB2 → Kernel → initramfs systemd → root pivot → disk systemd → default target.
  • systemd targets define the system operating state. Use systemctl get-default and systemctl set-default to manage the default; use systemctl isolate for runtime switching.
  • rescue.target provides a root shell with logging and filesystems mounted; emergency.target provides the most minimal environment with the root filesystem read-only.
  • Boot into special targets by appending systemd.unit=target.target to the kernel command line in GRUB2 at boot time.
  • Common /etc/fstab errors (bad UUID, missing device, missing mount point) cause boot failures recoverable from the emergency shell.
  • After editing /etc/fstab, always run systemctl daemon-reload then mount --all to verify before rebooting.
  • xfs_repair repairs XFS filesystems; fsck.ext4 (or e2fsck) repairs ext4 filesystems. Both require the filesystem to be unmounted first.
