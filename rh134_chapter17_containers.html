Introduction to Containers

What Is a Container?

In computing, a container An encapsulated, isolated process that bundles an application along with all its required runtime dependencies. Containers share the host OS kernel but are isolated from each other using Linux kernel features. is an encapsulated process that includes all the required runtime dependencies for a program to run. Within a container, application-specific libraries are independent of the host operating system's libraries. The host OS and kernel still provide functions that are not specific to the containerized application — helping ensure that containers remain compact and can quickly start and stop.

A container engine Software responsible for creating, running, and managing containers. Examples include Podman (default in RHEL 10), Docker, and CRI-O. A container engine creates a union file system by merging image layers. creates a union file system A file system that merges multiple directory layers into a single unified view. Container engines use this to stack immutable image layers, then add a single writable layer for runtime changes. by merging container image layers A component of a container image representing a specific set of files or changes (e.g., system libraries, binaries, config files). Layers are immutable and stack on top of each other to form the complete image.. Because image layers are immutable (read-only), the container engine adds a single writable layer for runtime file modifications. Containers are ephemeral Short-lived or temporary. Container storage is ephemeral by default, meaning all changes to the writable layer are lost when the container is removed. Persistent data requires volumes. by default — when you remove a container, the writable layer and any changes stored in it are discarded.

Core Linux Technologies Behind Containers

Containers are not a new operating system feature — they are a clever combination of existing Linux kernel capabilities. Understanding these building blocks helps you reason about container security and behavior.

🕵
Namespaces
Provide process isolation by giving each container its own view of the system. Different namespace types isolate PIDs, network interfaces, mount points, hostnames, and user IDs.
⚙
Control Groups (cgroups)
Limit and account for resource usage (CPU, memory, disk I/O, network). cgroups ensure that one container cannot monopolize host resources and starve others.
🔒
SELinux
Enforces mandatory access controls on container processes. SELinux prevents a compromised container from accessing files or resources it has no business touching, even if it escapes the namespace boundary.
📋
seccomp
Secure computing mode restricts which system calls a container process can make. This reduces the attack surface by blocking dangerous or rarely needed syscalls from inside containers.

Container Images vs. Container Instances

These two terms are often confused, but they describe very different things:

A container image contains immutable data, instructions, and libraries that define an application. Think of it as a template or blueprint. An image exists on disk and can be:

  • Downloaded from a registry and stored locally
  • Used to create many independent running instances
  • Tagged with version numbers to track releases
  • Shared and distributed through registries

Images themselves do not run — they are the specification for what should run.

A container instance (or simply a "container") is an executable, running version of an image. It includes:

  • A writable layer on top of the immutable image layers
  • Network interfaces and an IP address
  • References to volumes (optional, for persistent data)
  • A running process space (its own PID namespace)

You can run multiple container instances from the same image simultaneously, each isolated from the others. When a container is removed, its writable layer is deleted — the original image is untouched.

A helpful analogy from object-oriented programming: a container image is like a class, and a container instance is like an object.

  • A class (image) defines the structure and behavior
  • An object (instance) is a specific running realization of that class
  • You can instantiate many objects from one class, just as you can run many containers from one image

This relationship is also why the OCI defines two separate specifications: image-spec governs container images, while runtime-spec governs running container instances.

Containers vs. Virtual Machines

Containers and virtual machines (VMs) A software emulation of a complete computer system, including its own kernel, OS, and hardware interfaces. VMs are managed by a hypervisor (e.g., KVM, VMware, Hyper-V) and provide stronger isolation than containers. both provide isolated environments for running applications, but they do so at very different levels of the stack. The Open Container Initiative (OCI)

Containerization has a rich history, originating from technologies like the chroot utility. Today, the Open Container Initiative (OCI) A governance organization that defines open standards for container image formats (image-spec) and container runtimes (runtime-spec). OCI compliance ensures containers work across different engines and platforms. defines the standards that make containers interoperable across different engines. Because Podman, Docker, CRI-O, and other tools all conform to OCI specifications, a container image built with one tool will run correctly on another — no vendor lock-in.

Benefits and Challenges of Containers

  • Fast startup: Containers start in seconds (or milliseconds), ideal for dynamic scaling.
  • Resource efficiency: Containers use far fewer CPU and memory resources than VMs, enabling higher density on the same hardware.
  • Portability: A container runs consistently across development, testing, staging, and production environments.
  • Isolation: Application environments remain separate and predictable — a dependency conflict in one container does not affect another.
  • Cloud-native fit: Containers are the native deployment unit for Kubernetes and Red Hat OpenShift.
  • Rootless operation: With Podman, regular users can run containers without elevated privileges, improving the host system’s security posture.
  • Data persistence: Container storage is ephemeral. Applications that need persistent data require additional configuration using volumes A directory or storage area that exists outside the container's writable layer. Volumes persist data across container restarts and removals, and can be shared between containers..
  • Networking complexity: Routing traffic between multiple containers, or between containers on different hosts, requires careful planning.
  • Security management: Containers share the host kernel. A kernel vulnerability can potentially affect all containers. Rootless containers mitigate this but may limit access to certain system functions.
  • Orchestration overhead: At scale, managing hundreds or thousands of containers manually is impractical — orchestration platforms like Kubernetes are needed.

Running Containers with Podman

Why Podman?

Podman An open-source, daemonless container management tool that is the default container engine in RHEL 10. Podman can run, build, inspect, and manage OCI containers and images without requiring a background daemon or root privileges. is the default container management tool in Red Hat Enterprise Linux 10. Unlike tools that rely on a background daemon A background service process that runs continuously. In some container engines (like older Docker), all container operations are proxied through a central daemon, creating a single point of failure and a potential privilege escalation path., Podman is daemonless by design: it interacts directly with containers, images, and registries. This eliminates a single point of failure and avoids the security concerns of running a privileged background service.

Podman in RHEL 10 also supports rootless containers Containers started and managed by a non-root user. Rootless containers cannot access privileged kernel resources, limiting the damage if a container is compromised. They use user namespaces to map a container UID to a non-root host UID., allowing regular users to run containers without sudo or elevated privileges — a major security improvement over traditional container workflows.

Verify Podman installation
[student@workstation ~]$ podman -v podman version 5.4.0

Logging in to Container Registries

Before pulling images from authenticated registries like registry.redhat.io, you must authenticate using podman login. Credentials are stored in ${XDG_RUNTIME_DIR}/containers/auth.json in base64-encoded format (not encrypted — protect this file on shared systems).

Logging in to registries
# Authenticated registry (prompts for username and password) [student@workstation ~]$ podman login registry.redhat.io Username: your_rh_username Password: Login Succeeded! # Public registry (press Enter for empty credentials) [student@workstation ~]$ podman login registry.access.redhat.com Username: Password: Login Succeeded!
🔒 Security Note

The podman login command does not echo your password as you type. The registry configuration file (registries.conf) and the auth.json credentials file together control how Podman locates and authenticates to registries. Use podman info to view currently configured registries.

Running Your First Container

The podman run command is the primary way to start a container. It accepts an image name (local or remote) and optionally a command to run inside the container. If the image is not already present locally, Podman automatically pulls it from the configured registry.

Running a basic container
# Run a container and execute a command [student@workstation ~]$ podman run registry.redhat.io/ubi10 echo 'Hello World!' Trying to pull registry.redhat.io/ubi10:latest... Getting image source signatures ...output omitted... Hello World! # The container exits after the echo command completes

After the echo command finishes, the container stops — there are no more processes keeping it alive. This is normal behavior. To run a long-lived container without blocking your terminal, use the --detach or -d flag:

Running a detached container
# -d runs the container in the background; Podman prints the container ID [student@workstation ~]$ podman run -d registry.redhat.io/ubi10 sleep infinity 8699...c98b

Listing, Naming, and Inspecting Containers

Use podman ps to list running containers. By default it only shows running containers; add -a to see stopped ones too. If you do not assign a name with --name, Podman generates a random memorable name (like bold_chebyshev).

Exited (0) 7 min ago nervous_nobel 8699...1cae ubi10:... sleep infinity Up 53 seconds bold_chebyshev # Assign a meaningful name at creation time [student@workstation ~]$ podman run --name my_python registry.redhat.io/ubi9/python-312 which python /opt/app-root/bin/python

Podman provides a complete set of commands to control the container lifecycle. Understanding the signal semantics is important for graceful shutdown of services running in containers.

Lifecycle commands in action
# Stop a container gracefully [student@workstation ~]$ podman stop bold_chebyshev bold_chebyshev # Kill a container immediately (use when stop doesn't respond) [student@workstation ~]$ podman kill httpd httpd # Remove a stopped container [student@workstation ~]$ podman rm nervous_nobel fad3...00d3 # Run and auto-remove when done (great for one-shot tasks) [student@workstation ~]$ podman run --rm registry.redhat.io/ubi10 echo 'Hello World!' Exposing Container Ports

Containerized services like web servers listen on network ports inside the container’s private network namespace. To make these services accessible from the host or the network, you must explicitly map a host port to the container port using the -p host_port:container_port option with podman run.

Exposing a container port
# Map host port 8080 to container port 8080 [student@workstation ~]$ podman run -d -p 8080:8080 \ registry.redhat.io/rhel10/httpd-24:latest b7eb...636a # Verify the service is accessible on the host [student@workstation ~]$ curl 127.0.0.1:8080 <!DOCTYPE html> ...output omitted... <title>Test Page for the HTTP Server on Red Hat Enterprise Linux</title>
💡 Port Mapping Tip

The format is -p HOST_PORT:CONTAINER_PORT. You can map to a different host port: -p 9090:8080 would make a container’s port 8080 accessible on the host’s port 9090. You can specify multiple -p flags to expose multiple ports.

Creating and Managing Container Images

Working with Images: The Operations

Before you can run a container, you need an image. Podman provides a comprehensive set of image management operations. Images are stored locally in your user’s container storage (under ~/.local/share/containers/ for rootless usage).

🔎
podman search
Find images available in a registry by keyword. Always use a trailing slash when specifying a registry URL.
⬇
podman pull
Download a container image from a registry to local storage. Specify name:tag for a specific version, or omit the tag for :latest.
📷
podman image inspect
Display detailed metadata about an image in JSON format. Use --format with a Go template to extract specific fields.
🏷
podman tag
Assign a new name and optional tag to an existing image. Used to prepare an image for pushing to a specific registry location.
🚧
podman build
Build a new image from a Containerfile. Processes each instruction as a new layer, caching unchanged layers for efficiency.
⬆
podman image push
Upload a locally built or tagged image to a remote registry, making it available to others or to deployment systems.
🗑
podman rmi
Remove one or more images from local storage. Images referenced by a running container cannot be removed.
✂
podman image prune
Remove all dangling imagesUntagged images that are not referenced by any other image or container. They accumulate after rebuilds and take up disk space without being usable by name. (untagged, unreferenced images).

Searching and Pulling Images

Search, pull, and list images
# Search for images in a registry (trailing slash required) [student@workstation ~]$ podman search registry.redhat.io/rhel10/ NAME DESCRIPTION registry.redhat.io/rhel10/buildah rhcc_registry.access... registry.redhat.io/rhel10/toolbox rhcc_registry.access... registry.redhat.io/rhel10/httpd-24 rhcc_registry.access... ...output omitted... # Search for Universal Base Images (UBI) [student@workstation ~]$ podman search registry.redhat.io/ubi NAME DESCRIPTION registry.redhat.io/ubi8/ubi Provides the latest release... registry.redhat.io/ubi9/ubi rhcc_registry.access... registry.redhat.io/ubi10/ubi rhcc_registry.access... # Pull a specific image version [student@workstation ~]$ podman pull registry.redhat.io/rhel10/rhel-bootc:10.0 ...output omitted... a93f...d5b0 # List all locally stored images [student@workstation ~]$ podman images REPOSITORY TAG IMAGE ID CREATED SIZE registry.redhat.io/rhel10/rhel-bootc 10.0 a93f5ef0baa4 3 weeks ago 1.43 GB registry.redhat.io/rhel10/rhel-bootc latest a93f5ef0baa4 3 weeks ago 1.43 GB
🏷 Tags and the :latest Convention

Tags identify different versions of the same image. Common patterns include specific version numbers (10.0, 2.4) and the special latest tag, which refers to the most recently published version. When you omit a tag entirely (e.g., podman pull ubi10), Podman uses :latest automatically. Notice in the example above that both 10.0 and latest have the same Image ID — they are two tags pointing to the same underlying image.

Inspecting Images

The podman image inspect command returns a rich JSON document describing an image. You rarely need to read the entire JSON output — instead, use the --format option with a Go template A lightweight templating syntax from the Go programming language used by Podman and Docker to extract specific fields from JSON output. Fields are accessed with dot notation inside double curly braces: {{.FieldName}}. to extract exactly the fields you need.

Inspecting an image with --format
# Get the image name label [student@workstation ~]$ podman image inspect rhel-bootc:latest \ --format "{{.Config.Labels.name}}" rhel10/rhel-bootc # Get the image creation date [student@workstation ~]$ podman image inspect rhel-bootc:latest \ --format "{{.Created}}" 2025-05-15 15:00:38.956441305 +0000 UTC # Get the image description [student@workstation ~]$ podman image inspect rhel-bootc:latest \ --format "{{.Config.Labels.description}}" RHEL bootc container

Tagging Images

The podman tag command creates a new name and tag alias for an existing image. This is commonly used to prepare a local image for pushing to a specific registry location. Tagging does not create a copy — both the original and the new tag point to the same underlying image data.

Tagging a container image
# Tag by image ID [student@workstation ~]$ podman tag c6222576494f fedora:latest # Tag to prepare for pushing to a registry [student@workstation ~]$ podman tag localhost/my-httpd \ registry.lab.example.com:5000/my-httpd:1.0

Building Custom Images with a Containerfile

The real power of containers comes from building your own custom images. A Containerfile is a plain text file containing a sequence of instructions. Each instruction creates a new read-only layer in the image. The podman build command processes the Containerfile and produces a tagged image ready to run.

Click each line in the Containerfile below to learn what it does:

📄 Containerfile — Custom Apache Web Server (click any line to explain)
FROM ubi10/ubi
Sets the base image for all subsequent instructions. Every Containerfile must start with FROM. Here we use the Red Hat Universal Base Image 10, a free and redistributable foundation. All subsequent layers build on top of this.
RUN dnf install -y httpd
Executes a shell command inside a temporary container built from the layers so far, then commits the result as a new layer. Here we install the Apache httpd package. The -y flag auto-confirms the installation.
COPY index.html /var/www/html/index.html
Copies a file from the build context (the directory on your host) into the image. Here we copy a custom index.html to the Apache web root, giving our web server a custom page.
EXPOSE 80
Documents that the containerized application listens on port 80. This is metadata only — it does not actually open the port. You still need -p when running the container to make the port accessible from the host.
ENTRYPOINT ["/usr/sbin/httpd", "-DFOREGROUND"]
Defines the default executable that runs when the container starts. The exec form (JSON array) is preferred because it does not invoke a shell, ensuring signals like SIGTERM are delivered directly to the process.

Cleaning Up: Removing and Pruning Images

Remove and prune images
# Remove a specific image by ID or name [student@workstation ~]$ podman rmi localhost/my-httpd:latest Untagged: localhost/my-httpd:latest Deleted: 3067a8d97686... # Remove all dangling (untagged, unreferenced) images [student@workstation ~]$ podman image prune 84ada70bc51a... ...output omitted... # Remove ALL unused images (not just dangling) [student@workstation ~]$ podman image prune --all
⚠ Before You Prune

podman image prune --all removes every image that is not currently in use by a running container. If you have large images you downloaded (like the rhel-bootc image at 1.4 GB), they will be removed and will need to be pulled again. Use with caution in bandwidth-constrained environments.

Chapter Summary

In this chapter you learned to manage container images and run containers using Podman on Red Hat Enterprise Linux 10.

Key Concepts

  • A container is an isolated, encapsulated process that bundles an application with all its runtime dependencies.
  • Containers use Linux kernel features — namespaces (isolation), cgroups (resource limits), SELinux, and seccomp (security) — rather than full hardware virtualization.
  • A container image is an immutable, layered template; a container instance is a running version of that image with a writable layer added on top.
  • Podman is the default, daemonless container engine in RHEL 10. It supports rootless containers and is fully OCI-compliant.
  • Red Hat provides container images through registry.redhat.io (authenticated), registry.access.redhat.com (public), and registry.connect.redhat.com (partner program).
  • A Containerfile defines how to build a custom image layer by layer, starting from a base image with FROM.
  • Container storage is ephemeral by default — data written inside a running container is lost when the container is removed unless a volume is used.
