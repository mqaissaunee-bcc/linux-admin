Image Mode for Red Hat Enterprise Linux

Introduction to Image Mode

Image mode A deployment and management approach for RHEL 10 that uses container-native tools to build, deploy, and manage the entire operating system as a bootable container image rather than installing individual RPM packages. for Red Hat Enterprise Linux 10 is a new deployment approach that uses container-native tools to build, deploy, and manage the operating system. With this method, you can deploy the operating system from a bootable container (bootc) image A container image that includes not just application software but also the Linux kernel, boot loader, and systemd service manager — all components needed to boot a full operating system, not just run an application..

Container technology has proven successful at deploying applications. The same efficient technique of using layered images can be applied to entire hosts. A bootable container image must include the kernel, a boot loader Software that runs during system startup before the OS loads. GRUB2 is the standard boot loader in RHEL. It reads configuration, presents the boot menu, and transfers control to the Linux kernel., and other OS components typically excluded from application containers. Red Hat provides bootc base images for AMD/Intel x86_64 and 64-bit ARM architectures.

Package Mode vs. Image Mode

Package mode is the traditional RHEL installation method. The operating system is assembled from individual RPM packages Red Hat Package Manager format. RPM packages contain compiled software, metadata, and installation scripts. The dnf command manages these packages in RHEL. managed by dnf. System administrators can install, update, or remove individual components at any time. The root filesystem is mutable, meaning files can be changed on the running system.

  • Flexibility: Install or remove any package at any time.
  • Granular control: Each component can be updated independently.
  • Mutable root: Configuration files can be modified directly.
  • Risk of drift: Systems can diverge over time from their intended state.

Image mode treats the entire OS as a single versioned container image. Updates are delivered as new container images, not as individual package changes. The root filesystem is immutable Read-only; cannot be modified at runtime. In image mode RHEL, the root file system is immutable, meaning you cannot install or remove packages on the live system — all changes must be made by rebuilding the container image. by default.

  • Consistency: Every deployed system runs the exact same image.
  • Atomic updates: All changes apply as a single unit; no partial upgrades.
  • Rollback support: Instantly revert to a previous working image.
  • Immutable OS: Runtime tampering is prevented by design.

Creating Installable Images — The Containerfile Format

When using image mode for RHEL, you deploy software and configuration to hosts by using container images that conform to the Open Container Initiative (OCI) A governance organization that defines open standards for container image formats and runtimes. OCI compliance ensures images work across different engines and platforms. standard. A key piece of the build process is the Containerfile A text file containing a series of instructions used by Podman to build a container image layer by layer. Based on the Dockerfile format but follows the OCI standard. Each instruction creates a new image layer., a text file with build instructions. Each line in the Containerfile creates an image layer A read-only component of a container image representing a specific set of changes (packages installed, files copied, etc.). Layers stack on each other to form the complete image, enabling efficient storage and reuse..

Application Container Containerfile

The following is a standard application container Containerfile using the UBI base image:

Application Container Example
# Use the UBI10 base image FROM registry.redhat.io/ubi10 # Install httpd RUN dnf install -y httpd # Copy index.html to web root COPY index.html /var/www/html/index.html # Document the port the service listens on EXPOSE 80 # Run httpd in the foreground ENTRYPOINT ["/usr/sbin/httpd", "-DFOREGROUND"]

Bootable Container Containerfile

Building a bootable container requires a different base image. Red Hat provides the rhel-bootc Red Hat's official bootable container base image for RHEL 10. It includes the Linux kernel, GRUB2 boot loader, systemd service manager, and OSTree/composefs tooling needed to boot a full operating system. base image, which incorporates tooling from the OSTree An upgrade system for Linux-based operating systems that performs bootable, in-place upgrades of the OS, with rollback support. Used internally by bootc to store multiple file system deployments side by side., composefs An overlay file system that enables a truly read-only root file system by composing files from the OSTree repository. Used in RHEL image mode to enforce immutability of the root filesystem., and bootc upstream projects.

🔴Important

The rhel-bootc image is subject to the Red Hat Enterprise Linux EULA. You cannot publicly redistribute the base or any derivative user-created images.

Bootable Container Containerfile (rhel-bootc base)
# Use the rhel-bootc base image (includes kernel + boot loader) FROM registry.redhat.io/rhel10/rhel-bootc:latest # Install and clean packages in one layer RUN dnf -y install httpd mod_ssl && dnf clean all # Enable service via systemd (not ENTRYPOINT) RUN systemctl enable httpd && firewall-cmd --add-service=https # Add /etc configuration files ADD ./etc/ /etc # Copy web content COPY ./index.html /var/www/html/index.html

ADD vs. COPY Instructions

Both ADD and COPY place files into the image, but they behave differently:

  • COPY — Copies files or directories from the build context to the image. Does not unpack archives. Simple and predictable.
  • ADD — More powerful: can fetch files over the network, and automatically unpacks compressed archives (tar.gz, etc.) into the destination. Also supports copying from local files.
📖Note

The Containerfile(5) man page details the full specification. Many instructions accept additional options that allow significant image customization.

Deploying Image Mode Systems

A bootable container image must ultimately be deployed to a host that boots from it. Red Hat Enterprise Linux supports two primary deployment paths: bare-metal via Kickstart An automated installation system for Red Hat Enterprise Linux. A Kickstart file pre-answers all installer prompts (storage, users, network) so that installations proceed without manual interaction. automation, and cloud/virtualization via the bootc-image-builder A containerized Red Hat tool that converts a bootc container image into disk image formats (QCOW2, VMDK, VHD, AMI, raw) suitable for various cloud, virtualization, and edge environments. tool.

Bare-Metal Deployment with Anaconda Kickstart

For bare-metal servers, the recommended approach is the Anaconda installer's Kickstart feature. In image mode, you replace the traditional %packages section with the ostreecontainer A Kickstart command that specifies a bootable container image registry URL to use as the source for an image-mode RHEL installation, replacing the traditional %packages section. command.

Traditional Package Mode Kickstart (excerpt)
%packages @*minimal-environment vim-enhanced %end # rootpw and user creation work the same way rootpw --iscrypted $6$KUnFfrTz... user --name=student --groups=wheel --iscrypted --password=$6$JJL...
Image Mode Kickstart (excerpt)
# REPLACE %packages with ostreecontainer command ostreecontainer --url=registry.lab.example.com:5000/student/webserver-bootc # User management still uses standard Kickstart commands rootpw --iscrypted $6$KUnFfrTz... user --name=student --groups=wheel --iscrypted --password=$6$JJL... # %packages section is REMOVED entirely
⚠Warning

If the Kickstart file is not referenced correctly on the kernel command line with inst.ks=URL, Anaconda starts the graphical installer, which does not support image mode.

Upgrading the System

The update workflow for image mode systems consists of five steps:

  1. Edit the Containerfile in the bootc image project directory.
  2. Rebuild the container image using podman build --squash.
  3. Push the new image to the registry with the same name tag.
  4. Fetch the update on the target system using bootc upgrade.
  5. Reboot the system to apply the changes.

By default, automatic updates are enabled via the bootc-fetch-apply-updates A systemd service and timer unit that automatically checks the container registry for newer versions of the deployed bootc image and applies them. Enabled by default in image mode RHEL deployments. systemd timer unit. To disable automatic updates and manage them manually:

Manage Automatic and Manual Updates
# Check the automatic update timer [root@host ~]# systemctl status bootc-fetch-apply-updates.timer ● bootc-fetch-apply-updates.timer - Apply bootc updates Loaded: loaded (/usr/lib/systemd/system/bootc-fetch-apply-updates.timer; enabled) Active: active (waiting) since Wed 2025-04-09 13:22:18 EDT; 5min ago # Permanently disable automatic updates [root@host ~]# systemctl mask bootc-fetch-apply-updates.timer Created symlink '/etc/systemd/system/bootc-fetch-apply-updates.timer' -> '/dev/null'. # Manually fetch and stage the latest update [root@host ~]# bootc upgrade layers already present: 69; layers needed: 2 (34.5 MB) Fetching layers 2/2 Fetched layers: 32.95 MiB in 36 seconds (945.46 KiB/s) Queued for next boot: registry.lab.example.com/user/example-bootc Version: 10.20250116.0 Digest: sha256:dc6a...27bc Total new layers: 71 Size: 739.4 MB Added layers: 2 Size: 34.5 MB # Reboot to activate the new deployment [root@host ~]# systemctl reboot
📖Note

bootc upgrade and bootc update are aliases for the same operation. Use --check to inspect available updates without applying them, and --apply to reboot automatically after staging.

Rolling Back a System

The bootc rollback command reverts the system to the previous available image version. When executed, the rollback deployment is queued for the next boot, and the GRUB2 GRand Unified Bootloader version 2 — the standard boot loader in RHEL. In image mode, each OSTree deployment gets its own GRUB2 menu entry, enabling boot selection of previous OS versions. menu is updated to show the previous deployment as the first entry.

Roll Back to Previous Image Version
[root@host ~]# bootc rollback bootfs is sufficient for calculated new size: 0 bytes Next boot: rollback deployment [root@host ~]# systemctl reboot
💡Tip

During GRUB2 boot, you can also manually select a previous OSTree deployment A specific version of the OS file system stored by OSTree. Multiple deployments can coexist on a system, each with its own GRUB2 menu entry, enabling easy switching between OS versions. from the boot menu using the arrow keys.

File-System Layout in Image Mode

Image mode systems use a combination of immutable and mutable volumes. Understanding which directories are writable — and how they behave during upgrades — is essential for managing these systems correctly.

The Root File System: composefs

Image mode for RHEL uses composefs An overlay filesystem that enforces a read-only root in RHEL image mode. Data comes from OSTree deployments in /ostree/repo. This prevents runtime modifications to OS files. for the root file system by default. The data comes from an OSTree deployment A specific file system snapshot stored in /ostree/repo. OSTree can maintain multiple parallel deployments, allowing you to boot into different OS versions without removing the current one. stored in /ostree/repo. OSTree stores multiple file systems in parallel, enabling new images to be staged without deleting the running one.

Viewing the File System Layout with df -h
[root@host ~]# df -h Filesystem Size Used Avail Use% Mounted on /dev/sda3 9.0G 1.9G 7.2G 21% /sysroot composefs 8.4M 8.4M 0 100% / devtmpfs 4.0M 0 4.0M 0% /dev tmpfs 983M 0 983M 0% /dev/shm tmpfs 394M 488K 393M 1% /run tmpfs 983M 0 983M 0% /tmp /dev/sda2 960M 153M 808M 16% /boot tmpfs 197M 4.0K 197M 1% /run/user/1000

Note that the root filesystem (/) is mounted on composefs, and it shows 100% used — because it is read-only with no free space. The actual persistent storage is on /dev/sda3 mounted at /sysroot.

Directory Behavior Reference

/etc is mutable and persistent. bootc performs a 3-way merge A merge strategy that combines changes from three sources: the base version, local modifications, and the new upstream version. This preserves local customizations while still applying upstream changes to unmodified files. during upgrades.

  • Local changes to /etc are preserved across all future upgrades.
  • Unmodified files in /etc get updated if the new bootc image provides a newer version.
  • The ostree-finalize-staged.service executes these tasks during shutdown.
💡Tip

Store all persistent configuration in /etc. Your custom configs survive upgrades, and upstream config improvements are automatically applied to unmodified files.

/var is mutable and persistent, but NOT merged during upgrades.

  • Contents are copied from the container image during initial installation only.
  • After installation, /var is shared between all deployments on the system.
  • Existing files in /var are never changed or deleted during upgrades, even if the Containerfile instructs otherwise.
  • Rolling back with bootc rollback does not roll back /var contents.
⚠Warning

You cannot push changes to /var content using image upgrades or rollbacks. Application data stored in /var (databases, logs, user data) is always preserved and never reverted.

/usr is immutable (read-only). All OS binaries, libraries, and system software live here.

  • Cannot be modified on a running image mode system.
  • Changes to /usr require rebuilding the container image.
  • This immutability prevents unauthorized software installation and configuration drift.
  • After a bootc upgrade + reboot, /usr is atomically replaced by the new image version.

Chapter Summary

In this chapter, you learned to build, deploy, and manage Red Hat Enterprise Linux 10 in image mode.

  • Image mode is a container-native approach where the OS is deployed and managed as a bootable container image, enabling immutability, atomic updates, and easy rollbacks.
  • Use a Containerfile with the rhel-bootc base image to define the OS. Use systemctl enable instead of ENTRYPOINT, and firewall-cmd instead of EXPOSE.
  • Use podman build --squash to build the image and podman push to publish it to a registry.
  • Deploy to bare metal using Anaconda Kickstart with the ostreecontainer command (replace %packages).
  • Generate disk images for cloud/virtualization environments with bootc-image-builder using a config.toml for user customization.
  • The bootc command manages deployed systems: status, upgrade, rollback, and switch.
  • The root filesystem is immutable (composefs). /etc is mutable and 3-way merged on upgrade; /var is mutable and persistent but never updated by upgrades or rollbacks.
