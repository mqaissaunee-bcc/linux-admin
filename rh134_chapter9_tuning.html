TuneD: The System Tuning Daemon

TuneD is the primary performance-tuning tool in Red Hat Enterprise Linux. Rather than requiring administrators to manually adjust dozens of kernel parameters, TuneD bundles optimal settings into named tuning profiles that can be applied with a single command. It runs continuously as a daemon in the background and applies settings automatically at boot.

A Linux daemon that simplifies system performance tuning by applying predefined profiles of kernel parameters and system settings optimized for specific workloads, from desktops to high-performance servers. Named collections of kernel parameters, sysctl settings, and I/O tuning values bundled together for a specific workload type. Switching profiles applies all included settings at once. A background process that runs continuously and performs system services without direct interaction. TuneD runs as a systemd service and applies tuning settings on start and when profiles are switched.

Static Tuning (Default)

In RHEL 10, TuneD defaults to static tuning. When the TuneD service starts — which happens on every boot since TuneD is enabled by default — it reads the active profile and applies all its settings immediately. Those settings remain constant regardless of what the system is doing, providing a stable, predictable performance baseline.

A TuneD operating mode that applies kernel parameters once (at service start or profile switch) and holds them constant. The default mode in RHEL. Provides predictable, stable tuning without ongoing overhead.

Dynamic Tuning (Optional)

Dynamic tuning allows TuneD to continuously monitor system activity and adjust settings in real time as workloads change. It is disabled by default to ensure predictability. Dynamic tuning uses two types of plug-ins:

An optional TuneD mode that uses monitor plug-ins to gather real-time metrics (CPU load, disk I/O, network) and tuning plug-ins to adjust settings dynamically as workloads change. Disabled by default in RHEL.
  • Monitor plug-ins — gather real-time data such as CPU load, disk I/O, and network traffic.
  • Tuning plug-ins — use that data to make adjustments. For example, the net plug-in can dynamically change the interface speed based on usage.

To enable dynamic tuning, set dynamic_tuning = 1 in /etc/tuned/tuned-main.conf. The update_interval variable (in seconds) controls how frequently TuneD checks and adjusts settings.

/etc/tuned/tuned-main.conf — enabling dynamic tuning
# Dynamically tune devices; if disabled only static tuning will be used. dynamic_tuning = 1 ... # Update interval for dynamic tunings (in seconds). # Must be a multiple of sleep_interval. update_interval = 10

Installing and Enabling TuneD

TuneD is preinstalled in RHEL 10. If it is ever missing, install and enable it as follows:

Install and enable the TuneD service
[root@serverb ~]# dnf install tuned ...output omitted... [root@serverb ~]# systemctl enable --now tuned Created symlink /etc/systemd/system/multi-user.target.wants/tuned.service → /usr/lib/systemd/system/tuned.service.

Profile Storage Locations

TuneD stores profiles in two separate directories, which are important to understand when you need to customize settings:

DirectoryPurposeEditable?

💡Tip: Customizing Profiles

To customize a built-in profile, copy its directory from /usr/lib/tuned/profiles/ to /etc/tuned/profiles/ and modify the copy. The include directive in tuned.conf supports profile inheritance, so you can inherit all settings from a parent profile and override only the parameters you need.

A TuneD feature where a child profile uses the include directive to inherit all settings from a parent profile, then adds or overrides specific parameters. Allows creating specialized profiles without starting from scratch.

Inside a Profile: tuned.conf

Each profile directory contains a tuned.conf file that defines the settings. Sections correspond to tuning plug-ins. The [main] section holds the summary and optional include directive. The [sysctl] section applies kernel parameters via the sysctl mechanism.

Named modules in TuneD that handle specific subsystems. Examples: [sysctl] adjusts kernel parameters, [vm] tunes virtual memory, [disk] sets I/O scheduler options, [net] handles network settings.
/usr/lib/tuned/profiles/virtual-guest/tuned.conf
# # tuned configuration # [main] summary=Optimize for running inside a virtual guest include=throughput-performance [vm] # Start writeback at 30% dirty ratio (system default is 20%) dirty_ratio = 30 [sysctl] # Keep swapping low — filesystem I/O is usually more efficient than swap vm.swappiness = 30

The vm.swappiness parameter controls how aggressively the kernel moves process memory to swap space. Values range from 0 (prefer physical memory) to 100 (aggressively swap). You can check the current value with:

A kernel parameter (0-100) that controls the balance between using physical RAM and swap space. Lower values reduce swapping; the virtual-guest profile sets it to 30 to favor memory over swap.
Inspect the current vm.swappiness value
[root@serverb ~]# sysctl vm.swappiness vm.swappiness = 30

Managing Profiles with tuned-adm

The tuned-adm command is the primary CLI tool for interacting with the TuneD daemon. It covers everything from querying the current profile to recommending, switching, verifying, and deactivating profiles.

The command-line administration tool for TuneD. Used to list available profiles, query and switch the active profile, request a recommendation, verify current settings, and turn off all TuneD tuning.

Use tuned-adm active to see which profile is currently applied, and tuned-adm list to see every available profile (both system and custom).

Query the active profile and list all profiles
[root@serverb ~]# tuned-adm active Current active profile: throughput-performance [root@serverb ~]# tuned-adm list Available profiles: - accelerator-performance - Throughput performance based tuning... - aws - Optimize for aws ec2 instances - balanced - General non-specialized tuned profile - balanced-battery - Balanced profile biased towards power savings... - desktop - Optimize for the desktop use-case ...output omitted... Current active profile: throughput-performance

The tuned-adm verify subcommand confirms that the system settings currently match what the active profile specifies. Useful after a manual sysctl change that may have drifted from the profile.

Verify that current settings match the active profile
[root@serverb ~]# tuned-adm verify Verification succeeded, current system settings match the preset profile See TuneD log file ('/var/log/tuned/tuned.log') for details

Switch the active profile persistently with tuned-adm profile <name>. The change survives reboots because TuneD applies the active profile on every start.

Switch to the virtual-guest profile
[root@serverb ~]# tuned-adm profile virtual-guest [root@serverb ~]# tuned-adm active Current active profile: virtual-guest
🔴Profile Switch is Persistent

Unlike a manual sysctl change (which resets on reboot), switching profiles with tuned-adm profile is persistent. TuneD re-applies the profile at every boot, so there is no need to edit /etc/sysctl.conf for settings covered by TuneD.

tuned-adm profile_info displays the name and summary of the currently active profile. Specify a profile name as an argument to get information on any profile without switching to it.

Display information for a specific profile
[root@serverb ~]# tuned-adm profile_info network-latency Profile name: network-latency Profile summary: Optimize for deterministic performance at the cost of increased power consumption, focused on low latency network performance ...output omitted...

tuned-adm recommend analyzes system characteristics (is it a VM? a server? desktop?) and suggests the most appropriate profile. RHEL uses this mechanism to set the default profile at installation time.

Ask TuneD to recommend a profile
[root@serverb ~]# tuned-adm recommend virtual-guest

On a bare-metal server, this might return throughput-performance. On a desktop, it might return balanced. The recommendation is system-specific.

tuned-adm off deactivates TuneD entirely, reverting all settings applied by the active profile. Use this to troubleshoot whether TuneD is causing unexpected behavior.

Deactivate all TuneD settings
[root@serverb ~]# tuned-adm off [root@serverb ~]# tuned-adm active No current active profile.
⚠️TuneD Off Is Not Persistent

Running tuned-adm off only stops TuneD for the current session. On the next boot, if the TuneD service is still enabled, it will re-apply whatever profile was last set. To prevent TuneD from applying any settings at boot, disable the service with systemctl disable tuned.

Managing Profiles with the Web Console

RHEL 10 includes the Cockpit web console, which provides a browser-based GUI for managing TuneD profiles without needing the command line. Starting with RHEL 10, Cockpit is preinstalled.

A web-based graphical interface for Red Hat Enterprise Linux administration. Accessible at https://hostname:9090, it provides system monitoring, journal viewing, storage management, and TuneD profile control.

Installing and Enabling the Web Console

Install and enable Cockpit (if not already present)
[root@serverb ~]# dnf install cockpit ...output omitted... [root@serverb ~]# systemctl enable --now cockpit.socket Created symlink '/etc/systemd/system/sockets.target.wants/cockpit.socket' → '/usr/lib/systemd/system/cockpit.socket'. # If using a custom firewall, allow the cockpit service: [root@serverb ~]# firewall-cmd --add-service=cockpit --permanent success [root@serverb ~]# firewall-cmd --reload success

Accessing and Using the Web Console

Navigate to https://hostname:9090 in a web browser and log in with your system credentials. Cockpit uses a self-signed TLS certificate by default — you can safely dismiss the browser's certificate warning.

A TLS certificate signed by its own private key rather than a trusted commercial Certificate Authority. Provides encryption but not identity verification. Web browsers display a warning for self-signed certificates.

To manage TuneD profiles in Cockpit:

  1. Log in and click Turn on administrative access (or Limited access) and enter your password to escalate privileges.
  2. Click Overview in the left navigation bar. The Performance profile field shows the current active profile.
  3. Click the active profile link to open the Change performance profile dialog.
  4. Scroll through the profile list, click the desired profile, then click Change profile.
  5. Return to the Overview page and confirm the Performance profile field now shows the new profile.
💡Tip: Cockpit and tuned-adm are synchronized

Changing a TuneD profile via the web console has exactly the same effect as running tuned-adm profile <name> from the command line. Both methods are persistent, and changes made in one place are immediately visible in the other.

Process Scheduling in Linux

Even the most powerful multi-core servers can run more processes than they have CPU cores. The Linux process scheduler is the kernel component responsible for deciding which processes run, when, and for how long — efficiently distributing CPU time across all runnable tasks.

The part of the Linux kernel that allocates CPU time to threads. It uses scheduling policies and process priorities to determine the order in which threads run, ensuring fairness and responsiveness.

Preemptive Multitasking

The Linux kernel uses preemption to prevent any single process from monopolizing CPU time. The scheduler forcibly interrupts a running process after its time slice expires and dispatches the next runnable thread. On modern multi-core CPUs, multiple threads run simultaneously across cores, but the fundamental preemption principle still applies to ensure fairness.

The scheduler's ability to forcibly interrupt a running process and switch to a different one, even if the running process hasn't voluntarily yielded. This prevents CPU monopolization. A short interval of CPU time allocated to a process before the scheduler can preempt it and run another process. The EEVDF scheduler in RHEL 10 calculates this based on the process's virtual deadline and weight.

Scheduling Policies

The Linux kernel supports several scheduling policies, each designed for different types of workloads:

A set of rules the kernel scheduler uses to determine how a process competes for CPU time. Different policies suit different needs: normal interactive use, deterministic real-time responses, or batch processing.
PolicyAliasUse CaseNice Values?
SCHED_NORMALSCHED_OTHERStandard time-sharing for most user processes. Default policy.Yes
SCHED_FIFO(FF)Real-time first-in, first-out. Runs until it yields or is preempted by a higher-priority RT task.No
SCHED_RR(RR)Real-time round-robin. Like SCHED_FIFO but with a time slice so equal-priority RT tasks share the CPU.No
🔴Nice Values Only Affect SCHED_NORMAL

Nice values only influence processes under the SCHED_NORMAL policy. Real-time processes (SCHED_FIFO, SCHED_RR) belong to a higher-priority scheduling class and always preempt normal processes, regardless of how low a normal process's nice value is set.

EEVDF: The New Scheduler in RHEL 10

Starting in RHEL 10, the algorithm implementing SCHED_NORMAL is the Earliest Eligible Virtual Deadline First (EEVDF) scheduler, replacing the Completely Fair Scheduler (CFS) from earlier RHEL versions.

Earliest Eligible Virtual Deadline First — the scheduling algorithm used for SCHED_NORMAL processes in RHEL 10. Each task is assigned a virtual deadline based on how much CPU time it is owed and its nice-value weight. The task with the earliest eligible deadline runs next. Completely Fair Scheduler — the predecessor to EEVDF in earlier Linux kernels. Used a red-black tree ordered by virtual runtime to maintain CPU fairness. EEVDF extends this with explicit deadline calculations for better latency.

EEVDF assigns each task an eligible virtual deadline based on two factors: how much CPU time the task is still owed, and the task's weight (derived from its nice value). The scheduler always picks the runnable task with the earliest eligible deadline. Compared to CFS, EEVDF offers better latency handling and responsiveness because deadline calculations are integrated directly into the scheduling decision.

Nice values influence scheduling under EEVDF by adjusting a process's weight: a lower nice value (e.g., −10) gives a higher weight, which results in an earlier virtual deadline, so the process runs sooner and more often. A higher nice value (e.g., +10) gives a lower weight, resulting in a later deadline and less frequent scheduling.

Influencing Priority with Nice Values

The nice value is a user-visible parameter that adjusts how willing a process is to share CPU time with others. The scale runs from −20 (least nice, highest priority) to +19 (most nice, lowest priority), with a default of 0. A process with a lower nice value demands a larger share of CPU time; a process with a higher nice value voluntarily yields more.

A per-process integer from -20 (highest priority) to +19 (lowest priority) that influences how much CPU time the EEVDF scheduler allocates to the process. Default is 0. Only privileged users can decrease (raise priority) a nice value.

At the default nice value of 0, a process competes equally with other default-priority processes. The EEVDF scheduler assigns it a standard weight and virtual deadline.

Permission Rules

There is an important asymmetry in who can modify nice values:

  • Privileged users (root) can decrease a process's nice value (increase priority) or set any value from −20 to +19 for any process.
  • Unprivileged users can only increase the nice value of their own processes (lower priority). They cannot decrease a nice value once raised, and cannot modify another user's processes.

Inspecting Process Priorities with top

The top command shows two scheduling-related columns: PR (the scheduler's calculated priority) and NI (the nice value). For SCHED_NORMAL processes, PR = 20 + nice_value. Real-time processes show negative PR values.

top command — NI and PR columns
# Press q to quit top; press R to renice a process interactively [student@serverb ~]$ top top - 13:27:06 up 3 min, 3 users, load average: 0.45, 0.49, 0.22 Tasks: 250 total, 1 running, 249 sleeping, 0 stopped, 0 zombie %Cpu(s): 0.0 us, 4.3 sy, 0.0 ni, 95.7 id, 0.0 wa, 0.0 hi, 0.0 si MiB Mem : 5662.9 total, 4358.2 free, 1041.5 used, 511.4 buff/cache PID USER PR NI VIRT RES SHR S %CPU %MEM TIME+ COMMAND 4201 student 30 10 232212 2912 2400 R 100.0 0.1 8:08.03 sha1sum 4203 student 39 19 232212 2992 2480 R 90.9 0.1 8:03.72 sha1sum 4163 student 20 0 232212 2904 2392 R 81.8 0.1 8:37.92 sha1sum 2189 student 20 0 232016 4944 2896 R 8.3 0.1 0:00.02 top 1 root 20 0 49516 41284 10312 S 0.0 0.7 0:02.11 systemd # PR = 20 + NI for SCHED_NORMAL: nice 10 → PR 30; nice 19 → PR 39

Inspecting Process Priorities with ps

The ps command provides a static snapshot. Use -o to select specific columns including pid, priority (PRI), nice (NI), cls (scheduling class), and pcpu (CPU usage).

ps — view scheduling class, priority, and nice values
[student@serverb ~]$ ps -o pid,priority,nice,cls,pcpu,comm -C sha1sum PID PRI NI CLS %CPU COMMAND 2176 20 0 TS 99.2 sha1sum 2183 30 10 TS 99.1 sha1sum 2185 -11 - RR 99.3 sha1sum # CLS: TS=SCHED_NORMAL, RR=real-time round-robin; NI shows dash for RT processes

Sort all processes by nice value to find the highest- and lowest-priority processes on the system:

Sort all processes by nice value (descending)
[student@serverb ~]$ ps -eo pid,priority,nice,cls,pcpu,comm --sort=-nice | head PID PRI NI CLS %CPU COMMAND 60 39 19 TS 0.0 khugepaged 2185 39 19 TS 99.2 sha1sum 2183 30 10 TS 99.4 sha1sum 59 25 5 TS 0.0 ksmd 895 21 1 TS 0.0 rtkit-daemon 1 20 0 TS 0.2 systemd

Starting a Process with a Nice Value: nice

By default, a process inherits its parent's nice value (usually 0 from the shell). Use the nice command to start a new process with a specific nice value. Without -n, nice uses a default of 10.

A command that starts a new process with a specified nice value. Usage: nice -n VALUE COMMAND. Without -n, nice uses the default value of 10. Privileged users can set values from -20 to +19; unprivileged users can only set 0 to +19.
Start processes with specific nice values
# Start with default nice 0 (inherits from shell) [student@serverb ~]$ sha1sum /dev/zero & [1] 2794 [student@serverb ~]$ ps -o pid,nice,comm -C sha1sum PID NI COMMAND 2794 0 sha1sum # Start with nice default of 10 (no -n flag) [student@serverb ~]$ nice sha1sum /dev/zero & [2] 2810 # Start with explicit nice value of 15 [student@serverb ~]$ nice -n 15 sha1sum /dev/zero & [3] 2814 [student@serverb ~]$ ps -o pid,nice,comm -C sha1sum PID NI COMMAND 2794 0 sha1sum 2810 10 sha1sum 2814 15 sha1sum

Changing the Nice Value of a Running Process: renice

Use the renice command to adjust the nice value of an already-running process, identified by its PID.

A command that changes the nice value of a running process. Usage: renice -n VALUE PID. Unprivileged users can only increase (worsen priority) their own processes; decreasing nice values requires root. Process IDentifier — a unique integer assigned by the kernel to every running process. Used to identify processes in commands like kill, renice, and ps.
Change the nice value of a running process
# Increase nice value to 19 (lower priority — any user can do this) [student@serverb ~]$ renice -n 19 3033 3033 (process ID) old priority 10, new priority 19 # Attempt to lower nice value back — FAILS for unprivileged users [student@serverb ~]$ renice -n 10 3033 renice: failed to set priority for 3033 (process ID): Permission denied # Privileged users can freely lower nice values [root@serverb ~]# renice -n -5 3033 3033 (process ID) old priority 19, new priority -5
🔴Terminate CPU-Intensive Test Processes

When running CPU-intensive background processes (like sha1sum /dev/zero) for testing, always terminate them afterward to prevent resource exhaustion. Use pkill sha1sum or bring each job to the foreground with fg and press Ctrl+C.

Terminate all sha1sum test processes
[student@serverb ~]$ pkill sha1sum [1] Terminated sha1sum /dev/zero [2]- Terminated nice sha1sum /dev/zero [3]+ Terminated nice -n 15 sha1sum /dev/zero
📝Renicing with top

You can also change nice values interactively inside top. While top is running, press R to enter the renice mode. You will be prompted to enter the PID and the new nice value. This requires the same privilege rules as renice.

Chapter Summary

This chapter covered two independent mechanisms for tuning system performance in RHEL 10: the TuneD profile system for system-wide kernel parameter tuning, and nice values for influencing individual process CPU scheduling.

Key Concepts

  • TuneD applies predefined sets of kernel parameters via named profiles. Static tuning (the default) applies settings on service start and holds them constant. Dynamic tuning continuously adjusts settings based on real-time monitoring.
  • RHEL 10 ships with 15 built-in profiles covering power saving, general use, high performance, networking, and virtualization workloads.
  • Custom profiles live in /etc/tuned/profiles/ and can inherit from system profiles using the include directive. Never edit /usr/lib/tuned/profiles/ directly.
  • tuned-adm is the CLI tool for all TuneD operations: active, list, profile, profile_info, recommend, verify, and off.
  • The EEVDF scheduler (new in RHEL 10, replacing CFS) implements SCHED_NORMAL. It assigns eligible virtual deadlines to tasks based on their weight and owed CPU time.
  • Nice values (−20 to +19, default 0) influence a process's weight in the EEVDF scheduler. Lower nice = higher weight = earlier deadline = more CPU time.
  • Only privileged users can decrease nice values. Unprivileged users can only increase the nice values of their own processes.
