Processes and the Process Lifecycle

What Is a Process?

A processA running instance of an executable program, including allocated memory, security properties, execution threads, and a process state. is a running instance of a launched, executable program. From the moment a process is created, it consists of:

  • An address spaceA range of virtual memory addresses allocated to a process containing program code, data, stack, and heap segments. of allocated memory
  • Security properties, including ownership credentials and privileges
  • One or more execution threadsThe smallest unit of execution within a process. Threads share the same address space but execute independently. of program code
  • A process state

The environmentA list of information associated with a process, including local and global variables, scheduling context, and allocated system resources such as file descriptors and network ports. of a process includes local and global variables, a current scheduling context, and allocated system resources such as file descriptors and network ports.

The Process Lifecycle

An existing parent processThe process that creates a new (child) process through the fork system call. Its PID becomes the child's PPID. duplicates its own address space through a forkA system call that creates a new child process by duplicating the parent process. The child inherits the parent's environment, file descriptors, and security context. to create a new child processA process created by another process (the parent) through the fork system call, receiving its own unique PID.. Every process is assigned a unique PIDProcess ID. A unique integer number assigned by the kernel to identify each process. Process 1 (systemd) is the ancestor of all other processes. for tracking and security. Through the fork routine, a child inherits security identities, file descriptors, port and resource privileges, environment variables, and program code. The child can then execute its own program code via the exec system call.

Typically, a parent process sleeps while the child process runs, setting a wait request to be signaled when the child completes. After the child exits, it discards its resources and leaves a zombieA process that has completed execution but still has an entry in the process table. Exists until the parent process reads its exit status. entry in the process table. The parent, signaled to wake when the child exits, cleans the process table of the child's entry.

Linux Process States

In a multitasking OS, each CPU core can work on one process at a time. Processes are assigned a state, which changes as circumstances dictate. The S column of top or the STAT column of ps displays the current state.

StateFlagDescription
RunningRTASK_RUNNINGThe process is either executing on a CPU or waiting to run in the runnable queue.: Executing on a CPU or waiting to run. Can be executing user or kernel routines, or be queued and ready.
SleepingSTASK_INTERRUPTIBLEProcess is waiting for a condition. When an event or signal satisfies the condition, it returns to Running. Responds to signals.: Waiting for a condition. When an event or signal satisfies the condition, the process returns to Running.
DTASK_UNINTERRUPTIBLELike S, but does NOT respond to signals. Used when interruption might cause unpredictable device states. Cannot be killed with SIGKILL.: Also sleeping, but does not respond to signals. Used when interruption might cause unpredictable device states.
KTASK_KILLABLE: Same as D, but modified to allow the process to respond to a kill signal. Often displayed as D.
ITASK_REPORT_IDLE: A subset of D for kernel threads. Not counted in load average calculation.
StoppedTTASK_STOPPEDProcess is suspended, usually by user signal (Ctrl+Z sends SIGTSTP) or another process. Can be resumed with SIGCONT.: Suspended by a user or another process. Can be resumed by another signal.
ZombieZEXIT_ZOMBIEChild process has exited and signaled its parent. All resources except the PID are released. Remains until the parent reaps it.: Child has exited. All resources except PID released. Remains until parent reads exit status.
XEXIT_DEAD: Parent has reaped the remaining child process structure. Cannot be observed in process-listing utilities.
⚠️Important

A process in the D (uninterruptible sleep) state cannot be killed, even with SIGKILL. Many D-state processes indicate a possible hardware or storage problem.

Listing and Inspecting Processes

The ps command lists current processes. It supports three option styles: UNIX (dash), BSD (no dash), and GNU long (double dash).

Displays all processes. Key columns: USER, PID, %CPU, %MEM, VSZ, RSS, TTY, STAT, START, TIME, COMMAND.

[student@workstation ~]$ ps aux USER PID %CPU %MEM VSZ RSS TTY STAT START TIME COMMAND root 1 0.0 0.7 49516 41072 ? S 11:57 0:02 /usr/lib/systemd/systemd root 2 0.0 0.0 0 0 ? S 11:57 0:00 [kthreadd] student 2496 0.0 0.0 231132 3872 pts/0 R+ 16:45 0:00 ps aux

The STAT column shows process state. Additional characters: + (foreground process group), s (session leader), l (multi-threaded).

Includes the PPIDParent Process ID. The PID of the process that created (forked) this process. Useful for tracing process hierarchy. (Parent Process ID) column, useful for tracing parent-child relationships:

[student@workstation ~]$ ps -ef UID PID PPID C STIME TTY TIME CMD root 1 0 0 11:57 ? 00:00:02 /usr/lib/systemd/systemd student 2510 2450 0 16:46 pts/0 00:00:00 ps -ef

Use -o for custom columns and --sort to order output. Use pgrep to search by name or attribute:

[student@workstation ~]$ ps -eo pid,ppid,user,%cpu,%mem,stat,comm --sort=-%cpu | head [student@workstation ~]$ pgrep -l -u student 2450 bash 2510 ps
💡Tip

Use pgrep -l pattern instead of ps aux | grep pattern for cleaner output without the grep process itself appearing in results.

Managing Processes with Job Control

Foreground and Background Processes

A foreground processA process connected to the terminal that receives keyboard input. Only one foreground process per terminal. The shell waits for it to finish. receives keyboard input and controls the terminal. A background processA process running without terminal keyboard input. The shell does not wait for it to finish. Started by appending & to the command. runs without that connection, allowing continued terminal use.

Running Jobs in the Background

Append & to run a command in the background. The shell displays a job numberA number unique to the shell session identifying a background job. Shown in brackets, e.g., [1]. Referenced with %1, %2, etc. and PID:

[student@workstation ~]$ sleep 10000 & [1] 5947 [student@workstation ~]$ sleep 20000 & [2] 5948 [student@workstation ~]$ jobs [1]- Running sleep 10000 & [2]+ Running sleep 20000 &
Command / ShortcutDescription
command &Run a command in the background
jobsList background jobs for the current session
fg %nBring background job n to the foreground
bg %nResume a suspended job n in the background
Ctrl+ZSuspend the foreground process (sends SIGTSTP)
Ctrl+CTerminate the foreground process (sends SIGINT)
# Start a long-running process in the foreground [student@workstation ~]$ find / -name "*.conf" > /tmp/configs.txt 2>/dev/null # Press Ctrl+Z to suspend it ^Z [1]+ Stopped find / -name "*.conf" ... # Resume it in the background [student@workstation ~]$ bg %1 [1]+ find / -name "*.conf" > /tmp/configs.txt 2>/dev/null & # Bring it back to the foreground when ready [student@workstation ~]$ fg %1

Sending Signals to Processes

A signalA software interrupt delivered to a process to report events. Processes can catch, ignore, or use the default action for most signals. is a software interrupt delivered to a process. You can use keyboard shortcuts for the foreground process, or commands to signal any process by PID or name.

Fundamental Process Management Signals

Signal#DescriptionDefault
SIGHUP1Hangup. Many daemons reload their configuration upon receiving this signal.Terminate
SIGINT2Keyboard interrupt (Ctrl+C). Can be caught or ignored by the process.Terminate
SIGQUIT3Keyboard quit (Ctrl+\). Similar to SIGINT but produces a core dump.Core dump
SIGKILL9Kill, unblockable. Forces immediate termination. Cannot be caught, blocked, or ignored.Terminate
SIGTERM15Terminate. Default signal sent by kill. Allows the process to perform cleanup.Terminate
SIGCONT18Continue a stopped process. Sent automatically by fg or bg.Resume
SIGSTOP19Stop, unblockable. Cannot be caught, blocked, or ignored.Stop
SIGTSTP20Keyboard stop (Ctrl+Z). Unlike SIGSTOP, this can be caught.Stop
⚠️Warning

SIGKILL (9) should only be used as a last resort. It does not allow the process to clean up temporary files, close network connections, or report termination to child processes. Always try SIGTERM (15) first.

Commands for Sending Signals

Sends a signal to a process by PID. Default signal is SIGTERM (15).

[student@workstation ~]$ kill 5947 # SIGTERM (graceful) [student@workstation ~]$ kill -9 5947 # SIGKILL (forced) [student@workstation ~]$ kill -SIGKILL 5947 # Same as above [student@workstation ~]$ kill -l # List all signals

Sends a signal to all processes matching a command name. Exact match required.

[student@workstation ~]$ killall sleep # SIGTERM to all sleep processes [student@workstation ~]$ killall -9 httpd # SIGKILL to all httpd processes [student@workstation ~]$ killall -u student # SIGTERM to all student's processes

Like killall but uses pattern matching. Supports user, terminal, and parent PID filters.

[student@workstation ~]$ pkill sleep # Matches command name pattern [student@workstation ~]$ pkill -u student sleep # Only student's sleep processes [student@workstation ~]$ pkill -t pts/0 # All processes on terminal pts/0 [student@workstation ~]$ pkill -P 2450 # Kill children of PID 2450

Monitoring Process Activity and Load Average

Understanding Load Average

Load averageA measurement from the Linux kernel representing perceived system load. Shown as three numbers for the 1-minute, 5-minute, and 15-minute averages. Includes processes in Running (R) and Uninterruptible Sleep (D) states. is the Linux kernel's measurement of perceived system load over time. It counts the number of processes that are currently in the Running (R) and Uninterruptible Sleep (D) states. Load average is reported as three numbers: 1-minute, 5-minute, and 15-minute averages.

📝Interpreting Load Average

Load average must be compared to the number of CPU cores. To determine the number of logical CPUs:

[student@workstation ~]$ lscpu | grep "^CPU(s):" CPU(s): 4 [student@workstation ~]$ grep -c "^processor" /proc/cpuinfo 4
Scenario (4 CPUs)Load AvgMeaning
Idle0.0No active demand for CPU resources
Moderate2.050% utilized, resources available for spikes
Full4.0Fully utilized; no waiting, but no spare capacity
Overloaded8.0Overloaded; requests are waiting for CPU time

Tools for Checking Load

# uptime - quick load average check [student@workstation ~]$ uptime 16:55:40 up 5:00, 2 users, load average: 0.12, 0.24, 0.18 # w - logged-in users and load average [student@workstation ~]$ w 16:55:40 up 5:00, 2 users, load average: 0.12, 0.24, 0.18 USER TTY FROM LOGIN@ IDLE JCPU PCPU WHAT student pts/0 192.168.1.100 11:57 0.00s 0.06s 0.01s w

Comparing the three values (1, 5, 15 min) reveals load trends:

PatternMeaning
1 > 5 > 15Load is increasing. Recent activity spike. Monitor closely.
1 < 5 < 15Load is decreasing. System is recovering from peak load.
1 ≈ 5 ≈ 15Load is stable. Sustained workload over time.

Real-Time Monitoring with top

The top command provides a dynamic, real-time view of running processes. It combines process listing, load average, CPU usage, and memory statistics in a single interactive interface that updates every few seconds.

Hands-On Lab: Monitor and Manage Linux Processes

💻Lab Environment

This lab requires access to a RHEL system (physical, virtual machine, or classroom environment). You will practice listing processes, using job control, sending signals, and monitoring system load.

Step 1 List All Processes

View all running processes with BSD and UNIX format options.

[student@workstation ~]$ ps aux | head -15 [student@workstation ~]$ ps -ef | head -15

Compare the two outputs. Note the STAT column in ps aux and the PPID column in ps -ef.

Step 2 Start Background Processes

Create three background sleep processes and use jobs to view them.

[student@workstation ~]$ sleep 300 & [1] 12345 [student@workstation ~]$ sleep 400 & [2] 12346 [student@workstation ~]$ sleep 500 & [3] 12347 [student@workstation ~]$ jobs

Step 3 Practice Job Control

Bring a job to the foreground, suspend it, and resume it in the background.

[student@workstation ~]$ fg %2 sleep 400 # Press Ctrl+Z to suspend ^Z [2]+ Stopped sleep 400 [student@workstation ~]$ bg %2 [2]+ sleep 400 & [student@workstation ~]$ jobs

Step 4 Send Signals to Processes

Use kill and pkill to terminate processes.

# Gracefully terminate the first sleep [student@workstation ~]$ kill %1 [1] Terminated sleep 300 # Force kill the second sleep [student@workstation ~]$ kill -9 %2 [2] Killed sleep 400 # Kill remaining sleep processes by name [student@workstation ~]$ pkill sleep # Verify all sleep processes are gone [student@workstation ~]$ jobs

Step 5 Monitor System Load

Check system load and explore the top utility.

# Check current load average [student@workstation ~]$ uptime # Determine the number of CPUs [student@workstation ~]$ lscpu | grep "^CPU(s):" # Start top and explore [student@workstation ~]$ top # Press 1 to toggle per-CPU view # Press M to sort by memory # Press P to sort by CPU # Press u then type "student" to filter # Press q to quit

Step 6 Generate and Observe Load

Create CPU load and watch how it affects the system.

# Generate CPU load with dd (run in background) [student@workstation ~]$ dd if=/dev/urandom of=/dev/null & [student@workstation ~]$ dd if=/dev/urandom of=/dev/null & # Watch the load average rise [student@workstation ~]$ watch uptime # Press Ctrl+C to stop watching # Clean up: kill the dd processes [student@workstation ~]$ pkill dd # Verify load starts decreasing [student@workstation ~]$ uptime
✅Lab Complete

You have successfully listed processes, used job control, sent signals, monitored load average, and explored the top utility.

Chapter Summary

Key Takeaways

  • A process is a running instance of a program that includes an address space, security properties, execution threads, and a state. All processes descend from systemd (PID 1).
  • Processes use fork and exec to create new child processes. The parent sleeps while the child runs, then cleans up the zombie entry when the child exits.
  • Process states include Running (R), Sleeping (S/D/K/I), Stopped (T), and Zombie (Z). The ps and top commands display each process's current state.
  • ps aux and ps -ef are the most common ways to list processes. Use pgrep to search for processes by name or attribute.
  • Job control allows you to manage foreground and background processes using &, fg, bg, jobs, and Ctrl+Z.
  • Signals are software interrupts. Use SIGTERM (15) for graceful termination and SIGKILL (9) only as a last resort. The kill, killall, and pkill commands send signals.
  • Load average represents system busyness over 1, 5, and 15 minutes. Compare it to the number of CPU cores to determine if the system is overloaded.
  • top provides real-time process monitoring with interactive sorting, filtering, and process management capabilities.
