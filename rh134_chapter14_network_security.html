Linux Firewall Architecture

To effectively manage network security on RHEL 10, it helps to understand the layered architecture that governs how network packets are processed. There are three distinct layers, each building on the one below it.

The netfilter Framework (Kernel Layer)

The Linux kernel provides the netfilterA Linux kernel subsystem offering hooks for packet filtering, NAT, and connection tracking. All firewall tools ultimately write rules that netfilter enforces. framework for network traffic operations such as packet filteringThe process of inspecting network packets and deciding whether to allow, drop, or reject them based on criteria such as source/destination IP, port, and protocol., network address translation (NATNetwork Address Translation — a technique that rewrites source or destination IP addresses in packet headers, commonly used to share a single public IP across multiple private hosts.), and port translation. This is the lowest-level component — everything runs through here.

The nftables Framework (Rules Layer)

In RHEL 10, the nftablesThe successor to iptables/ip6tables in RHEL 10. Provides improved performance, unified IPv4/IPv6 handling, and more expressive rule syntax. It writes classification rules that the netfilter kernel framework enforces. packet classification framework is the system firewall core. It builds on netfilter to apply firewall rules to network traffic.

📋 nftables Replaces iptables

The nftables framework is the successor to the older iptables, ip6tables, and ebtables utilities. It offers many improvements in convenience, features, and performance. If you have existing iptables configuration files, you can convert them using the iptables-translate and ip6tables-translate utilities.

The firewalld Service (Management Layer)

Most administrators interact with neither netfilter nor nftables directly. Instead, RHEL 10 provides the firewalld service as a dynamic, high-level front end. You write rules using firewall-cmd or the web console, and firewalld translates those rules into nftables syntax and applies them.

netfilter

📋

Rules engine built on netfilter. Replaces iptables in RHEL 10. Rarely used directly — firewalld manages it automatically.
🛡

Dynamic firewall manager. The recommended interface for administrators. Uses zones, services, and ports as high-level concepts.

The firewalld Service

The firewalld service is a dynamic firewall managerA firewall that can update rules at runtime without interrupting established network connections or requiring a service restart, unlike static firewall tools., and is the recommended front end to the nftables framework. Red Hat Enterprise Linux 10 includes the firewalld package by default.

Zone-Based Traffic Classification

The firewalld service simplifies firewall management by classifying network traffic into zonesA named firewall policy profile in firewalld. Each zone defines which traffic is trusted, and multiple zones can be active simultaneously for different network interfaces or source addresses.. A network packet's assigned zone depends on criteria such as the source IP address or the incoming network interface. Each zone has its own list of ports and services that are permitted.

Zone Assignment Logic

When a packet arrives, firewalld determines which zone applies using the following priority order:

  1. Source address match: If the packet's source IP address is assigned to a specific zone, that zone's rules apply.
  2. Interface zone: If no source address match, the zone assigned to the incoming network interface is used.
  3. Default zone: If the interface has no zone assignment, the packet is sent to the default zone.

The default zone is not a separate zone — it is a designation applied to one of the existing zones. Initially, firewalld designates the public zone as the default, and maps the lo loopback interface to the trusted zone.

💡 NetworkManager Integration

For machines that frequently change networks, the NetworkManager service can automatically set the firewall zone for a connection. This is useful when switching between home, work, or public wireless networks. For example, you might want sshd reachable on your home network but blocked on a public coffee shop Wi-Fi.

Runtime vs. Permanent Configuration

A key concept in firewalld is the distinction between two configuration states:

⚡
Runtime Configuration
Currently active rules in memory. Changes take effect immediately but are lost on reboot or firewalld reload. Good for testing changes before making them permanent.
💾
Permanent Configuration
Rules stored on disk. Survive reboots, but only activate after a --reload. Use the --permanent flag with firewall-cmd to write to permanent config.
⚠ Always Reload After --permanent

When you use firewall-cmd --permanent, the rule is saved to disk but not yet active. You must follow up with firewall-cmd --reload to apply the permanent configuration as the new runtime. Alternatively, firewall-cmd --runtime-to-permanent saves the current runtime state to permanent storage.

If none of the predefined service configurations fit your needs, you can manually specify ports and protocols using --add-port=PORT/PROTOCOL. You can also create custom service definitions in /etc/firewalld/services/.

💡 Best Practice: Use Services Over Ports

Whenever possible, open firewall rules using named services (--add-service=http) rather than raw ports (--add-port=80/tcp). Service names make configurations more readable and maintainable. If a service requires multiple ports, all are added automatically.

Configuring firewalld

System administrators can interact with firewalld through two interfaces: the web console (graphical) and the firewall-cmd command-line tool. Both accomplish the same tasks; the CLI is more scriptable and works on headless servers.

Using the Web Console

To manage firewall services with the web console, log in and escalate privileges by clicking the Limited access or Turn on administrative access button. Enter your password when prompted — this elevates privileges based on your sudo configuration.

  1. Click Networking in the left navigation menu
  2. Locate the Firewall section on the networking page
  3. Click Edit rules and zones to open the Firewall management page
  4. The Firewall page shows active zones and their allowed services
  5. Click the Add services button in a zone to add new services
  6. In the Add Services dialog, scroll or filter to find a service, check its box, and click Add services
💡 Remember to Toggle Access Mode

After performing administrative tasks in the web console, toggle back to Limited access mode. Administrative mode grants elevated privileges for the duration of the session.

SELinux Port Labeling

Firewall rules control which hosts and ports can reach your system from the network. But a second layer of protection exists at the application level: SELinuxSecurity-Enhanced Linux — a mandatory access control system built into the Linux kernel. SELinux enforces security policies based on labels attached to files, processes, and network ports. port labeling controls which processes on your system are allowed to bind to which ports.

Why Port Labeling Matters

In addition to file context and process type labeling, SELinux assigns security context labelsA label in the format user:role:type:level assigned to SELinux subjects (processes) and objects (files, ports). Policy rules define what access is allowed between labels. to network ports. The targeted policy controls which targetedThe default SELinux policy mode in RHEL, where only specific high-risk processes run in confined domains. Most user processes run unconfined. processes can bind to which ports by matching process type labels against port type labels.

How Port Labels Work

Consider two common examples built into the targeted policy:

  • The SSH targeted policy assigns the ssh_port_t context label to port 22/TCP. The SSH daemon (sshd) runs with the sshd_t domain, and the policy permits sshd_t to bind to ssh_port_t ports.
  • The HTTP targeted policy assigns the http_port_t context to ports 80/TCP and 443/TCP. The httpd_t domain is permitted to bind to http_port_t ports.

This enforcement means that even if an attacker compromises a web server and tries to redirect traffic to an unusual port, SELinux will block the attempt — because the rogue process cannot bind to a port with an incompatible label.

⚠ SELinux Blocks Non-Standard Port Binding

When you configure a legitimate service to listen on a non-default port (e.g., running an HTTP server on port 8888), SELinux will block the service from starting unless port 8888 is labeled with the correct SELinux type (e.g., http_port_t). This is a very common cause of service startup failures on RHEL systems with SELinux enforcing mode.

The Diagnostic Workflow

When a service fails to start because of an SELinux port binding issue, the typical diagnostic workflow is:

  1. Attempt to start the service: systemctl start servicename
  2. Check the service status: systemctl status servicename
  3. Look for a “Permission denied” error on the port bind attempt in the output
  4. Use sealert -a /var/log/audit/audit.log to get a detailed SELinux alert with suggested remediation
  5. Use semanage port -l | grep service_name to find the correct port type
  6. Use semanage port -a -t port_type -p tcp|udp PORT to add the port label
Example: httpd fails to bind to port 1001
[root@serverb ~]# systemctl status httpd.service ✕ httpd.service - The Apache HTTP Server Loaded: loaded (/usr/lib/systemd/system/httpd.service; enabled; preset: disabled) Active: failed (Result: exit-code) ... Jul 14 12:54:15 serverb httpd[4202]: (13)Permission denied: AH00072: make_sock: could not bind to address [::]:1001 Jul 14 12:54:15 serverb httpd[4202]: (13)Permission denied: AH00072: make_sock: could not bind to address 0.0.0.0:1001 Jul 14 12:54:15 serverb httpd[4202]: no listening sockets available, shutting down # Permission denied on port 1001 is an SELinux block, not a Linux permission issue [root@serverb ~]# sealert -a /var/log/audit/audit.log SELinux is preventing /usr/sbin/httpd from name_bind access on the tcp_socket port 1001. #=== Plugin bind_ports (99.5 confidence) suggests ======================== If you want to allow /usr/sbin/httpd to bind to network port 1001 Then you need to modify the port type. Do # semanage port -a -t PORT_TYPE -p tcp 1001 where PORT_TYPE is one of the following: http_cache_port_t, http_port_t, jboss_management_port_t, jboss_messaging_port_t, ntop_port_t, puppet_port_t.

Managing SELinux Port Labels with semanage

The semanageThe SELinux policy management tool. Used to manage SELinux file contexts, port labels, user mappings, and other policy configurations. Listing Port Labels

Use semanage port -l to list all current port label assignments. The output is long — use grep to filter.

Listing SELinux port labels
# List all port labels (output is lengthy) [root@host ~]# semanage port -l ...output omitted... http_cache_port_t tcp 8080, 8118, 8123, 10001-10010 http_cache_port_t udp 3130 http_port_t tcp 80, 81, 443, 488, 8008, 8009, 8443, 9000 ...output omitted... # Filter by service name (e.g., ftp) [root@host ~]# semanage port -l | grep ftp ftp_data_port_t tcp 20 ftp_port_t tcp 21, 989, 990 ftp_port_t udp 989, 990 tftp_port_t udp 69 # Filter by a specific port number (exact word match) [root@host ~]# semanage port -l | grep -w 70 gopher_port_t tcp 70 gopher_port_t udp 70

Adding, Modifying, and Deleting Port Labels

The semanage port command supports four key operations: add (-a), modify (-m), delete (-d), and list (-l). These operations are summarized below, and then illustrated with a worked example using the gopher service on port 71/TCP.

⚠ Cannot Change Default Policy Labels

Most services in RHEL provide an SELinux policy module that includes the service's default port contexts. You cannot change these default port labels using semanage — the -a option only adds new labels on top of the policy. To change a default, you must modify and reload the targeted policy module, which is beyond the scope of this course. You can add additional ports to any existing label.

Add a new port label
# Syntax: semanage port -a -t PORT_TYPE -p tcp|udp PORT # Enable the gopher service to listen on the non-default port 71/TCP [root@host ~]# semanage port -a -t gopher_port_t -p tcp 71 # Verify the new label was added (show only local policy changes with -C) [root@host ~]# semanage port -l -C SELinux Port Type Proto Port Number gopher_port_t tcp 71
Modify an existing port label
# Change the port 71/TCP label from gopher_port_t to http_port_t # -m (modify) is more efficient than -d then -a [root@host ~]# semanage port -m -t http_port_t -p tcp 71 # Verify the change [root@host ~]# semanage port -l -C SELinux Port Type Proto Port Number http_port_t tcp 71 [root@host ~]# semanage port -l | grep http http_cache_port_t tcp 8080, 8118, 8123, 10001-10010 http_cache_port_t udp 3130 http_port_t tcp 71, 80, 81, 443, 488, 8008, 8009, 8443, 9000 pegasus_http_port_t tcp 5988 pegasus_https_port_t tcp 5989
Delete a port label
# Remove the binding of port 71/TCP from gopher_port_t [root@host ~]# semanage port -d -t gopher_port_t -p tcp 71

SELinux Man Pages for Services

Service-specific SELinux man pages are named using the service name plus _selinux (e.g., httpd_selinux(8)). These man pages contain detailed information about the service's SELinux types, Booleans, and port types. They are not installed by default.

Install SELinux policy documentation and search man pages
# Install the SELinux policy documentation package [root@host ~]# dnf -y install selinux-policy-doc # Search for available SELinux-specific man pages [root@host ~]# man -k _selinux | head -10 abrt_selinux (8) - Security Enhanced Linux Policy for the abrt processes apache_selinux (8) - Security Enhanced Linux Policy for the apache processes chronyd_selinux (8) - Security Enhanced Linux Policy for the chronyd processes httpd_selinux (8) - Security Enhanced Linux Policy for the httpd processes mysqld_selinux (8) - Security Enhanced Linux Policy for the mysqld processes ...output omitted... # Read the httpd-specific SELinux policy [root@host ~]# man httpd_selinux

Guided Exercise: Manage a Server Firewall and SELinux Port Labels

In this guided exercise, you will configure the firewall on serverb to allow HTTP traffic on a standard and a non-standard port, and configure SELinux to allow the web server to bind to the non-standard port.

⚠ Lab Environment Note

The Red Hat Online Learning environment requires port 5900/TCP to remain available for its graphical interface (associated with the vnc-server service). If you accidentally lock yourself out of serverb, recover via ssh student@serverb from workstation. The configuration includes a custom zone named ROL that opens these ports.

Objective: Confirm that the web server on serverb is inaccessible on both port 80 and port 1001.

Test connectivity from workstation
# Test standard HTTP port (80) -- should fail [student@workstation ~]$ curl http://serverb.lab.example.com curl: (7) Failed to connect to serverb.lab.example.com port 80 after 3 ms: Could not connect to server # Test virtual host on port 1001 -- also fails [student@workstation ~]$ curl http://serverb.lab.example.com:1001 curl: (7) Failed to connect to serverb.lab.example.com port 1001 after 2 ms: Could not connect to server

Both connections fail. The problem has two possible causes: the firewall is blocking the ports, or the web service is not running (or cannot start). Next steps will investigate both.

Objective: Log in to serverb, attempt to start httpd, and diagnose why it fails.

Log in to serverb and check httpd
[student@workstation ~]$ ssh student@serverb ...output omitted... [student@serverb ~]$ sudo -i [sudo] password for student: student # Check whether httpd is currently active [root@serverb ~]# systemctl is-active httpd inactive # Attempt to start and enable httpd -- this will fail [root@serverb ~]# systemctl enable --now httpd Created symlink '/etc/systemd/system/multi-user.target.wants/httpd.service' -> '/usr/lib/systemd/system/httpd.service'. Job for httpd.service failed because the control process exited with error code. See "systemctl status httpd.service" and "journalctl -xeu httpd.service" for details. # Check the detailed status [root@serverb ~]# systemctl status httpd.service ✕ httpd.service - The Apache HTTP Server Active: failed (Result: exit-code) ... Jul 14 httpd[4202]: (13)Permission denied: AH00072: make_sock: could not bind to address [::]:1001 Jul 14 httpd[4202]: (13)Permission denied: AH00072: make_sock: could not bind to address 0.0.0.0:1001 Jul 14 httpd[4202]: no listening sockets available, shutting down

The “Permission denied” error on port 1001 is an SELinux enforcement block. Port 1001/TCP has no http_port_t label, so SELinux prevents httpd from binding to it.

Objective: Use semanage to label port 1001/TCP with http_port_t so httpd can bind to it.

Add SELinux port label and start httpd
# Check current http port labels to confirm 1001 is missing [root@serverb ~]# semanage port -l | grep 'http' http_cache_port_t tcp 8080, 8118, 8123, 10001-10010 http_cache_port_t udp 3130 http_port_t tcp 80, 81, 443, 488, 8008, 8009, 8443, 9000 # Add port 1001/TCP to the http_port_t label [root@serverb ~]# semanage port -a -t http_port_t -p tcp 1001 # Confirm port 1001 is now labeled http_port_t [root@serverb ~]# semanage port -l | grep 'http_port_t' http_port_t tcp 1001, 80, 81, 443, 488, 8008, 8009, 8443, 9000 # Now start httpd -- it should succeed [root@serverb ~]# systemctl enable --now httpd [root@serverb ~]# systemctl is-active httpd active

Objective: Even with httpd running, traffic cannot reach the server until the firewall allows it. Check and update firewall rules.

Verify and update firewalld configuration
# Verify the default zone [root@serverb ~]# firewall-cmd --get-default-zone public # Check what's currently allowed -- note port 1001 is absent [root@serverb ~]# firewall-cmd --zone=public --list-all public (default, active) interfaces: ens3 sources: services: cockpit dhcpv6-client http ssh ports: ... # Port 80 is covered by the 'http' service. Add port 1001/TCP permanently. [root@serverb ~]# firewall-cmd --permanent --zone=public --add-port=1001/tcp success # Apply the change [root@serverb ~]# firewall-cmd --reload success # Verify the change [root@serverb ~]# firewall-cmd --zone=public --list-all public (default, active) interfaces: ens3 services: cockpit dhcpv6-client http ssh ports: 1001/tcp ...

Objective: Return to workstation and confirm both ports are now accessible.

Test access from workstation
[root@serverb ~]# exit [student@serverb ~]$ exit Connection to serverb closed. # Test port 80 -- should return "SERVER B" [student@workstation ~]$ curl http://serverb.lab.example.com SERVER B # Test port 1001 -- should return "VHOST 1" [student@workstation ~]$ curl http://serverb.lab.example.com:1001 VHOST 1

Both tests succeed. The web server is now accessible on both ports because the SELinux port label allows httpd to bind to port 1001, and the firewall allows traffic to reach both ports 80 and 1001.

Chapter Summary

In this chapter, you learned how to secure network connections to services using the system firewall, and how to enforce the SELinux targeted policy to protect service ports through port labeling.

Key Concepts

  • The Linux kernel's netfilter framework handles packet filtering. nftables provides the rules layer, and firewalld is the recommended high-level management tool in RHEL 10.
  • firewalld classifies traffic into zones. Zone assignment uses source IP first, then interface, then the default zone.
  • RHEL 10 defaults to the public zone as the default, allowing only ssh and dhcpv6-client services.
  • Rules added without --permanent are runtime-only and lost on reload/reboot. Use --permanent plus --reload for persistence.
  • SELinux port labeling assigns security context types to network ports. Processes can only bind to ports with compatible labels.
  • The semanage port -a -t TYPE -p PROTOCOL PORT command adds a new port to an existing SELinux label.
  • The semanage port -l -C command shows only local policy customizations (not default policy).
  • When a service cannot start with “Permission denied” on a port, the first troubleshooting step is to check SELinux with sealert or semanage port -l.
