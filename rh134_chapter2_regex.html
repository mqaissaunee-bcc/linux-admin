Introduction to Regular Expressions

A regular expression (regex or regexp) is a sequence of characters that defines a search pattern. Unlike simple string searches that look for exact matches, regular expressions can describe an almost unlimited variety of text patterns. They are a fundamental skill for any Linux system administrator. A formal pattern language used to describe sets of strings. Regular expressions are interpreted by an engine that searches for or matches text conforming to the pattern.

Regular expressions are supported natively by dozens of Linux tools including grep, sed, awk, vim, less, and many programming languages. Learning regular expressions once means you can apply them across all of these tools.

A command-line utility that searches text files or standard input for lines matching a specified pattern and prints matching lines to standard output. The Stream EDitor — a non-interactive line-oriented text editor that reads input line by line and applies commands including substitution, deletion, and insertion based on regular expressions. A programming language and command-line tool designed for text processing. Named after its creators Aho, Weinberger, and Kernighan. Awk processes files record by record (usually line by line) and applies patterns and actions.

Regular Expressions vs. Shell Globbing

Students often confuse regular expressions with shell globbing patterns. While both use special characters to describe patterns, they are completely different systems with different syntax and are processed by different programs.

Shell filename expansion (globbing) uses wildcards like *, ?, and [] to match filenames before a command runs. The shell itself expands the glob; the command never sees the wildcard characters.

⚠Common Mistake

In shell globbing, * matches anything. In a regular expression, .* matches anything (dot = any char, star = zero or more). A bare * in a regex means "zero or more of the preceding character," not "anything." This is one of the most common sources of confusion for new regex users.

Practical Value for System Administrators

Regular expressions unlock capabilities that no amount of manual searching can match. Consider these real system administration scenarios:

  • Log analysis: Find all SSH authentication failures for a specific IP range in a 500 MB log file in under a second
  • Configuration management: Change a service address across 200 configuration files simultaneously with one sed command
  • User provisioning: Extract username and UID columns from /etc/passwd to generate reports
  • Security auditing: Search scripts for hardcoded passwords or IP addresses matching specific patterns
  • Monitoring: Filter systemd journal output to show only critical events matching a pattern

Red Hat recommends that system administrators develop fluency with regular expressions as part of the RH134 curriculum, since automation scripts that cannot process text intelligently require constant manual intervention.

Basic Regular Expression Patterns

Basic regular expression patterns are the building blocks you combine to describe any text pattern. Every regex is made up of literal characters and metacharacters. Understanding each metacharacter's purpose is the foundation of regex fluency.

Characters in a regex that match themselves exactly. The letter 'a' matches the letter 'a'; the digit '5' matches the digit '5'. Special characters in a regex that have meaning beyond their literal character value. Examples: . * ^ $ [ ] \ ( ) + ? { } |

POSIX Character Classes

POSIX defines named character classes that work correctly with the system locale. These are preferred over simple ranges like [A-Z] because locale settings can affect how ranges are interpreted.

POSIX (Portable Operating System Interface) character classes are locale-aware named sets enclosed in [: :] notation used inside bracket expressions. They ensure consistent behavior across different character encodings and languages.
ClassEquivalent toMatches
[:alpha:][A-Za-z]Any alphabetic letter
[:digit:][0-9]Any decimal digit
[:alnum:][A-Za-z0-9]Any alphanumeric character
[:lower:][a-z]Any lowercase letter
[:upper:][A-Z]Any uppercase letter
[:space:][ \t\n\r]Any whitespace character
[:blank:][ \t]Space or tab only
[:punct:]Punctuation charsAny punctuation character
[:print:]Printable charsAny printable character including space
💡Syntax Note

POSIX classes must be used inside a bracket expression. To match any digit, write [[:digit:]] — double brackets are required. The outer [] creates the character class; the inner [: :] names the POSIX class within it.

POSIX Character Class Examples
# Match lines containing only alphabetic characters [student@workstation ~]$ grep '^[[:alpha:]]*$' /etc/hostname workstation # Find lines in /etc/passwd where the UID contains only digits [student@workstation ~]$ grep '^[[:alpha:]]*:[^:]*:[[:digit:]]' /etc/passwd | head -3 root:x:0:0:root:/root:/bin/bash bin:x:1:1:bin:/bin:/sbin/nologin daemon:x:2:2:daemon:/sbin:/sbin/nologin # Find IP addresses (simplified pattern) [student@workstation ~]$ grep '[[:digit:]]\{1,3\}\.[[:digit:]]\{1,3\}\.[[:digit:]]\{1,3\}\.[[:digit:]]\{1,3\}' /var/log/secure

Extended Regular Expressions

Extended Regular Expressions (ERE) add several powerful features to basic regex syntax. ERE is used by grep -E, egrep, and awk. The key advantage is cleaner syntax — you don’t need backslashes before grouping, quantifier, and alternation operators.

Extended Regular Expressions (ERE) are a POSIX-standardized superset of basic regex. ERE adds the +, ?, |, and () operators without requiring backslash escaping, making complex patterns more readable.

ERE-Only Quantifiers

Grouping and Alternation

Grouping with parentheses lets you apply quantifiers to multi-character sequences and create backreferences for matching repeated patterns. Alternation with the pipe character | acts like a logical OR between two sub-patterns.

Parentheses in regex create a capture group that treats the enclosed characters as a single unit. Quantifiers after the closing parenthesis apply to the entire group. Groups can also capture matched text for backreferences. A backreference in regex refers back to what a capture group matched earlier in the same pattern. Written as \1, \2, etc. Useful for finding repeated words or duplicate fields. The pipe character | in ERE means "or" — the pattern matches if either the left side or the right side matches. Alternation has the lowest precedence, so use grouping to limit its scope.
ERE Grouping and Alternation
# Match lines with "error" or "warning" (case-insensitive) [student@workstation ~]$ grep -Ei '(error|warning)' /var/log/messages | tail -5 Mar 5 09:12:03 workstation kernel: ERROR: Unable to mount device Mar 5 09:15:44 workstation sshd: WARNING: too many authentication failures # Match IPv4 addresses using groups and quantifiers [student@workstation ~]$ grep -E '([0-9]{1,3}\.){3}[0-9]{1,3}' /etc/hosts 127.0.0.1 localhost localhost.localdomain 192.168.1.100 workstation.lab.example.com workstation # Find duplicate words in a file (backreference) [student@workstation ~]$ grep -E '\b([a-z]+) \1\b' document.txt The the server was was restarted. # Match lines with "failed" or "refused" in journal [student@workstation ~]$ journalctl -u sshd | grep -E 'failed|refused|invalid' Mar 05 10:22:11 workstation sshd[1423]: Failed password for invalid user admin

Anchors and Word Boundaries

Beyond ^ and $ for line anchors, the word boundary metacharacter \b is extremely useful for matching whole words without accidentally matching substrings within longer words.

A word boundary \b matches the position between a word character (alphanumeric or underscore) and a non-word character. It does not consume any characters; it matches a position. Use \b to find whole words and avoid partial matches.
Anchors and Word Boundaries
# ^ matches beginning of line — only lines starting with "root" [student@workstation ~]$ grep '^root' /etc/passwd root:x:0:0:root:/root:/bin/bash # $ matches end of line — only accounts using /bin/bash shell [student@workstation ~]$ grep '/bin/bash$' /etc/passwd root:x:0:0:root:/root:/bin/bash student:x:1000:1000:Student User:/home/student:/bin/bash # \b prevents "error" from matching inside "erroring" or "errored" [student@workstation ~]$ grep -E '\berror\b' /var/log/messages # Combine anchors: match lines that contain ONLY whitespace or are empty [student@workstation ~]$ grep -E '^[[:space:]]*$' /etc/ssh/sshd_config | wc -l 18 # Match blank lines to count them [student@workstation ~]$ grep -c '^$' /etc/ssh/sshd_config 18

Using grep and egrep

The grep command is arguably the most important text-searching tool on a Linux system. Its name comes from the ed editor command g/re/p (Global Regular Expression Print). Mastering grep's options dramatically speeds up log analysis, configuration auditing, and troubleshooting.

grep (Global Regular Expression Print) searches input files or stdin for lines matching a pattern and outputs matching lines. Originally derived from the g/re/p command in the ed text editor.

The sed Stream Editor

sed (Stream EDitor) reads input line by line, applies editing commands to each line based on address or pattern matching, and writes the result to standard output. It is the preferred tool for automated, non-interactive text transformation — especially modifying configuration files in scripts.

sed is a non-interactive, line-oriented stream editor. It processes each line of input through a script of editing commands and writes the transformed text to standard output. It never modifies the input file unless the -i (in-place) flag is used.

sed Addressing

Before a sed command, you can specify an address that restricts which lines the command applies to. Without an address, the command applies to every line.

In sed, an address specifies which lines a command should act on. Addresses can be line numbers (5), ranges (3,7), last line ($), regex patterns (/pattern/), or step counts (first~step).

The Substitution Command: s///

The substitution command s/pattern/replacement/flags is the most-used sed command. It replaces text matching the pattern with the replacement string. The delimiter is usually / but can be any character — useful when the pattern contains forward slashes.

The sed s command (substitute) has the form s/regexp/replacement/flags. It finds the first occurrence of regexp on the line and replaces it with replacement. The g flag replaces all occurrences; the I flag makes matching case-insensitive.
sed Substitution Examples
# Basic substitution: replace first "old" with "new" on each line [student@workstation ~]$ echo "old config old" | sed 's/old/new/' new config old # Global flag (g): replace ALL occurrences on each line [student@workstation ~]$ echo "old config old" | sed 's/old/new/g' new config new # Change delimiter to | when pattern contains slashes [student@workstation ~]$ sed 's|/usr/local/bin|/usr/bin|g' /etc/environment # In-place edit with backup (-i.bak creates sshd_config.bak) [root@serverb ~]# sed -i.bak 's/^#PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config # Use & to include the matched text in the replacement [student@workstation ~]$ echo "192.168.1.100" | sed 's/[0-9]\+/[&]/g' [192].[168].[1].[100] # Backreferences in replacement using \1, \2 [student@workstation ~]$ echo "John Smith" | sed 's/\([A-Z][a-z]*\) \([A-Z][a-z]*\)/\2, \1/' Smith, John # Delete comment lines and blank lines from a config file (preview) [student@workstation ~]$ sed '/^[[:space:]]*#/d; /^[[:space:]]*$/d' /etc/ssh/sshd_config Port 22 AddressFamily any ListenAddress 0.0.0.0 ...

Text Processing with awk

awk is a complete programming language optimized for processing structured text. Where grep selects lines and sed transforms text, awk understands structure — it can extract specific columns, perform calculations, reformat data, and generate reports from text files.

awk is a programming language for text processing. It automatically splits each input line into fields (separated by a delimiter), making it ideal for processing structured text like CSV files, /etc/passwd, log files with fixed columns, or any tabular data.

awk's Data Model

awk processes input as a series of records (by default, lines). Each record is split into fields based on the field separator. awk automatically sets variables for each field: $1 is the first field, $2 the second, and so on. $0 is always the entire record.

In awk, a record is one unit of input — by default a single line terminated by a newline. The record separator (RS variable) can be changed to split on other characters. In awk, a field is one unit within a record — by default a whitespace-delimited word. Fields are numbered starting at $1. $NF always refers to the last field regardless of how many fields there are. The awk field separator (FS) defines how records are split into fields. Default is any whitespace. Set with -F option or FS= in BEGIN block. Common values: FS=":" for /etc/passwd, FS="," for CSV files.

awk Program Structure

An awk program consists of one or more rules of the form: pattern { action }

  • If pattern is omitted, the action runs on every record
  • If action is omitted, matching records are printed
  • The special patterns BEGIN and END run before or after all input
awk Fundamentals
# Print the first and third fields (username and UID) from /etc/passwd [student@workstation ~]$ awk -F: '{print $1, $3}' /etc/passwd | head -5 root 0 bin 1 daemon 2 adm 3 lp 4 # Print only users with UID >= 1000 (regular user accounts) [student@workstation ~]$ awk -F: '$3 >= 1000 {print $1, $3, $6}' /etc/passwd student 1000 /home/student operator 1001 /home/operator # Count lines matching a pattern (like grep -c but with awk math) [student@workstation ~]$ awk '/error/{count++} END{print "Errors found:", count}' /var/log/messages Errors found: 47 # Print formatted report with BEGIN header and END footer [student@workstation ~]$ awk -F: 'BEGIN{print "USER\t\tUID\tHOME"} $3>=1000{printf "%-15s %d\t%s\n",$1,$3,$6} END{print "---done---"}' /etc/passwd USER UID HOME student 1000 /home/student ---done---

Practical awk: System Administration Tasks

awk for System Administration
# Disk usage report: show filesystems over 80% used [student@workstation ~]$ df -h | awk 'NR>1 && $5+0 > 80 {print $1, $5, "ALERT"}' /dev/sda1 84% ALERT # Calculate average load from /proc/loadavg [student@workstation ~]$ uptime | awk -F'[, ]+' '{print "1min:", $(NF-2), "5min:", $(NF-1), "15min:", $NF}' 1min: 0.45 5min: 0.38 15min: 0.31 # Sum the size of all .log files in /var/log [student@workstation ~]$ ls -l /var/log/*.log | awk '{sum += $5} END {print "Total bytes:", sum}' Total bytes: 184729600 # Parse Apache/nginx access log for status code counts [student@workstation ~]$ awk '{codes[$9]++} END {for(c in codes) print c, codes[c]}' /var/log/httpd/access_log | sort -k2 -rn 200 18423 304 2145 404 387 500 12 # Extract lines between timestamps (e.g., 10:00 to 11:00) [student@workstation ~]$ awk '/10:00:00/,/11:00:00/' /var/log/messages # Show processes by user with total memory % [student@workstation ~]$ ps aux | awk 'NR>1 {mem[$1]+=$4} END {for(u in mem) printf "%s: %.1f%%\n", u, mem[u]}' | sort -t: -k2 -rn | head student: 8.3% root: 5.1% dbus: 0.4%
💡Choosing the Right Tool

Use grep when you need to select lines based on a pattern.
Use sed when you need to modify or transform text on matching lines.
Use awk when you need to work with specific columns, do arithmetic, or generate reports from structured data.

Guided Exercise: Log Analysis with Regex

In this guided exercise, you analyze system logs to identify and summarize security events. This mirrors a real-world incident response scenario where you must quickly extract meaningful information from large log files using regex-powered tools.

ℹScenario

The security team has flagged unusual activity on serverb.lab.example.com. You must analyze /var/log/secure and /var/log/messages to identify failed login attempts, extract attacker IP addresses, and produce a summary report. Work from workstation.lab.example.com via SSH.

Exercise Steps

Objective: Connect to serverb and identify relevant log files.

Step 1: Initial Connection and Log Examination
[student@workstation ~]$ ssh student@serverb.lab.example.com Activate the web console with: systemctl enable --now cockpit.socket Last login: Wed Mar 5 08:00:00 2025 from workstation.lab.example.com [student@serverb ~]$ ls -lh /var/log/secure /var/log/messages -rw-------. 1 root root 2.3M Mar 5 11:00 /var/log/secure -rw-------. 1 root root 18M Mar 5 11:00 /var/log/messages [student@serverb ~]$ sudo tail -5 /var/log/secure Mar 5 10:58:01 serverb sshd[5423]: Failed password for root from 203.0.113.42 port 44821 ssh2 Mar 5 10:59:14 serverb sshd[5501]: Accepted publickey for student from 192.168.1.5 port 52341 ssh2

Objective: Use grep to count and categorize failed authentication events.

Step 2: grep for Failed Logins
# Count all failed password attempts [student@serverb ~]$ sudo grep -c 'Failed password' /var/log/secure 847 # Count attempts by username [student@serverb ~]$ sudo grep 'Failed password' /var/log/secure | grep -oE 'for [a-z_]+ ' | sort | uniq -c | sort -rn | head -5 412 for root 203 for admin 89 for oracle 67 for postgres 44 for mysql # Show attempts for invalid usernames [student@serverb ~]$ sudo grep -c 'invalid user' /var/log/secure 393

Objective: Extract IP addresses from failed login entries and identify the top attackers.

Step 3: IP Extraction with grep and awk
# Extract all attacker IPs using grep -oE [student@serverb ~]$ sudo grep 'Failed password' /var/log/secure | grep -oE 'from [0-9]{1,3}(\.[0-9]{1,3}){3}' | awk '{print $2}' | sort | uniq -c | sort -rn | head -10 215 203.0.113.42 189 198.51.100.7 99 192.0.2.134 88 203.0.113.8 76 198.51.100.99 # Get unique attacker count [student@serverb ~]$ sudo grep 'Failed password' /var/log/secure | grep -oE '[0-9]{1,3}(\.[0-9]{1,3}){3}' | sort -u | wc -l 23

Objective: Use awk to produce a formatted security report from the log data.

Step 4: awk Security Report
# Create a formatted security summary report [student@serverb ~]$ sudo grep 'Failed password' /var/log/secure | \ awk '{ # Extract IP: field after "from" for(i=1;i<=NF;i++) if($i=="from") ip=$(i+1) ips[ip]++ total++ } END { printf "%-20s %s\n", "SOURCE IP", "ATTEMPTS" printf "%-20s %s\n", "----------", "--------" for(ip in ips) printf "%-20s %d\n", ip, ips[ip] print "" print "Total failed attempts:", total }' | sort -k2 -rn | head -15 SOURCE IP ATTEMPTS ---------- -------- 203.0.113.42 215 198.51.100.7 189 192.0.2.134 99 Total failed attempts: 847

Objective: Transform the IP list into firewall-cmd commands using sed and awk.

Step 5: Generating Firewall Commands
# Extract unique IPs with 50+ attempts and generate firewall-cmd commands [student@serverb ~]$ sudo grep 'Failed password' /var/log/secure | \ grep -oE '[0-9]{1,3}(\.[0-9]{1,3}){3}' | \ sort | uniq -c | sort -rn | \ awk '$1 >= 50 {print $2}' | \ sed 's/.*/firewall-cmd --permanent --add-rich-rule="rule family=ipv4 source address=& reject"/' firewall-cmd --permanent --add-rich-rule="rule family=ipv4 source address=203.0.113.42 reject" firewall-cmd --permanent --add-rich-rule="rule family=ipv4 source address=198.51.100.7 reject" firewall-cmd --permanent --add-rich-rule="rule family=ipv4 source address=192.0.2.134 reject" firewall-cmd --permanent --add-rich-rule="rule family=ipv4 source address=203.0.113.8 reject"
⚠Review Before Running

Always review the generated commands before executing them. Pipe to a file first, review the contents, then execute. Never automatically block IPs without human review — legitimate traffic could be blocked.

Chapter Summary

In this chapter, you learned to harness the power of regular expressions for practical Linux system administration. Regular expressions are a force multiplier: a single well-crafted pattern can replace hours of manual searching and editing.

Key Concepts

  • Regular expressions are pattern-matching languages processed by tools (not the shell), distinct from shell globbing
  • POSIX defines BRE (Basic) and ERE (Extended) regex; ERE has cleaner syntax for complex patterns
  • Core metacharacters: . ^ $ * [] [^] \
  • ERE adds: + ? {n,m} () | without backslash escaping
  • POSIX character classes ([[:alpha:]], [[:digit:]], etc.) are locale-safe alternatives to ranges
