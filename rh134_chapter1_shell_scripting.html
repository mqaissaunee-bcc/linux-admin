Introduction to Shell Scripting

A shell scriptA text file containing a sequence of shell commands. When executed, the shell reads and runs each command in order, enabling automation of complex tasks. is simply a text file containing a sequence of commands that the shell can execute. Instead of typing each command individually, you write them into a file once and run the file whenever you need that sequence. This is the foundation of Linux automation.

Why Use Shell Scripts?

The power of shell scripting lies in its ability to combine three capabilities:

  • Sequencing: Run many commands in a defined order
  • Decision making: React differently to different conditions
  • Repetition: Perform the same operation on many targets

Anatomy of a Bash Script

Every well-formed Bash script follows a consistent structure. Let's examine each component:

Basic Script Structure
#!/bin/bash # Script name: backup_logs.sh # Description: Archives log files older than 7 days # Author: student # Date: 2026-03-05 # Variable declarations LOG_DIR="/var/log" ARCHIVE_DIR="/backup/logs" DAYS=7 # Main script body echo "Starting log backup..." mkdir -p $ARCHIVE_DIR find $LOG_DIR -name "*.log" -mtime +$DAYS -exec mv {} $ARCHIVE_DIR \; echo "Backup complete."

The Shebang Line (#!/bin/bash)

The very first line of a script must be the shebang (also called hashbang). It tells the kernel which interpreter to use to run the script. The #! characters are followed immediately by the absolute path to the interpreter.

#!/bin/bash # Use the Bash shell (most common) #!/usr/bin/env bash # More portable: finds bash in PATH #!/bin/sh # POSIX sh (more portable, fewer features) #!/usr/bin/python3 # Even non-shell interpreters work!
💡Best Practice

For Red Hat system administration scripts, use #!/bin/bash. This guarantees you have all Bash features available. Using #!/usr/bin/env bash is slightly more portable across systems.

Comments in Shell Scripts

Lines beginning with # (except the shebang) are commentsText in a script ignored by the interpreter. Comments explain what the code does and are essential for maintainability.. Comments are ignored by the shell but are essential for humans. Always include a header comment block and inline comments for complex logic.

#!/bin/bash # ============================================================ # Script: monitor_disk.sh # Purpose: Alert when disk usage exceeds threshold # Usage: ./monitor_disk.sh [threshold_percent] # Example: ./monitor_disk.sh 85 # ============================================================ THRESHOLD=${1:-90} # Use $1 or default to 90%

Making Scripts Executable

A script file must have execute permissionA file permission bit that allows a file to be run as a program. Set with chmod +x filename or chmod 755 filename. to run directly. Use chmod to add this permission.

# Create the script [student@workstation ~]$ vim backup_logs.sh # Make it executable (owner can read, write, execute; others read+execute) [student@workstation ~]$ chmod 755 backup_logs.sh # Or use symbolic notation [student@workstation ~]$ chmod u+x backup_logs.sh # Verify the permission bit was added (note the 'x' flags) [student@workstation ~]$ ls -l backup_logs.sh -rwxr-xr-x. 1 student student 342 Mar 5 10:15 backup_logs.sh

Running Shell Scripts

There are multiple ways to execute a Bash script, each with different behavior:

# Method 1: Direct execution (requires execute bit + shebang) [student@workstation ~]$ ./backup_logs.sh # Method 2: Specify the interpreter explicitly [student@workstation ~]$ bash backup_logs.sh # Method 3: Source (runs in current shell, inherits variables) [student@workstation ~]$ source backup_logs.sh [student@workstation ~]$ . backup_logs.sh # Shorthand for source # Debug mode: prints each command before executing it [student@workstation ~]$ bash -x backup_logs.sh + LOG_DIR=/var/log + ARCHIVE_DIR=/backup/logs + DAYS=7 + echo 'Starting log backup...'
🔴Important: Path Requirements

You must use ./script.sh (explicit path) to run a script in the current directory. Simply typing script.sh fails unless the current directory is in your PATH, which is a security risk on production systems.

Exit Codes and Error Handling

Every command and script exits with a numeric exit codeAn integer returned by a command or script upon completion. Exit code 0 means success; any non-zero value indicates failure. Accessible via the special variable $?. (also called a return status). This is fundamental to reliable scripting.

Working with Exit Codes
# $? holds the exit code of the last command [student@workstation ~]$ ls /etc/passwd /etc/passwd [student@workstation ~]$ echo $? 0 # 0 = success [student@workstation ~]$ ls /no/such/file ls: cannot access '/no/such/file': No such file or directory [student@workstation ~]$ echo $? 2 # non-zero = failure # In scripts, set explicit exit codes if ! mkdir -p /backup/logs; then echo "ERROR: Cannot create backup directory" >&2 exit 1 fi
Exit CodeMeaningCommon Cause
0SuccessCommand completed without errors
1General errorCatchall for miscellaneous errors
2Misuse of shell builtinsMissing or invalid argument

127Command not foundCommand or script doesn't exist in PATH

Variables, Parameters, and Quoting

VariablesNamed storage locations in memory that hold values used by your script. Variables make scripts flexible and reusable by separating data from logic. are named containers that store values your script can use and modify. Mastering variables — including their scope, quoting, and special types — is essential to writing effective scripts.

Declaring and Using Variables

In Bash, variable names are case-sensitive and conventionally written in uppercase for constants and lowercase for local variables. There is no space around the = sign.

Variable Basics
#!/bin/bash # Variable assignment: NO spaces around = NAME="Red Hat Enterprise Linux" VERSION=10 admin_user="student" # Reference with $ prefix echo "System: $NAME version $VERSION" System: Red Hat Enterprise Linux version 10 # Braces isolate the variable name echo "Logfile: ${NAME}_system.log" Logfile: Red Hat Enterprise Linux_system.log # Command substitution: store command output in a variable HOSTNAME=$(hostname -s) TODAY=$(date +%Y-%m-%d) DISK_USAGE=$(df -h / | tail -1 | awk '{print $5}') echo "Disk usage on $HOSTNAME today ($TODAY): $DISK_USAGE"

Quoting Rules

Understanding quoting is critical — it determines how the shell interprets special characters.

Quote TypeSyntaxVariable ExpansionUse Case
Double quotes"..."YesStrings containing spaces, allow variable expansion
Single quotes'...'NoLiteral strings, no interpretation whatsoever
Backticks`...`YesCommand substitution (legacy; use $() instead)
No quotesYesSimple values without spaces (avoid when in doubt)
Quoting Behavior
GREETING="Hello" NAME="World" # Double quotes: variables expand echo "$GREETING, $NAME!" Hello, World! # Single quotes: NOTHING expands - literal dollar sign echo '$GREETING, $NAME!' $GREETING, $NAME! # Always quote variables that may contain spaces FILENAME="my report 2026.txt" ls -l "$FILENAME" # Correct: treated as one argument ls -l $FILENAME # Wrong: splits into 3 arguments!

Positional Parameters and Special Variables

Positional parametersVariables automatically set by the shell when a script is called with arguments. $1 holds the first argument, $2 the second, and so on up to $9 (use ${10} for more). allow scripts to accept input from the command line, making them flexible and reusable.

Special Variables Reference
#!/bin/bash # Script called as: ./myscript.sh alpha beta gamma echo "Script name: $0" # ./myscript.sh echo "First argument: $1" # alpha echo "Second argument: $2" # beta echo "Third argument: $3" # gamma echo "All arguments: $@" # alpha beta gamma echo "Argument count: $#" # 3 echo "Current PID: $$" # e.g., 12345 echo "Last exit code: $?" # 0 (if last command succeeded)

Variable Expansion and Default Values

Bash provides powerful parameter expansionAdvanced variable manipulation syntax using ${} that can provide defaults, test for empty values, extract substrings, and perform substitutions. syntax for common operations on variables:

Parameter Expansion Patterns
# Default value: use "default" if VAR is unset or empty echo "${VAR:-default}" # Assign default: set VAR to "default" if it's unset echo "${VAR:=default}" # Error if unset: display error message and exit echo "${VAR:?'VAR must be set'}" # String length NAME="RHEL" echo "Length: ${#NAME}" # 4 # Uppercase/lowercase conversion input="hello world" echo "${input^^}" # HELLO WORLD echo "${input^}" # Hello world # Substring extraction: ${var:offset:length} STRING="Red Hat Enterprise Linux" echo "${STRING:0:7}" # Red Hat # Pattern removal: strip prefix/suffix FILENAME="report_2026.tar.gz" echo "${FILENAME%.tar.gz}" # report_2026 (remove suffix) echo "${FILENAME##*_}" # 2026.tar.gz (remove longest prefix)

Arithmetic in Scripts

Bash supports integer arithmetic using the $(( )) syntax:

Integer Arithmetic
# Basic arithmetic with $(( )) a=10; b=3 echo "Sum: $((a + b))" # 13 echo "Product: $((a * b))" # 30 echo "Quotient:$((a / b))" # 3 (integer division) echo "Remainder:$((a % b))" # 1 # Increment and decrement count=0 count=$((count + 1)) # Increment (( count++ )) # C-style increment # The let command is another option let "result = a * b + 2" echo "Result: $result" # 32

Conditional Logic: if, elif, else, and case

Conditional statements allow your scripts to make decisions. They test a condition and execute different code depending on whether the condition is true or false. In Bash, conditions are evaluated using the test command or its shorthand [ ] and the more powerful [[ ]].

The if/elif/else Statement

if / elif / else Syntax
if condition1 ; then # commands run if condition1 is TRUE elif condition2 ; then # commands run if condition2 is TRUE else # commands run if ALL conditions are FALSE fi # "if" spelled backwards closes the block

Test Conditions

The test commandA shell builtin that evaluates conditions and returns exit code 0 (true) or 1 (false). Can be written as [ ] or [[ ]] brackets. evaluates expressions and returns exit code 0 for true or 1 for false:

OperatorTrue WhenExample
-e fileFile or directory exists[ -e /etc/passwd ]
-f fileIs a regular file[ -f /etc/hostname ]
-d pathIs a directory[ -d /var/log ]
-r fileFile is readable[ -r /etc/shadow ]
-w fileFile is writable[ -w /tmp/test ]
-x fileFile is executable[ -x /usr/bin/bash ]
-s fileFile size > 0[ -s /var/log/messages ]
-L fileIs a symbolic link[ -L /etc/localtime ]
f1 -nt f2f1 is newer than f2[ new.txt -nt old.txt ]
OperatorTrue WhenExample
-z stringString is empty (zero length)[ -z "$VAR" ]
-n stringString is NOT empty[ -n "$NAME" ]
s1 = s2Strings are equal[ "$OS" = "Linux" ]
s1 != s2Strings are NOT equal[ "$USER" != "root" ]
s1 < s2s1 sorts before s2[[ "$a" < "$b" ]]
s1 =~ regexString matches regex[[ "$IP" =~ ^[0-9] ]]
💡Double Brackets [[ ]]

Use [[ ]] (double brackets) instead of [ ] when using pattern matching (=~) or logical operators. Double brackets are a Bash extension that prevent word splitting and provide more natural behavior.

OperatorMeaningExample
-eqEqual to[ "$count" -eq 0 ]
-neNot equal to[ "$retval" -ne 0 ]
-ltLess than[ "$age" -lt 18 ]
-leLess than or equal[ "$score" -le 100 ]
-gtGreater than[ "$disk" -gt 90 ]
-geGreater than or equal[ "$files" -ge 1 ]
ℹ️Note: Use Words, Not Symbols

For numeric comparisons inside [ ], use alphabetic operators like -eq, -lt rather than == or <. This avoids interpretation as string operations or redirections.

Combining Conditions

OperatorMeaningExample
! exprNOT — negates the result[ ! -f /tmp/lock ]
-a / &&AND — both must be true[[ -f "$f" && -r "$f" ]]
-o / ||OR — either must be true[[ -z "$1" || "$1" = "-h" ]]
# Check if file exists AND is non-empty AND readable if [[ -f "$LOGFILE" && -s "$LOGFILE" && -r "$LOGFILE" ]]; then echo "Processing $LOGFILE..." fi

Practical if/elif/else Example

Disk Space Alert Script
#!/bin/bash # Check disk usage and alert based on threshold USAGE=$(df -h / | tail -1 | awk '{gsub(/%/,""); print $5}') if [[ $USAGE -ge 90 ]]; then echo "CRITICAL: Disk usage is ${USAGE}% - immediate action required!" logger "DISK CRITICAL: / at ${USAGE}%" exit 2 elif [[ $USAGE -ge 75 ]]; then echo "WARNING: Disk usage is ${USAGE}% - consider cleanup" exit 1 else echo "OK: Disk usage is ${USAGE}% - within normal range" exit 0 fi

The case Statement

The case statementA multi-branch conditional that matches a value against a series of patterns. More readable than long if/elif chains when testing one variable against many values. is ideal when you need to test a single variable against multiple patterns. It is cleaner than a long chain of elif statements.

case Statement Syntax and Example
#!/bin/bash # Service management wrapper SERVICE="httpd" ACTION=$1 case $ACTION in start) systemctl start $SERVICE echo "$SERVICE started" ;; stop) systemctl stop $SERVICE echo "$SERVICE stopped" ;; restart|reload) # OR pattern with | systemctl restart $SERVICE echo "$SERVICE restarted" ;; status) systemctl status $SERVICE ;; *) # Default/catch-all pattern echo "Usage: $0 {start|stop|restart|reload|status}" exit 1 ;; esac # "case" spelled backwards closes the block
💡Pattern Matching in case

Case patterns support glob-style wildcards: * (any string), ? (single character), [abc] (character class). Use | to match multiple patterns in one branch. Always include a *) default branch for unexpected input.

Loops: for, while, and until

Loops are one of the most powerful constructs in scripting. They allow you to repeat an action many times — across a list of servers, files, users, or any collection of data. Without loops, you would need to write the same commands manually for each item.

The for Loop

The for loopIterates over a fixed list of items (or a range), executing the loop body once for each item. The current item is assigned to the loop variable on each iteration. iterates over a list, executing commands for each item. This is the most commonly used loop in system administration scripts.

for Loop Patterns
#!/bin/bash # Pattern 1: Explicit list for HOST in web1 web2 web3 db1 db2; do ping -c 1 $HOST >/dev/null && echo "$HOST: UP" || echo "$HOST: DOWN" done # Pattern 2: Brace expansion for ranges for i in {1..5}; do echo "Processing item $i" done # Pattern 3: C-style numeric loop for (( i=1; i<=10; i++ )); do echo "Count: $i" done # Pattern 4: Iterate over files using glob for LOGFILE in /var/log/*.log; do echo "Compressing: $LOGFILE" gzip "$LOGFILE" done # Pattern 5: Iterate over command output for USER in $(awk -F: '$3 >= 1000 {print $1}' /etc/passwd); do echo "Regular user: $USER" done

The while Loop

The while loopRepeatedly executes a block of commands as long as a condition remains true. Used when you don't know in advance how many iterations are needed. continues executing as long as its condition is true. It's ideal for polling, counting, and reading input.

while Loop Examples
#!/bin/bash # Count down from 5 count=5 while [[ $count -gt 0 ]]; do echo "T-minus $count seconds..." (( count-- )) sleep 1 done echo "Liftoff!" # Read a file line by line (most common pattern) while IFS= read -r line; do echo "Processing: $line" done < /etc/passwd # Wait for a service to start (poll loop) TIMEOUT=30 elapsed=0 while ! systemctl is-active --quiet httpd; do if [[ $elapsed -ge $TIMEOUT ]]; then echo "ERROR: httpd failed to start within ${TIMEOUT}s" exit 1 fi sleep 2; (( elapsed += 2 )) done echo "httpd is running"
🔴Infinite Loop Warning

If the condition of a while loop never becomes false, the loop runs forever. Always ensure your loop has a clear exit condition, and consider adding a timeout counter as a safeguard. Press Ctrl+C to interrupt a runaway script.

The until Loop

The until loopThe logical opposite of while: it executes commands until a condition becomes true (i.e., while the condition is false). Useful for "keep trying until it succeeds" patterns. is the logical inverse of while — it loops until a condition becomes true:

until Loop
# Wait until a file exists before processing it DATAFILE="/tmp/import_data.csv" until [[ -f $DATAFILE ]]; do echo "Waiting for $DATAFILE to appear..." sleep 5 done echo "File found! Processing $DATAFILE" ./process_data.sh "$DATAFILE"

Loop Control: break and continue

Two keywords give you fine-grained control within loops:

  • break — exits the loop immediately
  • continue — skips the rest of the current iteration and moves to the next
break and continue
#!/bin/bash # Search servers until first failure found, skip maintenance hosts MAINTENANCE_HOST="web3" for HOST in web1 web2 web3 web4 web5; do # Skip maintenance host if [[ "$HOST" = "$MAINTENANCE_HOST" ]]; then echo "Skipping $HOST (maintenance)" continue fi # Check if host responds if ! ping -c 1 $HOST >/dev/null 2>&1; then echo "CRITICAL: $HOST is unreachable! Stopping scan." break # Exit loop immediately fi echo "$HOST: OK" done

Shell Functions

As scripts grow in complexity, functions become essential. A shell functionA named block of commands within a script that can be called by name, optionally accepting arguments. Functions improve code organization, reusability, and readability. is a named block of code that you can call by name from anywhere in a script. Functions eliminate repetition, improve readability, and make testing easier.

Defining and Calling Functions

Function Syntax (Both Forms Are Valid)
# Style 1: function keyword (Bash-specific) function greet { echo "Hello, $1!" } # Style 2: POSIX-compatible (preferred for portability) greet() { echo "Hello, $1!" } # Call the function greet "Alice" Hello, Alice! greet "World" Hello, World!

Function Arguments and Return Values

Functions receive arguments the same way scripts do — via positional parameters. They can return a status code with return, or output data via echo and command substitution.

Functions with Arguments and Return Values
#!/bin/bash # Function that validates an IP address format validate_ip() { local ip="$1" local pattern='^([0-9]{1,3}\.){3}[0-9]{1,3}$' if [[ "$ip" =~ $pattern ]]; then return 0 # 0 = success = valid else return 1 # 1 = failure = invalid fi } # Function that returns a computed value via echo get_disk_usage() { local mount_point="${1:-/}" df -h "$mount_point" | tail -1 | awk '{print $5}' } # Test the functions if validate_ip "192.168.1.100"; then echo "Valid IP address" fi USAGE=$(get_disk_usage "/") echo "Root disk usage: $USAGE" Root disk usage: 45%

Local Variables and Variable Scope

The local keywordDeclares a variable that exists only within the function where it is defined. Local variables prevent functions from accidentally modifying global script variables. creates variables that only exist inside the function. Without local, variables are globalVariables visible throughout the entire script and any functions it calls. Changes to global variables inside functions affect the whole script. by default and could interfere with the rest of the script.

Variable Scope: Local vs Global
#!/bin/bash NAME="Global Alice" # Global variable test_scope() { local NAME="Local Bob" # Local — doesn't affect global UNSCOPED="I leaked!" # No local — affects global scope! echo "Inside function: $NAME" } echo "Before: $NAME" # Global Alice test_scope echo "After: $NAME" # Global Alice (unchanged) echo "Leaked: $UNSCOPED" # I leaked! (now visible globally)

A Complete Script Using Functions

Production-Style Script with Functions
#!/bin/bash # Script: user_report.sh - Generate a report of system users # Usage: ./user_report.sh [min_uid] # === Configuration === MIN_UID=${1:-1000} # Default: regular users (UID >= 1000) REPORT_FILE="/tmp/user_report_$(date +%Y%m%d).txt" # === Functions === log() { echo "[$(date '+%H:%M:%S')] $*" } print_header() { local title="$1" echo "==============================" echo " $title" echo "==============================" } get_users() { local min_uid="$1" awk -F: -v min="$min_uid" '$3 >= min {print $1, $3, $6}' /etc/passwd } # === Main === log "Starting user report (min UID: $MIN_UID)" { print_header "User Report - $(date)" get_users "$MIN_UID" } > "$REPORT_FILE" log "Report written to $REPORT_FILE"

🔬 Lab: Shell Scripting and the Command Line

You are a system administrator at a company running RHEL 10. Your manager has asked you to create automation scripts that will run weekly to:

  2. Verify that critical services are running and report their status

Objective: Create a working directory for your scripts and write a basic script skeleton with proper shebang, comments, and executable permissions.

# Step 1: Create script directory [student@workstation ~]$ mkdir -p ~/scripts && cd ~/scripts # Step 2: Create the main script [student@workstation scripts]$ cat > syscheck.sh <<'EOF' #!/bin/bash # syscheck.sh - Weekly system health check # Author: student # Date: 2026-03-05 LOGDIR="/var/log/syscheck" LOGFILE="$LOGDIR/report_$(date +%Y%m%d).log" echo "System check started: $(date)" EOF # Step 3: Make it executable [student@workstation scripts]$ chmod 755 syscheck.sh [student@workstation scripts]$ ls -l syscheck.sh -rwxr-xr-x. 1 student student 178 Mar 5 10:00 syscheck.sh # Step 4: Run it to verify [student@workstation scripts]$ ./syscheck.sh System check started: Thu Mar 5 10:00:12 EST 2026

Chapter Summary

📚 Key Concepts Covered

  • Script structure: The shebang line, header comments, variable declarations, and the main body form the anatomy of every well-written script.
  • Exit codes: Every command returns 0 for success or non-zero for failure; scripts should propagate these meaningfully.
  • Variables: Must use proper quoting to handle spaces; parameter expansion provides powerful manipulation like ${VAR:-default} and ${VAR^^}.
  • Conditionals: if/elif/else and case use [ ] or [[ ]] test conditions; file, string, and numeric tests all have different operators.
  • Loops: for iterates over lists; while loops while a condition is true; until loops until a condition becomes true; break and continue control flow.
  • Functions: Declared with name() {} syntax; use local for scoped variables; return values via exit codes or stdout.
