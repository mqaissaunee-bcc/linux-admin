The RHEL 10 Logging Architecture

Log management is one of the most critical skills for any system administrator. Logs tell you what happened on a system, when it happened, which service or user triggered an event, and — crucially — why something went wrong. On RHEL 10, two complementary subsystems handle logging: systemd-journald and rsyslog.

systemd-journald is the logging daemon built into systemd. It collects structured binary log entries from the kernel, boot messages, and all systemd-managed services, storing them in /run/log/journal. rsyslog (Rocket-fast Syslog) is a traditional syslog daemon that receives messages from journald and other sources, then routes them to text log files under /var/log based on configurable rules.

Why Two Logging Subsystems?

RHEL 10 uses both systems because each has distinct strengths. systemd-journald captures structured, indexed metadata (unit name, PID, UID, message ID) that enables powerful filtering. rsyslog writes plain-text files compatible with legacy tools and UNIX conventions. The two systems are bridged: journald forwards messages to rsyslog via a socket, and rsyslog writes them to the traditional /var/log/ hierarchy.

💡Tip: Persistent Journal

By default, systemd-journald stores the journal in /run/log/journal, which is a tmpfs filesystem and does not survive reboots. To persist the journal across reboots, create the directory /var/log/journal and restart journald. You can also set Storage=persistent in /etc/systemd/journald.conf.

tmpfs is a temporary filesystem stored in RAM (and possibly swap). Its contents are lost when the system is powered off or rebooted, making it unsuitable for persistent log storage without extra configuration.

The Syslog Protocol: Facilities and Priorities

Before log messages can be routed to the right destination, they must be categorized. The syslog protocol classifies every message along two dimensions: its facility (what part of the system generated it) and its priority (how severe it is).

The syslog protocol (RFC 5424) is a standard for system logging that defines message format, classification by facility and severity, and transport over TCP/UDP. It dates back to the early days of UNIX and is still central to Linux logging.

Syslog Priorities (Severity Levels)

Priorities are ordered from most severe (0 = emerg) to least severe (7 = debug). In rsyslog rules, specifying a priority also matches all higher-severity levels (lower numbers) unless you use the = exact-match modifier.

Syslog Facilities

A facility identifies the subsystem or source that generated the message. Facilities allow rsyslog to route messages from different parts of the system to different log files.

A syslog facility is a numeric code (0–23) that identifies the source or subsystem of a log message, such as the kernel, mail system, or local applications. It is used as the basis for routing rules in rsyslog.

The Selector Syntax: facility.priority

In rsyslog rules, you specify facility.priority to select which messages to act on. The full syntax supports wildcards and modifiers:

Selector syntax examples
SelectorMeaning
kern.emergKernel messages at emergency level
mail.warningMail messages at warning level or higher severity
*.infoAll facilities at info level or higher
*.debugAll messages from all facilities
mail.=infoMail messages at exactly info level only
mail.noneExclude all mail messages
*.info;mail.none;authpriv.noneAll info+ except mail and authpriv (typical /var/log/messages rule)
ℹPriority Inheritance

Specifying a priority in a selector matches that level and all more-severe levels. So *.warning captures warning, err, crit, alert, and emerg. To match only one exact level, use the = prefix: mail.=info.

Configuring rsyslog

rsyslog is the default syslog implementation on RHEL 10. It provides both backward-compatible syslog functionality and powerful advanced features including property-based filtering, output buffering, and TLS-encrypted log forwarding.

rsyslog (Rocket-fast SYStem LOGger) is an enhanced syslog daemon that extends traditional syslog with high-performance features, property-based filtering, and pluggable modules. It replaces older syslogd and syslog-ng implementations.

Configuration File Structure

rsyslog is configured through /etc/rsyslog.conf and any files matching /etc/rsyslog.d/*.conf. The recommended practice is to place custom rules in separate files under /etc/rsyslog.d/ rather than editing the main config file directly. This simplifies management and avoids issues during package updates.

/etc/rsyslog.conf (structure overview)
#### MODULES #### module(load="imuxsock") # Local Unix socket (from journald) module(load="imklog") # Kernel log messages (/proc/kmsg) module(load="imjournal")# Forward from systemd journal #### GLOBAL DIRECTIVES #### $WorkDirectory /var/lib/rsyslog $ActionFileDefaultTemplate RSYSLOG_TraditionalFileFormat $IncludeConfig /etc/rsyslog.d/*.conf #### RULES #### # facility.priority /path/to/logfile *.info;mail.none;authpriv.none;cron.none /var/log/messages authpriv.* /var/log/secure mail.* -/var/log/maillog cron.* /var/log/cron *.emerg :omusrmsg:* uucp,news.crit /var/log/spooler local7.* /var/log/boot.log
ℹThe Minus (-) Prefix Explained

A - prefix before a log file path (as in -/var/log/maillog) tells rsyslog to use asynchronous writes. This improves performance but means some messages could be lost if the system crashes before the write buffer is flushed to disk.

Asynchronous (buffered) write mode allows rsyslog to accumulate log entries in memory and write them to disk in batches rather than after each individual message. This reduces disk I/O at the cost of potential data loss on a crash.

Writing Custom rsyslog Rules

Best practice is to create a new file under /etc/rsyslog.d/ for custom rules. File names should start with a number to control load order (lower numbers load first).

Log all auth and authpriv messages at notice level or higher to a dedicated security log file:

/etc/rsyslog.d/40-security.conf
# Capture all auth and authpriv messages notice and above auth,authpriv.notice /var/log/auth-custom.log
Apply the new configuration
[root@servera ~]# systemctl restart rsyslog [root@servera ~]# logger -p auth.notice "Test auth notice message" [root@servera ~]# tail /var/log/auth-custom.log Mar 14 10:15:30 servera root: Test auth notice message

Forward all messages at warning level or higher to a central syslog server over UDP:

/etc/rsyslog.d/50-remote.conf
# Forward warning+ messages to logserver via UDP (port 514) *.warning @logserver.lab.example.com # For TCP forwarding, use @@ instead of @ *.warning @@logserver.lab.example.com:514
💡UDP vs TCP Forwarding

A single @ uses UDP (faster, no delivery guarantee). Double @@ uses TCP (reliable, ordered delivery). For critical logs or high-volume environments, TCP is preferred.

Temporarily enable debug logging for a specific service (useful during troubleshooting):

/etc/rsyslog.d/99-debug.conf (temporary!)
# Log ALL messages including debug to a separate file # WARNING: Very verbose — remove when done troubleshooting! *.debug /var/log/debug-all.log
⚠Warning: Debug Logging Is Verbose

Debug-level logging generates enormous amounts of data and can fill disks quickly. Only enable it during active troubleshooting sessions and remove or disable the rule when done. Never leave *.debug rules active in production.

Use a property-based filter to match messages containing specific text:

A property-based filter in rsyslog matches messages based on message content fields (properties) rather than just facility/priority. It uses the syntax :property, compare-op, "value" and supports regex matching.
/etc/rsyslog.d/60-oom.conf
# Capture Out-of-Memory killer events to a dedicated log :msg, contains, "Out of memory" /var/log/oom-events.log # Use regex for more flexible matching :msg, regex, "oom_kill|OOM killer" /var/log/oom-events.log

Testing rsyslog Configuration

Use logger to send test messages to the syslog facility from the command line, and rsyslogd -N1 to syntax-check your configuration without restarting the service.

The logger command is a shell command interface to the syslog(3) system call. It allows scripts and administrators to send messages to the system log from the command line, useful for testing rsyslog rules.
Testing rsyslog rules
# Test configuration syntax before restarting [root@servera ~]# rsyslogd -N1 rsyslogd: version 8.2312.0, config validation run (level 1), master config /etc/rsyslog.conf rsyslogd: End of config validation run. Bye. # Send test messages at specific facility.priority [root@servera ~]# logger -p local0.info "Test info from local0 facility" [root@servera ~]# logger -p mail.err "Simulated mail delivery failure" [root@servera ~]# logger -p auth.crit "Simulated authentication failure" # Send a tagged message (shows in log as: tag: message) [root@servera ~]# logger -t myapp -p user.warning "Low disk space on /data" # Verify message arrived in the expected log file [root@servera ~]# tail -5 /var/log/messages Mar 14 10:22:01 servera root: Test info from local0 facility Mar 14 10:22:08 servera myapp: Low disk space on /data

systemd-journald: Structured Binary Logging

systemd-journald is a fundamental change from traditional text-file logging. Rather than appending plain text lines to files, journald stores messages as structured, indexed binary records. Each record includes metadata fields such as the systemd unit name, PID, UID, GID, hostname, and a message catalog ID that can link to documentation.

systemd-journald is the logging component of systemd. It collects log entries from kernel messages, early boot output, stdout/stderr of services, and applications using the native journal API, storing them as indexed binary records rather than plain text.

What journald Collects

journald is the single collection point for all system events, regardless of origin:

  • Kernel ring buffer: Kernel messages that would previously have gone to /var/log/dmesg
  • Early boot messages: Output captured before userspace services start
  • systemd unit output: stdout and stderr of every service managed by systemd
  • Native journal API: Applications that use sd_journal_send() to log structured data
  • Traditional syslog socket: Messages from programs using syslog(3) or openlog()

Journal Storage and Persistence

The /etc/systemd/journald.conf file controls how the journal stores data. The most important setting for production systems is Storage=:

Journal Storage= configuration values
ValueBehavior
volatileStore only in RAM (/run/log/journal). Lost on reboot. Good for stateless containers.
persistentStore on disk (/var/log/journal). Survives reboots. Journal files are created automatically.
autoDefault: Persistent if /var/log/journal exists, otherwise volatile.
noneDiscard all journal data. Forward to syslog socket only.
Enable persistent journal storage
# Method 1: Create the journal directory (triggers auto → persistent) [root@servera ~]# mkdir -p /var/log/journal [root@servera ~]# systemd-tmpfiles --create --prefix /var/log/journal [root@servera ~]# systemctl restart systemd-journald # Method 2: Explicitly configure in journald.conf [root@servera ~]# grep -n Storage /etc/systemd/journald.conf #Storage=auto [root@servera ~]# sed -i 's/#Storage=auto/Storage=persistent/' /etc/systemd/journald.conf [root@servera ~]# systemctl restart systemd-journald

Controlling Journal Size

The journal automatically manages its own size using rotation limits defined in journald.conf. Key size parameters:

The systemd journal maintains its disk usage within configured limits by automatically rotating (archiving old files) and deleting archived files when limits are reached. This is distinct from logrotate, which manages traditional text log files.

Querying Logs with journalctl

journalctl is the command-line tool for reading and filtering the systemd journal. Unlike browsing text log files with grep or tail, journalctl can filter on structured metadata fields, making it far more powerful for targeted investigations.

journalctl is the primary tool for querying and displaying systemd journal entries. It can filter by time, unit, priority, user, PID, and many other metadata fields, and it displays output through a pager for easy navigation.

Basic journalctl Usage

Basic journalctl commands
# Show all journal entries (oldest first, requires root for full access) [root@servera ~]# journalctl # Show most recent entries first (-r reverse) [root@servera ~]# journalctl -r # Follow new entries in real time (like tail -f) [root@servera ~]# journalctl -f # Show last N lines (-n) [root@servera ~]# journalctl -n 20 # Show journal in short, no-hostname format [root@servera ~]# journalctl -o short-monotonic

Filtering by Systemd Unit

One of the most powerful features of journalctl is filtering by systemd unit. The -u flag restricts output to messages from a specific service, making it far faster than grepping through combined log files.

A systemd unit is a configuration file that defines how systemd manages a resource. Unit types include .service (daemons), .timer (scheduled jobs), .socket (network listeners), .target (grouping units), and others.
Filtering by unit
# All messages from the sshd service [root@servera ~]# journalctl -u sshd -- Logs begin at Mon 2026-03-14 08:00:00 EDT, end at Sat 2026-03-14 10:15:00 EDT. -- Mar 14 08:00:15 servera sshd[1234]: Server listening on 0.0.0.0 port 22 Mar 14 08:00:15 servera sshd[1234]: Server listening on :: port 22 Mar 14 10:05:22 servera sshd[2891]: Accepted publickey for admin from 192.168.1.5 port 43812 # Follow sshd in real time [root@servera ~]# journalctl -u sshd -f # Multiple units (OR logic) [root@servera ~]# journalctl -u sshd -u httpd # Just the most recent 5 lines from crond [root@servera ~]# journalctl -u crond -n 5

Filtering by Priority and Time

Log Rotation with logrotate

Text log files under /var/log/ grow indefinitely unless managed. logrotate automates the process of archiving old log files, compressing them to save disk space, and purging archives older than a configured age. Without log rotation, busy servers can fill their disks entirely with log data within days.

logrotate is a utility that automatically renames, compresses, and deletes old log files on a schedule. It is typically run daily by a cron job or systemd timer, and its behavior is controlled by rules in /etc/logrotate.conf and /etc/logrotate.d/.

How logrotate Works

logrotate is triggered by a daily cron job or systemd timer (logrotate.timer). Each run processes all configured log files. When a log file meets its rotation criteria (size or age), logrotate:

On RHEL 10, logrotate runs via a systemd timer unit (logrotate.timer) rather than a traditional cron job. The timer is set to run daily. You can force an immediate run with systemctl start logrotate.service.
  1. Renames the current log file (e.g., messages → messages.1)
  2. Optionally compresses old rotated files using gzip (producing .gz archives)
  3. Signals the logging daemon to reopen its log file descriptors (postrotate)
  4. Deletes archives older than the configured rotate count
A postrotate script runs after log rotation is complete. It is used to signal daemons to reopen their log file handles. Without this signal, a daemon would continue writing to the renamed old file instead of the newly created one.

Configuration Files

logrotate has two levels of configuration:

  • /etc/logrotate.conf — Global defaults that apply to all log files unless overridden
  • /etc/logrotate.d/ — Per-service configuration files. Each installed package that uses log files typically ships its own file here (e.g., /etc/logrotate.d/httpd, /etc/logrotate.d/syslog)
/etc/logrotate.conf (global defaults)
# Rotate logs weekly by default weekly # Keep 4 weeks of rotated logs rotate 4 # Create new empty log file after rotation (with same permissions) create # Compress rotated files with gzip compress # Include per-package config files include /etc/logrotate.d
/etc/logrotate.d/myapp — custom application log rotation
/var/log/myapp/*.log { daily # Rotate every day (overrides global weekly) rotate 14 # Keep 14 days of archives compress # gzip rotated files delaycompress # Don't compress last rotation (easier to read) missingok # No error if log file is missing notifempty # Don't rotate empty log files sharedscripts # Run postrotate script once even for multiple files create 0640 myapp adm # New log: permissions, owner, group size 100M # Also rotate if file exceeds 100MB postrotate /bin/systemctl reload myapp.service 2>/dev/null || true endscript }

Centralized Remote Logging with rsyslog

In enterprise environments, collecting logs from dozens or hundreds of servers to a single central log server is a best practice for security auditing, compliance, and operational troubleshooting. rsyslog natively supports both sending and receiving logs over the network using TCP or UDP.

A central log server aggregates log messages from multiple client hosts into one location. This simplifies log analysis, ensures logs survive if a client system is compromised or crashes, and supports security compliance requirements.

Configuring the Log Server (Receiver)

The receiving server must load an input module to listen for incoming syslog connections. Use imudp for UDP or imtcp for TCP. TCP is preferred for reliability.

/etc/rsyslog.d/10-server.conf — on logserver.lab.example.com
# Load TCP input module and listen on port 514 module(load="imtcp") input(type="imtcp" port="514") # Optional: also accept UDP on port 514 module(load="imudp") input(type="imudp" port="514") # Store incoming messages in per-host directories # %HOSTNAME% is a rsyslog template variable $template RemoteLogs,"/var/log/remote/%HOSTNAME%/%PROGRAMNAME%.log" *.* ?RemoteLogs & ~ # Stop processing — don't also write to local /var/log/messages
Open firewall and restart rsyslog on the server
[root@logserver ~]# firewall-cmd --add-port=514/tcp --permanent success [root@logserver ~]# firewall-cmd --add-port=514/udp --permanent success [root@logserver ~]# firewall-cmd --reload success [root@logserver ~]# systemctl restart rsyslog [root@logserver ~]# ss -tlnp | grep :514 LISTEN 0 25 0.0.0.0:514 0.0.0.0:* users:(("rsyslogd",pid=3201,fd=4))

Configuring Log Clients (Senders)

On each client system, add a forwarding rule to rsyslog. The @@ prefix specifies TCP; @ specifies UDP.

/etc/rsyslog.d/50-forward.conf — on client systems
# Forward all messages at info level and above to the central server via TCP *.info @@logserver.lab.example.com:514 # For critical messages, ensure reliable delivery with queue $ActionQueueType LinkedList $ActionQueueFileName fwdRule1 # Spool to disk if server unreachable $ActionResumeRetryCount -1 # Retry forever *.emerg @@logserver.lab.example.com:514
Test the remote log forwarding
# On the client: send a test message [root@servera ~]# logger -p user.info "Test message from servera to central log server" # On the log server: verify message arrived [root@logserver ~]# ls /var/log/remote/ servera/ [root@logserver ~]# cat /var/log/remote/servera/root.log Mar 14 10:45:01 servera root: Test message from servera to central log server
🔴Security Considerations

Plain UDP/TCP syslog sends log data in cleartext, making it vulnerable to eavesdropping and tampering. For security-sensitive environments, configure rsyslog to use TLS encryption with the gtls module (module(load="imtcp" StreamDriver="gtls")). Also restrict which source IPs can connect to the log server using firewall rules or rsyslog’s built-in AllowedSender directive.

Eavesdropping means an attacker can read log traffic in transit; tampering means they can inject false log entries. Both attacks are trivial on unencrypted syslog. TLS (Transport Layer Security) encrypts the connection and verifies server identity with certificates.

Chapter Summary

In this chapter, you explored the full RHEL 10 logging stack, from the kernel to persistent storage to centralized collection. These skills are fundamental to system administration and are directly tested on the RHCSA exam.

Key Concepts Reviewed

  • RHEL 10 uses two complementary logging subsystems: systemd-journald collects structured binary events; rsyslog routes them to traditional text files under /var/log/.
  • Syslog messages carry two attributes: facility (source subsystem) and priority (severity 0–7). rsyslog uses facility.priority selectors to route messages.
  • rsyslog configuration lives in /etc/rsyslog.conf and /etc/rsyslog.d/*.conf. Always add custom rules in /etc/rsyslog.d/ and validate with rsyslogd -N1.
  • journald persistence requires creating /var/log/journal or setting Storage=persistent in journald.conf; use --vacuum-size and --vacuum-time to manage journal size.
  • journalctl filters by -u (unit), -p (priority), --since/--until (time), -b (boot), and -f (follow). Use -o verbose or -o json for detailed output.
  • logrotate automates log file archiving. Key directives: rotate, compress, daily/weekly, notifempty, postrotate. Test with -d (dry run).
  • Remote logging uses rsyslog input modules (imtcp/imudp) on the server and forward rules (@@host for TCP, @host for UDP) on clients.

Command Reference

journalctl -u UNIT -p PRIO

Query journal filtered by systemd unit and minimum priority

journalctl -b -1

Show messages from the previous system boot

journalctl --since "-1 hour" -f

Follow journal in real time, starting 1 hour back

journalctl --disk-usage

Report total disk space used by journal files

journalctl --vacuum-time=2weeks

Delete journal entries older than 2 weeks

journalctl --list-boots

List all recorded boot sessions with their IDs and timestamps

logger -p facility.priority "msg"

Send a test message to rsyslog at the specified facility and priority

rsyslogd -N1

Validate rsyslog configuration syntax without restarting

logrotate -d /etc/logrotate.d/FILE

Perform a dry run of a logrotate config (no changes made)

logrotate -f /etc/logrotate.d/FILE

Force immediate log rotation, ignoring schedule

systemctl restart rsyslog

Apply rsyslog configuration changes

systemctl restart systemd-journald

Apply journald configuration changes

🔴Important: RHCSA Exam Focus Areas

  • Know how to make the journal persistent (creating /var/log/journal)
  • Know how to write an rsyslog rule and place it in /etc/rsyslog.d/
  • Know the format of the syslog selector: facility.priority, and what *.none and = mean
