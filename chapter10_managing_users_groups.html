Introduction to Users and Groups

Linux is a multi-user operating system, meaning multiple users can use the system simultaneously. To manage access and security, Linux uses a system of A user is an entity that can access files, run processes, and perform various operations on the system. Each user has a unique User ID (UID) and belongs to at least one group. and A group is a collection of users that share common access permissions. Groups simplify permission management by allowing you to grant permissions to multiple users at once. to control access to files, directories, and system resources.

Why Users and Groups Matter

Key Benefits

  • Security: Users can only access files and resources they have permission to use
  • Accountability: Actions can be traced back to specific users through logging
  • Organization: Groups allow efficient management of permissions for multiple users
  • Isolation: Each user has their own home directory and environment settings
  • Resource Control: Quotas and limits can be applied per user or group

Types of User Accounts

Linux systems have three main types of user accounts:

Account Type UID Range Purpose Example
Superuser (root) 0 Full system access, administration root
System Users 1-999 Run system services and daemons apache, mysql, nobody
Regular Users 1000+ Interactive login for people student, jdoe, admin
üí° RHEL 10 UID/GID Ranges

In RHEL 10, the UID and GID ranges are defined in /etc/login.defs. Regular users start at UID 1000, and system users use UIDs 1-999. These ranges can be customized if needed.

Understanding User IDs (UIDs)

Every user account has a unique numeric identifier called a The UID is a numeric value that the kernel uses to identify users. While usernames are for human convenience, the system internally works with UIDs. UID 0 is always root. . The kernel uses UIDs internally to identify users and determine permissions.

Viewing User Information # Display current user's UID and groups [student@workstation ~]$ id uid=1000(student) gid=1000(student) groups=1000(student),10(wheel) # Display information for a specific user [student@workstation ~]$ id root uid=0(root) gid=0(root) groups=0(root) # Show just the username [student@workstation ~]$ whoami student

Understanding Linux Users

Each user account in Linux is defined by several key attributes stored in system configuration files. Understanding these attributes is essential for effective user management.

User Account Attributes

Attribute Description Stored In
Username Human-readable name for the account /etc/passwd
UID Unique numeric identifier /etc/passwd
GID Primary group ID /etc/passwd
GECOS/Comment Full name and other info /etc/passwd
Home Directory User's personal file space /etc/passwd
Login Shell Default command interpreter /etc/passwd
Password Hash Encrypted password /etc/shadow

The Root User

The The root user (also called superuser) has UID 0 and has unrestricted access to all files, commands, and system resources. Root can override any permission and perform any administrative task. (UID 0) is the superuser with unlimited privileges. Root can:

  • Read, write, or delete any file regardless of permissions
  • Create, modify, or delete user accounts
  • Install and remove software
  • Modify system configuration
  • Start and stop system services
‚ö†Ô∏è Security Warning

Avoid logging in directly as root for regular tasks. Use sudo to perform administrative tasks while logged in as a regular user. This provides better accountability and reduces the risk of accidental system damage.

System Users

System users (UIDs 1-999) are created to run system services. They typically:

  • Cannot log in interactively (shell set to /sbin/nologin)
  • Own files related to their service
  • Have limited permissions specific to their service
Example System Users # View system users [student@workstation ~]$ grep -E "nologin|false" /etc/passwd | head -5 bin:x:1:1:bin:/bin:/sbin/nologin daemon:x:2:2:daemon:/sbin:/sbin/nologin adm:x:3:4:adm:/var/adm:/sbin/nologin lp:x:4:7:lp:/var/spool/lpd:/sbin/nologin mail:x:8:12:mail:/var/spool/mail:/sbin/nologin

Understanding Linux Groups

Groups provide a way to organize users and grant permissions to multiple users simultaneously. Every user belongs to at least one group.

Primary vs. Supplementary Groups

Primary Group (Login Group)

Each user has exactly one primary group, specified in /etc/passwd.

  • The primary group is used for new files the user creates
  • By default, RHEL creates a private group with the same name as the username
  • This is called the User Private Group (UPG) scheme
  • The GID in /etc/passwd identifies the primary group
# User's primary group owns newly created files [student@workstation ~]$ touch newfile.txt [student@workstation ~]$ ls -l newfile.txt -rw-r--r--. 1 student student 0 Jan 29 10:00 newfile.txt

Supplementary (Secondary) Groups

Users can belong to multiple supplementary groups for additional permissions.

  • Listed in /etc/group
  • Grant access to shared resources
  • A user can belong to up to 65,535 supplementary groups
  • Common examples: wheel, developers, marketing
# View all groups a user belongs to [student@workstation ~]$ groups student student : student wheel # The 'groups' command shows primary first, then supplementary

Aspect Primary Group Supplementary Groups

Defined in /etc/passwd (GID field) /etc/group

The /etc/group File

Group information is stored in /etc/group. Each line defines one group with four colon-separated fields:

Structure of /etc/group
groupname : password : GID : members
wheel : x : 10 : student,admin
Field Description Example
Group Name Human-readable group identifier wheel
Password Usually 'x' (password in /etc/gshadow if used) x
GID Unique numeric Group ID 10
Members Comma-separated list of supplementary members student,admin
‚ÑπÔ∏è The wheel Group

The wheel group is special in RHEL. Members of this group can use sudo to run commands as root. Adding a user to the wheel group grants them administrative privileges.

The /etc/passwd File

The /etc/passwd file is the primary database for user account information. Despite its name, it does not contain passwords (those are in /etc/shadow). This file is readable by all users.

File Structure

Each line in /etc/passwd represents one user account with seven colon-separated fields:

Structure of /etc/passwd Entry
username: password: UID: GID: GECOS: home: shell
student: x: 1000: 1000: Student User: /home/student: /bin/bash

Field Descriptions

The username is the human-readable identifier for the account. It must:

  • Be unique on the system
  • Start with a lowercase letter or underscore
  • Contain only lowercase letters, digits, underscores, or dashes
  • Be no longer than 32 characters (recommended: 8 or fewer)

This field historically contained the encrypted password. Modern systems show:

  • x - Password is stored in /etc/shadow (most common)
  • * - Account is disabled / cannot login with password
  • Empty - No password required (dangerous!)

The numeric user identifier. Key ranges in RHEL 10:

  • 0 - Root (superuser)
  • 1-999 - System accounts
  • 1000-60000 - Regular users (default range)
  • 65534 - Often used for "nobody" user

The numeric ID of the user's primary group. With RHEL's UPG scheme:

  • Each user gets a private group with the same name
  • The GID typically matches the UID for regular users
  • New files are owned by this group

The GECOS field contains descriptive information about the user:

  • Typically contains the user's full name
  • Can include office location, phone numbers (comma-separated)
  • Used by programs like finger and mail systems
  • Can be changed with chfn command

The absolute path to the user's home directory:

  • Usually /home/username for regular users
  • Root's home is /root
  • System users may have service-specific directories
  • Set as the working directory upon login

The program that runs when the user logs in:

  • /bin/bash - Default shell for interactive users
  • /sbin/nologin - Prevents interactive login (system accounts)
  • /bin/false - Also prevents login
  • Valid shells are listed in /etc/shells

Viewing /etc/passwd

/etc/passwd Examples # View the entire file [student@workstation ~]$ cat /etc/passwd # View specific user [student@workstation ~]$ grep "^student:" /etc/passwd student:x:1000:1000:Student User:/home/student:/bin/bash # Use getent to query user database [student@workstation ~]$ getent passwd student student:x:1000:1000:Student User:/home/student:/bin/bash # Count total users [student@workstation ~]$ wc -l /etc/passwd 45 /etc/passwd

The /etc/shadow File

The /etc/shadow file stores encrypted passwords and password aging information. Unlike /etc/passwd, this file is only readable by root, protecting password hashes from unauthorized access.

File Structure

Each line has nine colon-separated fields:

Structure of /etc/shadow Entry
username:password:lastchange:min:max:warn:inactive:expire:reserved
student:$6$xyz...hash...:19750:0:99999:7:::
Field Name Description
1 Username Must match entry in /etc/passwd
2 Password Hash Encrypted password ($algorithm$salt$hash)
3 Last Change Days since Jan 1, 1970 when password was last changed
4 Minimum Age Minimum days before password can be changed
5 Maximum Age Maximum days before password must be changed
6 Warning Period Days before expiration to warn user
7 Inactive Period Days after expiration until account is disabled
8 Expiration Date Days since Jan 1, 1970 when account expires
9 Reserved Reserved for future use

Password Hash Format

The password hash field uses the format: $algorithm$salt$encrypted_hash

Algorithm ID Name Notes
$1$ MD5 Weak, not recommended
$5$ SHA-256 Good security
$6$ SHA-512 RHEL 10 default, recommended
$y$ yescrypt Modern, strong (optional in RHEL 10)

Special Password Field Values

# Locked account (!! or !) locked_user:!!:19750:0:99999:7::: # Never had a password (*) system_user:*:19000:0:99999:7::: # Empty field = no password required (dangerous!) risky_user::19750:0:99999:7:::

Viewing Password Aging Information

chage Command # View password aging info for current user [student@workstation ~]$ chage -l student Last password change : Jan 15, 2026 Password expires : never Password inactive : never Account expires : never Minimum number of days between password change : 0 Maximum number of days between password change : 99999 Number of days of warning before password expires : 7
üîí Security Note

Never edit /etc/shadow directly. Always use commands like passwd, chage, or usermod to modify password information. Direct editing can corrupt the file or introduce security vulnerabilities.

Managing Users and Groups

RHEL provides several commands for managing user accounts and groups. All user/group management commands require root privileges.

User Management Commands

useradd - Create New Users

The useradd command creates new user accounts.

Option Description
-c "comment" Set the GECOS/comment field
-d /path Specify home directory path
-g group Set primary group
-G group1,group2 Add to supplementary groups
-m Create home directory (default in RHEL)
-M Do not create home directory
-s /bin/shell Specify login shell
-u UID Specify user ID
-r Create a system account
# Create a basic user [root@workstation ~]# useradd jdoe # Create user with full details [root@workstation ~]# useradd -c "Jane Doe" -G wheel,developers jdoe # Create user with specific UID and shell [root@workstation ~]# useradd -u 2000 -s /bin/zsh newuser # Create a system user (no home, nologin shell) [root@workstation ~]# useradd -r -s /sbin/nologin appuser

usermod - Modify Existing Users

The usermod command modifies existing user accounts.

Option Description
-c "comment" Change the comment field
-d /path Change home directory
-g group Change primary group
-aG group Append to supplementary groups
-l newname Change username
-L Lock the account
-U Unlock the account
-s /bin/shell Change login shell
# Add user to wheel group (IMPORTANT: use -aG not just -G) [root@workstation ~]# usermod -aG wheel jdoe # Change user's shell [root@workstation ~]# usermod -s /bin/zsh jdoe # Lock a user account [root@workstation ~]# usermod -L jdoe # Unlock a user account [root@workstation ~]# usermod -U jdoe # Rename a user [root@workstation ~]# usermod -l jane_doe jdoe
‚ö†Ô∏è Important: -aG vs -G

Always use -aG (append to groups) when adding supplementary groups. Using just -G replaces all existing supplementary groups!

userdel - Delete Users

The userdel command removes user accounts.

Option Description
(none) Remove user but keep home directory
-r Remove user AND home directory/mail spool
-f Force removal even if user is logged in
# Remove user but keep their files [root@workstation ~]# userdel jdoe # Remove user and all their files [root@workstation ~]# userdel -r jdoe # Find files owned by deleted user's UID [root@workstation ~]# find / -uid 1001 -ls 2>/dev/null
üí° Best Practice

Before deleting a user, consider whether you need to preserve their files. If using userdel without -r, the orphaned files will show numeric UIDs in ls -l output.

passwd - Manage Passwords

The passwd command sets and manages user passwords.

Option Description
(none) Change your own password
username Change another user's password (root only)
-l Lock the password
-u Unlock the password
-d Delete password (make account passwordless)
-e Expire password (force change at next login)
-S Display password status
# Change your own password [student@workstation ~]$ passwd Changing password for user student. Current password: ******** New password: ******** Retype new password: ******** passwd: all authentication tokens updated successfully. # Set password for another user (as root) [root@workstation ~]# passwd jdoe Changing password for user jdoe. New password: ******** Retype new password: ******** passwd: all authentication tokens updated successfully. # Force password change at next login [root@workstation ~]# passwd -e jdoe Expiring password for user jdoe. passwd: Success # Check password status [root@workstation ~]# passwd -S jdoe jdoe PS 2026-01-29 0 99999 7 -1 (Password set, SHA512 crypt.)

Group Management Commands

groupadd - Create Groups

# Create a new group [root@workstation ~]# groupadd developers # Create a group with specific GID [root@workstation ~]# groupadd -g 5000 marketing # Create a system group [root@workstation ~]# groupadd -r appgroup # Verify the group was created [root@workstation ~]# getent group developers developers:x:1001:

groupmod - Modify Groups

# Rename a group [root@workstation ~]# groupmod -n dev_team developers # Change a group's GID [root@workstation ~]# groupmod -g 5001 dev_team

groupdel - Delete Groups

# Delete a group [root@workstation ~]# groupdel developers # Note: You cannot delete a group if it is any user's primary group [root@workstation ~]# groupdel jdoe groupdel: cannot remove the primary group of user 'jdoe'

Gaining Superuser Access with sudo

The sudo (superuser do) allows permitted users to run commands as root or another user. It provides better security than sharing the root password by logging all commands and limiting what users can do. command allows authorized users to run commands with elevated privileges without logging in as root. This is the preferred method for performing administrative tasks.

Why Use sudo Instead of root?

Benefits of sudo

  • Accountability: All sudo commands are logged with the username
  • Granular Control: Limit which commands users can run
  • No Shared Password: Users authenticate with their own password
  • Temporary Access: Elevated privileges only for specific commands
  • Reduced Risk: Less chance of running dangerous commands accidentally

The wheel Group

In RHEL, members of the wheel group can use sudo to run any command as root. This is configured in /etc/sudoers.

Using sudo # Run a single command as root [student@workstation ~]$ sudo useradd newuser [sudo] password for student: ******** # Run a command as a different user [student@workstation ~]$ sudo -u apache cat /etc/httpd/conf/httpd.conf # Start an interactive root shell [student@workstation ~]$ sudo -i [root@workstation ~]# # Check what you can run with sudo [student@workstation ~]$ sudo -l User student may run the following commands on workstation: (ALL) ALL

Granting sudo Access

To give a user full sudo privileges, add them to the wheel group:

# Add existing user to wheel group [root@workstation ~]# usermod -aG wheel jdoe # Or create user with wheel membership [root@workstation ~]# useradd -G wheel -c "Jane Doe" jdoe # Verify wheel membership [root@workstation ~]# groups jdoe jdoe : jdoe wheel
‚ÑπÔ∏è Group Changes Take Effect

After adding a user to a group, they must log out and log back in for the new group membership to take effect. Alternatively, use newgrp wheel to activate the group in the current session.

Configuring sudo

sudo is configured in /etc/sudoers. Always edit this file using visudo, which validates syntax before saving:

/etc/sudoers Configuration # Edit sudoers safely [root@workstation ~]# visudo # Example sudoers entries: # Allow root to run any commands anywhere root ALL=(ALL) ALL # Allow members of wheel group to run all commands %wheel ALL=(ALL) ALL # Allow jdoe to run specific commands without password jdoe ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart httpd # Allow developers group to run commands as apache user %developers ALL=(apache) ALL

sudo vs su

Feature sudo su
Password Required User's own password Target user's password
Logging Detailed logging to /var/log/secure Minimal logging
Granular Control Yes, per command/user No, all or nothing

Lab: Managing Local Users and Groups

In this lab, you will create user accounts, manage groups, set passwords, and configure sudo access.

Step 1 Create a New Group

Create a group called developers for your development team.

[root@workstation ~]# groupadd developers [root@workstation ~]# getent group developers developers:x:1001:

Step 2 Create User Accounts

Create two users: alice (a developer) and bob (a system administrator).

# Create alice as a developer [root@workstation ~]# useradd -c "Alice Developer" -G developers alice # Create bob as an administrator [root@workstation ~]# useradd -c "Bob Admin" -G wheel bob # Verify the users were created [root@workstation ~]# id alice uid=1001(alice) gid=1001(alice) groups=1001(alice),1002(developers) [root@workstation ~]# id bob uid=1002(bob) gid=1002(bob) groups=1002(bob),10(wheel)

Step 3 Set User Passwords

Set passwords for both users and force Alice to change her password on first login.

# Set passwords [root@workstation ~]# passwd alice Changing password for user alice. New password: redhat123 Retype new password: redhat123 passwd: all authentication tokens updated successfully. [root@workstation ~]# passwd bob Changing password for user bob. New password: redhat456 Retype new password: redhat456 passwd: all authentication tokens updated successfully. # Force Alice to change password at next login [root@workstation ~]# passwd -e alice Expiring password for user alice.

Step 4 View Password Aging Information

Check the password status and aging information for alice.

[root@workstation ~]# chage -l alice Last password change : password must be changed Password expires : password must be changed Password inactive : password must be changed Account expires : never ...

Step 5 Test sudo Access

Verify that Bob can use sudo (as a wheel member) but Alice cannot.

# Switch to bob and test sudo [root@workstation ~]# su - bob [bob@workstation ~]$ sudo whoami [sudo] password for bob: ******** root [bob@workstation ~]$ exit # Switch to alice and try sudo (should fail) [root@workstation ~]# su - alice [alice@workstation ~]$ sudo whoami [sudo] password for alice: ******** alice is not in the sudoers file. This incident will be reported.

Step 6 Modify User Group Membership

Add Alice to the wheel group so she can also use sudo.

[root@workstation ~]# usermod -aG wheel alice [root@workstation ~]# id alice uid=1001(alice) gid=1001(alice) groups=1001(alice),10(wheel),1002(developers)

Step 7 Clean Up (Optional)

Remove the users and group created in this lab.

[root@workstation ~]# userdel -r alice [root@workstation ~]# userdel -r bob [root@workstation ~]# groupdel developers

Chapter Summary

Key Takeaways

  • Linux uses UIDs and GIDs to identify users and groups internally. Root has UID 0, system users use 1-999, and regular users start at 1000.
  • /etc/passwd stores user account information including username, UID, GID, home directory, and shell. It is readable by all users.
  • /etc/shadow stores encrypted passwords and password aging information. It is only readable by root for security.
  • /etc/group stores group definitions and supplementary group memberships.
  • useradd, usermod, userdel create, modify, and remove user accounts. Always use -aG with usermod to append groups.
  • groupadd, groupmod, groupdel manage groups. You cannot delete a user's primary group.
  • passwd sets user passwords. Use passwd -e to force password change at next login.
  • sudo is the preferred method for running administrative commands. Add users to the wheel group for full sudo access.
