Logical Volume Manager Overview

Traditional disk partitioning has a significant limitation: once you create a partition of a certain size, resizing it requires unmounting the file system, repartitioning the disk, and potentially moving data—all of which cause downtime. On production servers running databases, web servers, or other critical services, this is unacceptable.

The Logical Volume Manager (LVM) solves this by inserting a virtualization layer between the physical hardware and the operating system. File systems no longer interact directly with a physical disk partition; instead, they sit on top of a logical volume that LVM manages. The result is flexible, live storage management.

The Three-Layer Architecture

LVM organizes storage into three abstraction layers. Understanding each layer is fundamental to working effectively with LVM.

Physical Devices

At the base of the LVM stack are the raw block devices. These can be any of the following: whole disks (e.g., /dev/sdb), disk partitions (/dev/sdb1), RAID arrays (/dev/md0), or SAN (Storage Area Network) LUNs.

Before using a device in LVM, you must initialize the entire device—not just part of it—as an LVM physical volume. Partitioning is optional; you can pass a whole raw disk directly to pvcreate. When you do use partitions, mark them with the lvm flag using the parted command so the kernel recognizes the intent.

📝Note

Device names like /dev/sdb are assigned by the kernel during boot and can change between reboots depending on hardware order. LVM internally tracks devices by UUID, so volume groups remain accessible even if device names shift. Always use lsblk, blkid, or cat /proc/partitions to verify device names on your specific system.

Physical Volumes (PVs)

A physical volume A raw block device or partition that has been labeled by pvcreate to store LVM metadata. The metadata is written to a small reserved area at the beginning of the device. is a block device that has been labeled with LVM metadata by the pvcreate command. The metadata includes a UUID that uniquely identifies the PV, and the device is then divided into small, fixed-size units called physical extents (PEs) The smallest allocatable unit in a physical volume, defaulting to 4 MiB. LVM assigns logical extents in a volume group by mapping them to these physical extents on the underlying device..

Physical extents are the fundamental unit of storage allocation in LVM. By default, each PE is 4 MiB. When you create a logical volume, LVM allocates a certain number of PEs from the volume group's pool. The LV size is always a multiple of the PE size.

Volume Groups (VGs)

A volume group combines one or more PVs into a single storage pool. Think of a VG as a virtual disk with a fixed total capacity—but one whose capacity can be increased at any time by adding more PVs.

A PV can only belong to one VG at a time. Within the VG, storage is tracked in units of logical extents (LEs) A logical extent is the VG-level unit of allocation, the same size as a PE. Each LE in a logical volume maps to one PE in the underlying physical volume by default., which correspond one-to-one to PEs. The VG maintains a map of which LEs in each LV are assigned to which PEs on which PVs.

Logical Volumes (LVs)

Logical volumes are created from the free extents in a VG. To the operating system and users, an LV appears as a regular block device (/dev/vg01/lv_data or /dev/mapper/vg01-lv_data)—you can format it with a file system, use it as swap space, or pass it to a database engine.

The critical advantage: LVs can be resized online. Because the mapping from LEs to PEs is maintained in the VG metadata, LVM can add or remove extents from an LV without disrupting running applications. This is the single most important capability LVM provides over raw partitions.

Why Use LVM? Key Advantages

↗ Online Resizing
Extend LVs and their file systems while applications continue running—no downtime needed for storage growth.
🔢 Storage Pooling
Combine multiple physical disks of different sizes into one VG and allocate storage from that pool freely.
📷 Snapshots
Create point-in-time snapshots of LVs for backups or testing without stopping services.
🔄 PV Migration
Move data off a failing disk (pvmove) without downtime, then remove the old device safely.

LVM Workflow Summary

Creating LVM storage always follows this sequence:

  1. Prepare physical devices (optional partitioning with parted)
  2. Initialize devices as physical volumes: pvcreate /dev/sdb1 /dev/sdb2
  3. Create a volume group from the PVs: vgcreate vg01 /dev/sdb1 /dev/sdb2
  4. Create logical volumes from the VG: lvcreate -n lv01 -L 300M vg01
  5. Format the LV with a file system: mkfs -t xfs /dev/vg01/lv01
  6. Mount persistently via /etc/fstab

Creating Logical Volumes

Building LVM storage is a multi-step process. Each step builds on the last, and each layer has its own set of commands. Let's walk through the complete workflow from raw disk to mounted file system.

Step 1 — Prepare Physical Devices

While partitioning is optional (you can use a whole disk), it is common practice on shared disks. Use parted to create partitions and label them for LVM:

Partition a disk for LVM use
# Label the disk with a GPT partition table and create first partition [root@host ~]# parted /dev/sdb mklabel gpt mkpart primary 1MiB 769MiB ...output omitted... # Create a second partition [root@host ~]# parted /dev/sdb mkpart secondary 770MiB 1026MiB # Set the lvm flag on both partitions [root@host ~]# parted /dev/sdb set 1 lvm on [root@host ~]# parted /dev/sdb set 2 lvm on # Register new partitions with the running kernel [root@host ~]# udevadm settle
💡Tip

The udevadm settle command waits until the kernel's device manager has fully processed all recent device events. Always run it after creating partitions so that new device files like /dev/sdb1 and /dev/sdb2 are available before proceeding.

Step 2 — Create Physical Volumes

The pvcreate Writes LVM metadata headers to a device, turning it into a physical volume. Multiple devices can be initialized in a single command by listing them space-delimited. command writes LVM metadata to one or more devices, marking them as physical volumes. You can process multiple devices in a single invocation:

Initialize physical volumes
[root@host ~]# pvcreate /dev/sdb1 /dev/sdb2 Physical volume "/dev/sdb1" successfully created. Physical volume "/dev/sdb2" successfully created. Creating devices file /etc/lvm/devices/system.devices

LVM also writes an entry to /etc/lvm/devices/system.devices—a RHEL 10 feature that maintains an explicit list of devices allowed for use in LVM. This improves security and prevents accidental LVM operations on unexpected devices.

Step 3 — Create a Volume Group

The vgcreate Creates a new volume group by combining one or more physical volumes into a shared storage pool. The first argument is the VG name; subsequent arguments are PV device paths. command creates a volume group by pooling one or more PVs. The first argument is the VG name you choose; subsequent arguments are the PV device paths:

Create a volume group
[root@host ~]# vgcreate vg01 /dev/sdb1 /dev/sdb2 Volume group "vg01" successfully created

Choose VG names that reflect their purpose (e.g., vg_data, vg_logs). Once created, the VG appears as a combined storage pool whose total capacity is the sum of all PV sizes minus a small amount reserved for LVM metadata.

Step 4 — Create a Logical Volume

The lvcreate Allocates extents from a volume group to create a new logical volume. Use -n for the LV name, -L for size in MiB/GiB, or -l for size in physical extents. command carves a logical volume out of free space in a VG:

Create a logical volume
# Create lv01 with 300 MiB from vg01 [root@host ~]# lvcreate -n lv01 -L 300M vg01 Logical volume "lv01" created. # Equivalent: specify 32 physical extents (32 x 4 MiB = 128 MiB) [root@host ~]# lvcreate -n lv01 -l 32 vg01
💡Tip

The -L option accepts M for mebibytes (MiB) and G for gibibytes (GiB). The -l option (lowercase L) accepts the number of physical extents. LVM rounds the requested size up to the nearest PE boundary if the size does not exactly match.

Step 5 — Format and Mount the Logical Volume

Logical volumes appear as block devices under two equivalent paths:

  • Traditional: /dev/vgname/lvname (e.g., /dev/vg01/lv01)
  • Device mapper: /dev/mapper/vgname-lvname (e.g., /dev/mapper/vg01-lv01)

Either path can be used interchangeably in commands and /etc/fstab.

Create XFS file system, mount point, and persistent fstab entry
# Format with XFS [root@host ~]# mkfs -t xfs /dev/vg01/lv01 ...output omitted... # Create a mount point [root@host ~]# mkdir /mnt/data # Add entry to /etc/fstab for persistent mounting /dev/vg01/lv01 /mnt/data xfs defaults 0 0 # Reload systemd to recognize the new fstab entry [root@host ~]# systemctl daemon-reload # Mount the volume [root@host ~]# mount /mnt/data
📝Note

You can reference an LV in /etc/fstab by its traditional device name (/dev/vg01/lv01) or by its UUID A universally unique identifier assigned to a file system when it is formatted. UUIDs are stable across reboots and disk changes, making them ideal for /etc/fstab entries.. LVM resolves the device by scanning PV metadata for the VG UUID during boot, so the traditional path also works reliably even if device names change.

Displaying LVM Component Status

LVM provides two levels of status commands for each component: a verbose display command and a compact summary command. Knowing both helps you troubleshoot and verify your storage configuration.

The pvdisplay Shows detailed information about a physical volume including its VG assignment, total PE count, free PEs, and PE size. Run without arguments to show all PVs. command shows detailed PV information:

pvdisplay /dev/sdb1
[root@host ~]# pvdisplay /dev/sdb1 --- Physical volume --- PV Name /dev/sdb1 ① Device name VG Name vg01 ② Assigned volume group PV Size 768.00 MiB / not usable 4.00 MiB ③ Total size Allocatable yes PE Size 4.00 MiB ④ Extent size Total PE 191 Free PE 116 ⑤ Available extents Allocated PE 75 PV UUID evAimq-gs8W-6X9C-3H6q-ndDk-pvTx-QEZ9Ic

Key fields to note: Free PE tells you how many extents remain available for allocating new LVs or extending existing ones. Allocated PE shows how much of this PV is already in use.

The vgdisplay Shows the total size, PE configuration, current PV and LV counts, and free space for a volume group. Essential for planning LV creation and extension. command shows VG details including available free space:

vgdisplay vg01
[root@host ~]# vgdisplay vg01 --- Volume group --- VG Name vg01 ① Name of the VG Format lvm2 Metadata Areas 2 VG Access read/write VG Status resizable Cur PV 2 VG Size 1016.00 MiB ② Total pool size PE Size 4.00 MiB Total PE 254 ③ Total extents in pool Alloc PE / Size 75 / 300.00 MiB Free PE / Size 179 / 716.00 MiB ④ Available for new LVs VG UUID F5a4Um-Ln8v-cYM0-YJth-f7fj-1QYZ-1AP8Zu

Always check Free PE / Size before creating or extending an LV. If there is not enough free space, you must first extend the VG by adding a new PV.

The lvdisplay Displays detailed information about a logical volume including its size, the VG it belongs to, the number of logical extents used, and its current status (available or suspended). command shows details for a specific LV:

lvdisplay /dev/vg01/lv01
[root@host ~]# lvdisplay /dev/vg01/lv01 --- Logical volume --- LV Path /dev/vg01/lv01 ① Device path LV Name lv01 VG Name vg01 ② Parent VG LV UUID 7Toe5z-022v-aW41-mJYQ-2VOV-MiP2-hNGFCf LV Write Access read/write LV Status available LV Size 300.00 MiB ③ Total LV size Current LE 75 ④ Logical extents used Segments 1 Block device 253:0

Important: LV Size shows the size of the logical volume itself, not the amount of data stored. Use df -h /mount/point to see file system usage (used vs. available space).

The compact summary commands—pvs, vgs, and lvs—show one line per entity and are ideal for a quick overview:

Compact LVM status commands
[root@host ~]# pvs PV VG Fmt Attr PSize PFree /dev/sdb1 vg01 lvm2 a-- 768.00m 464.00m /dev/sdb2 vg01 lvm2 a-- 248.00m 248.00m [root@host ~]# vgs VG #PV #LV #SN Attr VSize VFree vg01 2 1 0 wz--n- 1016.00m 716.00m [root@host ~]# lvs LV VG Attr LSize Pool Origin Data% Meta% Move Log Cpy%Sync Convert lv01 vg01 -wi-ao---- 300.00m

Extending a Logical Volume

One of LVM's greatest advantages is the ability to increase storage capacity while applications continue running. Extending an LV is a two-step process: first extend the logical volume block device, then expand the file system inside it to use the new space.

Extend the Logical Volume

The lvextend Adds physical extents to an existing logical volume. Use +SIZE to add to the current size, or SIZE without a plus sign to set the final absolute size. command adds extents to an existing LV. Prefix the size with + to add to the current size:

Extend lv01 by 500 MiB
[root@host ~]# lvextend -L +500M /dev/vg01/lv01 Size of logical volume vg01/lv01 changed from 300.00 MiB (75 extents) to 800.00 MiB (200 extents). Logical volume vg01/lv01 successfully resized.

Without the + prefix, the number specifies the final size rather than the amount to add. For example, lvextend -L 1024M /dev/vg01/lv01 sets the total size to 1024 MiB regardless of the current size.

🔴Important

Extending the LV only changes the block device size. The file system inside does not automatically grow. You must explicitly resize the file system as the next step; otherwise, the extra space is inaccessible.

Resize the File System

The resize command differs depending on the file system type:

The xfs_growfs Expands an XFS file system to fill the remaining space in its underlying block device. Takes the mount point as its argument and requires the file system to be mounted. XFS can only be grown, not shrunk. command expands an XFS file system to fill the extended LV. It requires the file system to be mounted:

Grow an XFS file system
[root@host ~]# xfs_growfs /mnt/data ...output omitted... data blocks changed from 76800 to 204800
⚠️Warning

XFS file systems can only be grown, never shrunk. Do not attempt to reduce the size of an XFS LV; it will result in data loss or a corrupted file system.

The resize2fs Resizes ext2, ext3, or ext4 file systems. Takes the device path (not the mount point) as its argument. Supports both online (mounted) and offline resizing, and can grow or shrink the file system. command resizes ext2, ext3, and ext4 file systems. Unlike xfs_growfs, it accepts the device path as its argument:

Grow an EXT4 file system
[root@host ~]# resize2fs /dev/vg01/lv01 resize2fs 1.47.1 (20-May-2024) Resizing the filesystem on /dev/vg01/lv01 to 259072 (4k) blocks. The filesystem on /dev/vg01/lv01 is now 259072 (4k) blocks long.

Unlike XFS, ext4 supports both growing and shrinking. To shrink, you must first unmount the file system, then specify a smaller size: resize2fs /dev/vg01/lv01 200M. Then use lvreduce to shrink the LV. Always back up data before shrinking.

Logical volumes used as swap space An area of disk used as virtual memory overflow when RAM is full. Swap can be a partition, a file, or a logical volume formatted with mkswap. must be taken offline before extension. The process requires turning swap off, extending the LV, reformatting the swap space, and re-enabling it:

Extend a swap logical volume
# Deactivate swap on the LV [root@host ~]# swapoff -v /dev/vg01/swap swapoff /dev/vg01/swap # Extend the LV [root@host ~]# lvextend -L +300M /dev/vg01/swap Logical volume vg01/swap successfully resized. # Reformat as swap (overwrites old swap signature) [root@host ~]# mkswap /dev/vg01/swap Setting up swapspace version 1, size = 800 MiB (838856704 bytes) # Re-enable swap [root@host ~]# swapon /dev/vg01/swap
🔴Important

Be cautious when running swapoff on a busy system. If the system is actively using swap due to memory pressure, turning it off may trigger OOM (out-of-memory) The out-of-memory condition occurs when the Linux kernel cannot allocate memory for a process. The OOM killer then terminates processes to free memory, which can cause application crashes or system instability. errors. Alternatively, create a new swap LV, enable it, then disable and remove the old one.

The lvextend -r option (resize) combines the LV extension and file system resize into a single command. This is the recommended method for XFS and ext4 file systems because it eliminates the risk of forgetting the file system resize step:

Extend LV and resize file system in one step
# Extend the LV AND resize the file system automatically [root@host ~]# lvextend -L +500M -r /dev/vg01/lv01 Size of logical volume vg01/lv01 changed from 300.00 MiB to 800.00 MiB. Logical volume vg01/lv01 successfully resized. ...output omitted... data blocks changed from 76800 to 204800

If the LV is not currently mounted, the -r option creates a temporary mount point to perform the file system resize, then unmounts it. This makes -r safe to use in both mounted and unmounted scenarios for supported file systems.

Adding Space to a Full Volume Group

If the volume group has no free extents, you must expand it before extending an LV. The vgextend Adds one or more new physical volumes to an existing volume group, increasing its total capacity. The PV must already be initialized with pvcreate. command adds new PVs to an existing VG:

Add a new PV to expand the volume group
# Initialize a new device as a PV [root@host ~]# pvcreate /dev/sdb3 Physical volume "/dev/sdb3" successfully created. # Add it to the existing VG [root@host ~]# vgextend vg01 /dev/sdb3 Volume group "vg01" successfully extended

Replacing Physical Volumes and Removing LVM Storage

Storage hardware eventually fails or needs replacement. LVM provides tools to migrate data off a failing disk while keeping services online, then safely remove the old device. This section also covers cleanly decommissioning LVM components that are no longer needed.

Replacing a Physical Volume

When a PV must be replaced (due to hardware failure, insufficient space, or planned upgrade), follow this process:

  1. Add the new disk to the VG using pvcreate and vgextend
  2. Move data off the old PV using pvmove
  3. Remove the old PV from the VG using vgreduce
  4. Wipe the old PV metadata using pvremove

Migrate Data with pvmove

The pvmove Moves physical extents from one PV to other PVs in the same VG that have free extents. The operation is online and the LV remains accessible during the move, though performance may be reduced. command migrates all extents from a specified PV to other PVs in the same VG with free space. The LV remains accessible throughout:

Move data off /dev/sdb1
# Move with automatic VG metadata backup after each segment [root@host ~]# pvmove -A y /dev/sdb1 /dev/sdb1: Moved: 2.74% /dev/sdb1: Moved: 41.21% ...output omitted... /dev/sdb1: Moved: 100.00%
⚠️Warning

Always back up data before running pvmove. An unexpected power loss during the operation can leave the VG in an inconsistent state and may cause data loss on all LVs in the VG. The -A y option instructs LVM to back up VG metadata automatically after each move segment, but this does not replace a full data backup.

Remove PV from VG with vgreduce

Once all extents have been moved off a PV (confirmed by checking that its Free PE equals its Total PE), remove it from the VG:

Remove /dev/sdb1 from vg01
[root@host ~]# vgreduce vg01 /dev/sdb1 Removed "/dev/sdb1" from volume group "vg01"

Removing LVM Components

When an LVM component is no longer needed, remove it in reverse order: LV first, then VG, then PV. Each command is irreversible—data is permanently lost.

⚠️Warning

The lvremove, vgremove, and pvremove commands are destructive and unrecoverable. Always unmount file systems and verify data is no longer needed before running these commands. Removing an LV deletes all data on it permanently.

Complete LVM teardown sequence
# 1. Unmount the file system and remove fstab entry [root@host ~]# umount /mnt/data # 2. Remove the logical volume (prompts for confirmation) [root@host ~]# lvremove /dev/vg01/lv01 Do you really want to remove active logical volume vg01/lv01? [y/n]: y Logical volume "lv01" successfully removed. # 3. Remove the volume group [root@host ~]# vgremove vg01 Volume group "vg01" successfully removed # 4. Wipe PV metadata from devices (space-delimited for multiple) [root@host ~]# pvremove /dev/sdb2 /dev/sdb3 Labels on physical volume "/dev/sdb2" successfully wiped. Labels on physical volume "/dev/sdb3" successfully wiped.

Chapter Summary

This chapter introduced the Logical Volume Manager, Red Hat's preferred storage abstraction layer for production systems. LVM provides flexibility, online resizeability, and safe PV migration that raw disk partitioning cannot match.

Key Concepts

  • Physical Volumes (PVs) are block devices initialized with pvcreate. They are divided into fixed-size physical extents (default 4 MiB).
  • Volume Groups (VGs) pool one or more PVs into a single storage reservoir. Created with vgcreate, expanded with vgextend, and shrunk with vgreduce.
  • Logical Volumes (LVs) are carved from VG free space using lvcreate. They appear as regular block devices, can be formatted with any file system, and can be resized online.
  • Extending an LV requires two steps: lvextend to grow the block device, then xfs_growfs or resize2fs to grow the file system. The -r option to lvextend combines both steps.
  • Replacing a PV uses pvmove to migrate data online, then vgreduce to remove the PV from the VG.
  • LVM components are removed in reverse order: lvremove → vgremove → pvremove.
