After completing this chapter, you will be able to:

Transferring Files with SFTP

The OpenSSH suite is best known for the ssh command, which opens an encrypted remote shell. But the suite also includes tools for secure file transfer. The most important of these is the SFTP subsystem, accessed through the sftp command.

A freely available implementation of the SSH (Secure Shell) protocol. It provides encrypted remote login, remote command execution, and file transfer utilities like sftp and scp. Secure File Transfer Program. A subsystem of SSH that provides file transfer and management capabilities over an encrypted channel. Unlike FTP, SFTP encrypts both commands and data.

SFTP provides an interactive session with its own set of commands for navigating and manipulating files on both the local and remote system — all within a single encrypted connection.

A terminal mode where you type commands and receive responses one at a time, rather than specifying all instructions upfront. The sftp prompt (sftp>) indicates you are in an interactive session.

Opening an SFTP Session

To start an SFTP session, provide the remote user and hostname using the format user@host. If you omit the username, SFTP uses your current local username.

Opening an interactive SFTP session
[student@workstation ~]$ sftp remoteuser@remotehost remoteuser@remotehost's password: password Connected to remotehost. sftp>

Navigating in SFTP

Once inside the SFTP session, familiar shell commands work on the remote system. To run the equivalent command on your local system, prefix it with the letter l (lowercase L).

pwd
Print current working directory on the remote host.
ls
List files in the current remote directory.
cd path
Change directory on the remote host.
mkdir dir
Create a new directory on the remote host.
rmdir dir
Remove an empty directory on the remote host.
rm file
Delete a file on the remote host.
lpwd
Print current working directory on the local host.
lls
List files in the current local directory.
lcd path
Change directory on the local host.
lmkdir dir
Create a directory on the local host.
💡Tip: The “l” prefix

Think of the l prefix as standing for “local.” When in doubt, try prefixing any remote command with l to see if a local equivalent exists.

put localfile
Upload a file to the remote system.
get remotefile
Download a file from the remote system.
put -r dir
Recursively upload an entire directory tree.
get -r dir
Recursively download an entire directory tree.
help
List all available SFTP commands.
exit / bye
Close the SFTP session and return to the local shell.

Uploading Files with put

The put command transfers a file from your local machine to the remote host. The following example creates a directory on the remote host, navigates into it, and uploads the local /etc/hosts file.

Uploading /etc/hosts to a remote directory
sftp> mkdir hostbackup sftp> cd hostbackup sftp> put /etc/hosts Uploading /etc/hosts to /home/remoteuser/hostbackup/hosts /etc/hosts 100% 227 0.2KB/s 00:00

To upload an entire directory tree recursively, add the -r flag:

Recursively uploading a directory
sftp> put -r directory Uploading directory/ to /home/remoteuser/directory Entering directory/ file1 100% 0 0.0KB/s 00:00 file2 100% 0 0.0KB/s 00:00

Downloading Files with get

The get command downloads a file from the remote host to your current local directory.

Downloading /etc/dnf/dnf.conf from the remote host
sftp> get /etc/dnf/dnf.conf Fetching /etc/dnf/dnf.conf to dnf.conf /etc/dnf/dnf.conf 100% 813 0.8KB/s 00:00

Non-Interactive (Single-Command) Download

You can download a single file from a remote host in one command line, without opening an interactive session. Specify the remote file path after the host: separator. Note that this non-interactive syntax only works for downloads — you cannot push files to a remote host this way.

Downloading a single file without opening an interactive session
[student@workstation ~]$ sftp remoteuser@remotehost:/home/remoteuser/remotefile Connected to remotehost. Fetching /home/remoteuser/remotefile to remotefile remotefile 100% 0 15.7KB/s 00:00

Try the SFTP Simulator

Practice SFTP commands in this browser-based simulator. Type a command or click one of the shortcut chips below the input, then press Run or Enter.

SFTP Simulator — remoteuser@serverb

Transferring Files with SCP

The Secure Copy Protocol (SCP), accessed via the scp command, provides a quick, non-interactive way to copy files between a local and a remote system. Unlike sftp, you specify everything — source, destination, options — on a single command line.

Secure Copy Protocol. A command-line tool for copying files between hosts using SSH for authentication and encryption. Starting in RHEL 10, the scp command uses the SFTP protocol for transport.
🔴RHEL 10 Change: scp Now Uses SFTP

Starting in RHEL 10, the scp command uses SFTP to transfer files by default. The legacy SCP protocol had a code injection vulnerability (CVE-2020-15778). Red Hat recommends you do not use the -O flag to revert to the old protocol. Additionally, if the file /etc/ssh/disable_scp exists on a system, the -O option is completely disabled.

Syntax

The scp command follows the same user@host:path convention used by sftp. You can use this format for either the source or the destination. If you omit the username, scp uses your current local username.

Copying Files to a Remote Host

Specify one or more local source files followed by the remote destination. The colon (:) after the hostname signals that this is a remote path.

Copying two local files to a remote directory
[student@workstation ~]$ scp /etc/dnf/dnf.conf /etc/hosts remoteuser@remotehost:/home/remoteuser remoteuser@remotehost's password: password dnf.conf 100% 813 0.8KB/s 00:00 hosts 100% 227 0.2KB/s 00:00

Copying Files from a Remote Host

Reverse the source and destination arguments to pull a file from a remote system to your local machine.

Copying /etc/hostname from a remote host to local ~/
[student@workstation ~]$ scp remoteuser@remotehost:/etc/hostname /home/user remoteuser@remotehost's password: password hostname 100% 12 0.0KB/s 00:00

Synchronizing Content with rsync

While sftp and scp copy entire files every time you run them, rsync is smarter. It compares the source and destination and transfers only the differences. This makes subsequent synchronizations far faster, particularly over slow or metered network connections.

A fast, versatile file-copying tool that uses a delta-transfer algorithm to minimize data transfer. It can sync files locally or over an SSH connection, preserving permissions, timestamps, and ownership.

Because rsync uses delta-transfer, an initial sync takes about the same time as a plain copy. But subsequent syncs — after a few files have changed — transfer only what changed, dramatically reducing bandwidth and time.

An algorithm that identifies and transmits only the portions of a file that have changed since the last transfer, rather than sending the entire file again.

Core Syntax

The general format mirrors scp and sftp — use user@host:path for remote locations. Either the source or destination can be remote, but not both.

rsync basic syntax
# Push: local source → remote destination [root@serverb ~]# rsync -av /var/log hosta:/tmp # Pull: remote source → local destination [root@serverb ~]# rsync -av hosta:/var/log /tmp # Local-to-local sync [student@workstation ~]$ sudo rsync -av /var/log /tmp

The Dry Run: Always Preview First

Before running a potentially destructive rsync command, use the -n (dry run) option. This shows exactly what would happen without actually making any changes.

A simulation mode (rsync -n) that reports what the command would do without modifying any files. Essential for verifying that the correct files will be synced before committing.
Previewing changes with a dry run
[root@serverb ~]# rsync -avn /var/log hosta:/tmp sending incremental file list log/ log/boot.log log/dnf.log ...output omitted... sent 1,254 bytes received 20 bytes 2,548.00 bytes/sec # No files were actually transferred — this was a simulation

Archive Mode (-a): Preserving Everything

The -a flag enables archive mode, the most common way to use rsync for backups and replication. It is a shorthand for a set of options that together preserve the most important file attributes.

An rsync flag (-a) equivalent to -rlptgoD: recursive, symbolic links, permissions, timestamps, group ownership, owner, and device files. It ensures the destination mirrors the source as faithfully as possible.
OptionDescription
-r, --recursiveSynchronize the entire directory tree recursively.
-l, --linksPreserve symbolic links as symbolic links.
-p, --permsPreserve file permissions.
-t, --timesPreserve modification timestamps.
-g, --groupPreserve group ownership.
-o, --ownerPreserve the file owner (requires root).
-D, --devicesPreserve device files.
💡Tip: Extended Attributes

Archive mode does not automatically preserve ACLs or SELinux file contexts. Add -A (preserve ACLs) and -X (preserve SELinux contexts) when you need full fidelity. Hard links also require the separate -H flag, as preserving them can significantly increase sync time.

Access Control List. A more granular permission system than standard UNIX permissions that allows you to grant read/write/execute rights to specific users or groups beyond the owner/group/other model. A label attached to every file and process by SELinux that determines what operations are permitted. Preserving these labels is critical when syncing files that will be used in their original security context.

Ownership and Root Privileges

To preserve file ownership on the destination, you must have root privileges. For a remote destination, authenticate as root. For a local destination, run rsync with sudo.

Full archive sync: pushing /var/log to a remote host
[root@serverb ~]# rsync -av /var/log hosta:/tmp root@hosta's password: password receiving incremental file list log/ log/README log/boot.log ...output omitted... sent 9,783 bytes received 290,576 bytes 85,816.86 bytes/sec total size is 11,585,690 speedup is 38.57
Full archive sync: pulling /var/log from a remote host
[root@serverb ~]# rsync -av hosta:/var/log /tmp root@hosta's password: password receiving incremental file list log/boot.log log/dnf.librepo.log log/dnf.log ...output omitted... sent 9,783 bytes received 290,576 bytes 85,816.86 bytes/sec total size is 11,585,690 speedup is 38.57

The Trailing Slash Rule

One of the most common sources of confusion with rsync is the meaning of a trailing slash on the source directory. Getting this wrong can result in an extra layer of nested directories at the destination, or missing files entirely.

A forward slash (/) placed at the end of a directory name. In rsync, a trailing slash on the source means "sync the contents of this directory," while omitting it means "sync the directory itself."

Without trailing slash: rsync -av /var/log /tmp

The log directory itself is copied into /tmp. Result: /tmp/log/ is created, containing all log files.

Result
[root@serverb ~]# ls /tmp log ssh-RLjDdarkKiw1 [root@serverb ~]# ls /tmp/log README boot.log dnf.log ...

With trailing slash: rsync -av /var/log/ /tmp

The contents of log/ are placed directly inside /tmp. No extra log/ subdirectory is created.

Result
[root@serverb ~]# ls /tmp anaconda audit boot.log dnf.log ... # Files land directly in /tmp — no log/ subdirectory
⚠️Bash Tab Completion Adds a Trailing Slash

When you use Tab to complete a directory name in bash, it automatically appends a trailing slash. This means rsync -av /var/log/ (syncs contents) vs. rsync -av /var/log (syncs directory) is an easy mistake to make. Be intentional: decide whether you want the directory itself or just its contents at the destination, then type accordingly.

With trailing slash: contents go directly into /tmp
[root@serverb ~]# rsync -av /var/log/ /tmp sending incremental file list / README boot.log ...output omitted... tuned/tuned.log sent 11,592,389 bytes received 778 bytes 23,186,334.00 bytes/sec total size is 11,586,755 speedup is 1.00

Hands-On Lab: Transfer Files

Run the lab setup command on workstation to prepare the environment and ensure all required resources are available:

Connect to serverb and become root
student@workstation:~$ ssh student@serverb ...output omitted... [student@serverb ~]$ sudo -i [sudo] password for student: student [root@serverb ~]#

Create directory and rsync /etc from servera

Chapter Summary

In this chapter, you learned three complementary tools for transferring files between RHEL systems — all based on SSH for security, but each suited to different workflows.

Key Concepts

  • SFTP provides an interactive session for browsing and transferring files. Use put to upload and get to download. Prefix commands with l (e.g., lpwd, lls) to run them on the local system.
  • SCP is a non-interactive, one-command-line tool. In RHEL 10, scp uses SFTP internally. Avoid the -O flag and the legacy SCP protocol due to CVE-2020-15778.
  • rsync minimizes network traffic by transferring only changed data. Use -a for archive mode, -v for verbose output, and -n for a safe dry run before committing.
  • The trailing slash on an rsync source directory is critical: without it, the directory itself is synced; with it, only the contents are synced to the destination.
  • To preserve ACLs and SELinux contexts with rsync, add -A and -X respectively.
