Installation Media and Requirements

Before installing Red Hat Enterprise Linux, you must choose the correct installation media and verify your hardware meets minimum requirements. Red Hat provides different media types depending on whether you have network access during installation and how much flexibility you need.

Installation Media Types

Red Hat distributes installation media through the Customer Portal for subscribers. Three primary forms are available: The Red Hat Customer Portal (access.redhat.com) is the central hub for Red Hat subscribers to download software, access documentation, manage subscriptions, and open support cases.

Binary DVD ISO (Full Installer)

An ISO 9660 image file containing the complete Anaconda installer plus the BaseOS and AppStream package repositories. ISO 9660 is the standard file system format used for optical disc media. An ISO image is a single file that represents the entire contents of a disc, used to create bootable USB drives or DVDs. Anaconda is the installation program used by Red Hat Enterprise Linux and Fedora. It handles disk partitioning, package selection, user creation, and all configuration tasks during RHEL installation. BaseOS is the core RHEL content repository providing the underlying operating system components: the kernel, libraries, and foundational packages required to run the system. AppStream is a RHEL content repository containing user-space applications, runtime languages, and database tools. It supplements BaseOS with additional application software.

  • Best for: Offline or air-gapped environments; no network required during install.
  • Size: Several gigabytes (full repository included).
  • Flexibility: Self-contained; installs minimal to full graphical environments.

Boot ISO (Network Installer)

A smaller image containing only the Anaconda installer. Package repositories must be accessed over the network during installation from HTTP, HTTPS, FTP, or NFS servers.

  • Best for: Network-based deployments; smaller initial download.
  • Requirement: Network connectivity to a package repository during install.
  • Use case: PXE boot environments or when deploying many systems from a central server.
💡 Tip

In classroom environments, the content server at content.example.com hosts a mirrored repository, making the boot ISO practical for lab exercises without requiring internet access.

Source DVD

Contains the human-readable source code for RHEL packages. This is not used for system installation.

  • Purpose: Transparency and compliance with open-source licenses; developers can review and compile their own builds.
  • Contains: Source RPMs (.src.rpm) for all packages included in RHEL.
  • Does not include: Documentation or the Anaconda installer.

Supported Architectures

RHEL 10 supports four processor architectures:

  • x86-64-v3 — AMD and Intel 64-bit processors (requires at least Haswell-generation microarchitecture)
  • ARMv8.0-A — 64-bit ARM processors (common in cloud and edge deployments)
  • POWER9 — IBM Power Systems, Little Endian
  • z14 — 64-bit IBM Z mainframe architecture

Disk and Memory Requirements

RHEL 10 requires a minimum of 10 GiB of disk space on the partition where it will be installed. RAM requirements depend on the installation method:

Installation Type x86-64-v3, ARMv8.0-A, z14 POWER9
Local media (USB/DVD) 1.5 GiB 3 GiB
NFS network installation 1.5 GiB 3 GiB
HTTP, HTTPS, or FTP network 3 GiB (x86-64-v3, z14) / 4 GiB (ARMv8.0-A) 4 GiB

RHEL Image Builder

For specialized deployments, Red Hat provides the RHEL Image Builder tool to create customized images for cloud platforms and virtual environments. RHEL Image Builder is a tool that creates customized RHEL images pre-configured for specific use cases. It can produce images in formats like QCOW2, AMI, VMware VMDK, or Azure VHD, enabling consistent deployments to cloud and virtual platforms. Access it through the composer-cli command or the Red Hat web console.

Interactive Installation with Anaconda

The Anaconda graphical installer is launched when you boot from the binary DVD or boot ISO. It presents an Installation Summary window that serves as a central dashboard for all configuration decisions. You can configure items in any order and return to the summary at any time.

Configuration Items Explained

Each item in the Installation Summary window corresponds to a system configuration decision. Understanding each one helps you install RHEL correctly the first time.

Keyboard: Add one or more keyboard layouts. The default is usually US English (us). Multiple layouts can be added for multilingual systems.

Language Support: Select additional languages to install alongside the primary language. These affect available fonts, locale data, and input methods, but individual users can change their language preference post-installation.

Select the system's time zone from a world map or dropdown list. Even when using NTP for automatic time synchronization, you must still specify the correct local time zone. Network Time Protocol (NTP) automatically synchronizes a computer's clock with internet time servers, ensuring consistent, accurate time across networked systems.

You may leave NTP enabled (recommended) or disable it and set the date and time manually. In production environments, NTP is strongly recommended.

Register the system with your Red Hat account and select the system purpose. System purpose describes the intended role of a RHEL system (e.g., Server, Desktop, Workstation) and helps subscription manager attach the most appropriate subscription automatically. The system purpose feature enables the registration process to automatically attach the most appropriate subscription.

You must first configure networking (via Network & Host Name) before this step can succeed. By default, the system may also be enrolled in Red Hat Insights. Red Hat Insights is a proactive analytics service that continuously analyzes your RHEL systems for configuration drift, security vulnerabilities, and performance issues, providing actionable recommendations. Registration can be deferred until after installation.

Installation Source: Specifies where Anaconda retrieves packages. When using the binary DVD, this is already set to the disc. For a boot ISO, point to an HTTP, FTP, or NFS server.

Software Selection: Choose a base environment such as:

  • Minimal Install — Only essential packages; smallest footprint; ideal for servers.
  • Server — Common server tools and infrastructure packages.
  • Server with GUI — Server tools plus the GNOME desktop environment.
  • Workstation — Desktop-focused package set for developer workstations.

Additional packages and add-ons can be selected from the right pane after choosing an environment.

Select at least one disk with enough space for RHEL. Two storage configuration modes are available:

  • Automatic partitioning: Anaconda creates an optimal layout using LVM. Logical Volume Management (LVM) is a flexible disk management system that creates an abstraction layer between physical storage and file systems, enabling easy resizing, snapshots, and spanning volumes across multiple physical disks. This is recommended for most installations because Anaconda considers total disk and RAM size to size partitions optimally.
  • Custom partitioning: Manually define partition sizes, mount points, file systems, and LVM configurations. Anaconda can still auto-generate a suggested layout as a starting point.
📋 Note

The default automatic configuration allocates all available space. Even in custom mode, you can have Anaconda suggest partitions and then modify them as needed.

The kdump feature captures a memory dump when the kernel crashes. Kdump is a kernel crash dumping mechanism that captures the state of system memory when the kernel panics. Red Hat engineers can analyze the dump file to determine the root cause of crashes, making kdump essential for enterprise support. Red Hat engineers analyze these crash dump files to identify failure root causes. Enabling kdump reserves memory specifically for the dump kernel, which slightly reduces available RAM. Use this Anaconda item to enable or disable the feature.

Detected network interfaces are listed on the left panel. Select a connection to view its details. Anaconda activates network connections automatically by default.

Set the system's fully qualified domain name (FQDN) in the host name field at the bottom. A Fully Qualified Domain Name (FQDN) is a complete, unambiguous domain name that specifies the exact location in the DNS hierarchy, e.g., serverb.lab.example.com. It includes both the host name and the domain name.

Root Account (RHEL 10 Change): Starting in RHEL 10, the root account is disabled by default. To enable it, check "Enable root account" and set a password. By default, even an enabled root account does not allow password-based SSH login, though this can be permitted during installation.

User Creation: Because root is disabled by default, creating a non-root user with administrative privileges is mandatory unless you explicitly enable root. This user is added to the wheel group and can use sudo to run privileged commands.

⚠ Important

If you lock the root account during installation and do not create a wheel group user, you will be unable to administer the system after installation. Always ensure at least one administrative account exists before proceeding.

Completing the Installation

Once all mandatory items are configured (no orange warning triangles remain), click Begin Installation. Clicking Quit halts without making any disk changes.

When installation finishes, click Reboot. On the first graphical login, RHEL may present an Initial Setup screen to accept license information and optionally register with subscription manager. System registration can be skipped until later.

Remote Installation with RDP

RHEL 10 introduces support for using Remote Desktop Protocol (RDP) to perform installations on headless or remote systems. Remote Desktop Protocol (RDP) is a proprietary protocol developed by Microsoft that provides a user with a graphical interface to connect to another computer over a network connection. In RHEL 10, Anaconda uses RDP to enable remote graphical installation sessions. This replaces the deprecated VNC method used in earlier RHEL versions. Virtual Network Computing (VNC) is a graphical desktop-sharing system previously used to perform remote RHEL installations. It is deprecated as of RHEL 10 in favor of RDP, which offers better performance.

Enabling RDP Mode

To start an installation in RDP mode, pass the inst.rdp option on the kernel command line in the boot loader stage:

Boot Loader Kernel Arguments — RDP Mode
# At the GRUB boot menu, press 'e' to edit the selected entry # Add these arguments to the linux line: linux /images/pxeboot/vmlinuz inst.stage2=hd:LABEL=RHEL-10 quiet \ inst.rdp inst.rdp.username=rdpuser inst.rdp.password=Redhat123 # Press Ctrl+X to boot with these arguments

If no authentication details are provided via inst.rdp.username and inst.rdp.password, Anaconda will interactively prompt for credentials before starting the RDP service.

📋 Note

Once the installer starts in RDP mode, it displays the IP address the RDP client should connect to. The default TCP port is 5900. Multiple RDP clients can connect simultaneously, enabling collaborative installation sessions.

RDP Client Connection

Use any standard RDP client application (such as xfreerdp on Linux, Microsoft Remote Desktop on Windows/macOS, or Remmina) to connect to the installer:

Connecting an RDP Client to Anaconda
# Connect using xfreerdp (replace IP with actual installer address) [student@workstation ~]$ xfreerdp /v:192.168.1.50:5900 /u:rdpuser /p:Redhat123 Connected to 192.168.1.50:5900 Anaconda graphical interface is now available.

Troubleshooting the Installation

Anaconda provides access to multiple virtual consoles and a tmux terminal multiplexer during installation. tmux is a terminal multiplexer that allows multiple terminal sessions within a single window. It supports splitting panes, detaching/reattaching sessions, and running commands in parallel — essential for managing multiple views during RHEL installation troubleshooting. These tools allow you to examine log files, run diagnostic commands, and monitor installation progress in real time.

Virtual Console Access

Two primary virtual consoles are available during installation. The first (accessed via Ctrl+Alt+F1) runs tmux with six windows of diagnostic information. The second (accessed via Ctrl+Alt+F6, the default view) shows the Anaconda graphical interface.

Tmux Window Reference

Inside the first virtual console's tmux session, press Ctrl+B followed by a number key to switch windows:

Key Sequence Content / Purpose
Ctrl+Alt+F1 Switch to the tmux virtual console (all tmux windows)
Ctrl+Alt+F6 Return to the Anaconda graphical interface
Ctrl+B then 1 Main installation information page
Ctrl+B then 2 Root shell prompt — logs are in /tmp/
Ctrl+B then 3 Live view of /tmp/anaconda.log
Ctrl+B then 4 Live view of /tmp/storage.log
Ctrl+B then 5 Live view of /tmp/program.log
Ctrl+B then 6 Live view of /tmp/packaging.log
💡 Compatibility Note

For compatibility with earlier RHEL versions, Ctrl+Alt+F2 through Ctrl+Alt+F5 also provide root shell prompts during installation. You can also press Alt+Tab inside tmux to rotate focus between windows.

Common Troubleshooting Scenarios

The shell in tmux window 2 is your primary debugging tool. Use it to:

  • Examine /tmp/anaconda.log for errors in the installation sequence.
  • Check /tmp/storage.log when partitioning or LVM configuration fails.
  • Review /tmp/packaging.log for package dependency or repository errors.
  • Run ip addr show to verify network configuration during a network installation.
  • Use ping to test connectivity to installation servers.

Introduction to Kickstart

The Kickstart feature automates RHEL installation by providing answers to all installer questions in a single text file. Kickstart is a Red Hat technology that enables unattended, automated RHEL installations. A Kickstart file contains all installation parameters (disk layout, packages, network config, users) that Anaconda would otherwise prompt for interactively. Instead of clicking through the graphical installer, Anaconda reads the Kickstart file and performs the entire installation without user intervention. This is conceptually similar to Microsoft Windows answer files for unattended installations.

Kickstart is essential in enterprise environments where dozens or hundreds of systems need to be deployed consistently and quickly. A single well-crafted Kickstart file ensures every deployed system has identical configuration.

Kickstart File Structure

A Kickstart file has three types of content:

Kickstart File Structure
Command Section
Installation commands (lang, keyboard, timezone, network, storage, users…). Lines starting with # are comments.
▼
%packages … %end
Software selection: individual packages, groups (@group-name), and environments (@^env-name). Prefix with - to exclude.
▼
%pre … %end
Runs BEFORE disk partitioning. Limited tools available (outside chroot).
%post … %end
Runs AFTER installation. Full access to newly installed system (chroot environment).

Pre- vs. Post-Installation Scripts

The timing of script execution matters significantly:

%pre scripts execute before any disk partitioning occurs. They run outside the chroot environment and have limited tool availability. A chroot (change root) environment changes the apparent root directory for a process, confining it to a subtree of the filesystem. During Kickstart, %post scripts run inside the chroot of the newly installed system, giving them full access to installed packages. Use %pre for initializing storage devices or network interfaces needed by the rest of the installation.

%post scripts execute after installation is complete. They run inside the chroot of the installed system and have access to all installed packages and tools. Common uses include configuring applications, downloading additional content, or adding custom files to /etc.

⚠ Caution

Copying files from the installation media does not work inside the %post chroot environment. Tasks requiring access to the installation media (such as copying scripts from a USB drive) must be done in a %pre script instead.

You can include multiple sections of the same type, and they execute in order of appearance in the file. For example, two %post sections are both executed sequentially.

A Minimal Kickstart Example

Minimal Kickstart File Structure
# version=RHEL10 # ── Command Section ────────────────────────────────── graphical # graphical installation keyboard --vckeymap=us --xlayouts='us' lang en_US.UTF-8 timezone America/New_York --utc network --bootproto=dhcp --device=enp1s0 --activate network --hostname=serverb.lab.example.com url --url="http://content.example.com/rhel10.0/x86_64/dvd/" zerombr clearpart --all autopart ignoredisk --only-use=vda rootpw --lock user --groups=wheel --name=student --gecos="student" reboot # ── Packages Section ────────────────────────────────── %packages @minimal-environment vim-enhanced %end # ── Post Section ────────────────────────────────────── %post echo "Deployed via Kickstart on $(date)" > /etc/motd %end

Kickstart Command Reference

Kickstart commands are organized by function. Some commands are required for installation to succeed; others are optional configuration. Understanding the most common commands allows you to write Kickstart files from scratch or modify existing ones confidently.

Installation Source Commands

url, repo, text
# url: installation source (HTTP or FTP) url --url="http://classroom.example.com/rhel10.0/x86_64/dvd/" # repo: additional package repository (must be a valid DNF repo) repo --name="appstream" --baseurl=http://classroom.example.com/rhel10.0/x86_64/dvd/AppStream/ # text: force text-mode installation (no graphical UI) text

Package Selection

%packages section
%packages # Install an environment group (note: @^ prefix) @^graphical-server-environment # Install a package group (note: @ prefix) @development-tools # Individual package (wildcards supported) vim* # Exclude a package -hyperv-tools %end

Storage and Partitioning Commands

clearpart, part, autopart, volgroup, logvol, bootloader
# zerombr: initialize disks with unrecognized formatting zerombr # clearpart: remove existing partitions before install clearpart --all --drives=vda,vdb # autopart: auto-create root, swap, and boot partitions (recommended) autopart # ignoredisk: prevent Anaconda from touching specified disks ignoredisk --only-use=vda # part: manually define a partition part /home --fstype=xfs --label=homes --size=4096 --maxsize=8192 --grow # bootloader: define boot loader placement bootloader --location=mbr --boot-drive=vda # LVM example: create VG and LVs part pv.01 --size=8192 volgroup myvg pv.01 logvol / --vgname=myvg --fstype=xfs --size=4096 --name=rootvol --grow logvol /var --vgname=myvg --fstype=xfs --size=2048 --name=varvol

Network Commands

network, firewall
# network: configure a network device network --device=enp1s0 --bootproto=dhcp --activate # Set the system hostname network --hostname=serverb.lab.example.com # Static IP configuration network --device=enp1s0 --bootproto=static \ --ip=172.25.250.10 --netmask=255.255.255.0 \ --gateway=172.25.250.254 --nameserver=172.25.250.254 # firewall: configure firewall for installed system firewall --enabled --service=ssh,http

Locale, Time, and Security Commands

lang, keyboard, timezone, rootpw, selinux, user, group
# Required: system language lang en_US.UTF-8 # Required: keyboard layout keyboard --vckeymap=us --xlayouts='us' # Required: time zone (use --utc for hardware clock in UTC) timezone America/New_York --utc # Optional: NTP source timesource --ntp-server classroom.example.com # rootpw: set root password (RHEL 10: root disabled by default) rootpw --lock # lock root (recommended) rootpw --plaintext redhat # plaintext (avoid in prod) rootpw --iscrypted $6$hash... # hashed (recommended) # selinux: set SELinux mode selinux --enforcing # services: adjust default services services --disabled=rsyslog --enabled=NetworkManager,firewalld # Create a user (wheel group for admin; REQUIRED if rootpw --lock) user --name=jdoe --gecos="John Doe" --groups=wheel --iscrypted \ --password=$6$hash... # Create a group group --name=admins --gid=10001
⚠ RHEL 10 Requirement

In RHEL 10, if you use rootpw --lock, you must create at least one user that belongs to the wheel administrative group. Without this, no one can administer the system after installation.

Miscellaneous Commands

logging, firstboot, reboot/poweroff/halt, %addon
# logging: send installation logs to a remote host logging --host=loghost.example.com # firstboot: control GNOME Initial Setup on first boot firstboot --disabled # Completion action (default: halt, waits for keypress) reboot # reboot automatically when done poweroff # power off when done halt # halt and wait (default) # %addon: configure optional components like kdump %addon com_redhat_kdump --enable --reserve-mb='auto' %end

Step 1: Create the Kickstart File

Three approaches are available:

  • Kickstart Generator website — https://access.redhat.com/labs/kickstartconfig presents dialog boxes and generates a Kickstart file based on your choices. Each dialog corresponds to an Anaconda installer screen.
  • Copy and modify an existing file — Every RHEL installation creates /root/anaconda-ks.cfg that captures the exact settings used during that installation. This is the recommended starting point for creating new Kickstart files.
  • Write from scratch with a text editor — Suitable once you understand the command syntax well.
Locating the Auto-Generated Kickstart File
# Every installation creates this file on the installed system [root@serverb ~]# ls -la /root/anaconda-ks.cfg -rw-------. 1 root root 1842 Mar 15 14:30 /root/anaconda-ks.cfg [root@serverb ~]# cat /root/anaconda-ks.cfg # Generated by Anaconda 34.25.0.58 # version=RHEL10 graphical ...output omitted...

Validating Your Kickstart File

Before deploying, validate the Kickstart file with ksvalidator, provided by the pykickstart package. pykickstart is a Python library and set of utilities for parsing, validating, and generating Kickstart files. It includes ksvalidator (syntax check) and ksverdiff (version comparison).

Validating and Diffing Kickstart Files
# Install pykickstart (provides ksvalidator and ksverdiff) [student@workstation ~]$ sudo dnf install pykickstart -y # Validate syntax (checks keywords and options, not URLs or packages) [student@workstation ~]$ ksvalidator /tmp/myserver.cfg The following problem occurred on line 10 of the kickstart file: Unknown command: frewall # Show Kickstart syntax changes between versions [student@workstation ~]$ ksverdiff -f RHEL9 -t RHEL10 The following commands were removed in RHEL10: auth authconfig autostep btrfs method nvdimm The following commands were deprecated in RHEL10: vnc The following options were added to the autopart command in RHEL10: --hibernation ...output omitted...
📋 Validation Scope

ksvalidator checks keyword spelling and option syntax. It does not validate URL paths, package names, group names, or the logic inside %pre/%post scripts. Always test Kickstart files in a VM before production use.

Step 2: Publish the Kickstart File

The file must be accessible to the target system during installation. Common publication methods:

  • HTTP/HTTPS server — Most common in enterprise. Copy to /var/www/html/ on a web server.
  • FTP server — Alternative network protocol, useful in some environments.
  • NFS share — Network file system export accessible at install time.
  • USB drive — Convenient for single-system rebuilds; write the .cfg file to a FAT32 USB drive.
  • Local hard disk — For quick rebuilds on the same machine; place the file on a non-target partition.
Publishing via HTTP (Apache Example)
# Copy Kickstart file to web server document root [root@classroom ~]# cp /root/serverb.cfg /var/www/html/ks/ [root@classroom ~]# chmod 644 /var/www/html/ks/serverb.cfg [root@classroom ~]# systemctl enable --now httpd # Verify file is accessible from a client [student@workstation ~]$ curl http://classroom.example.com/ks/serverb.cfg # version=RHEL10 graphical ...output omitted...

Step 3: Boot and Pass inst.ks=

When booting the target system from installation media, edit the boot loader entry to pass the inst.ks=LOCATION kernel argument:

Location Type inst.ks= Parameter Syntax
HTTP server inst.ks=http://server/dir/file.cfg
FTP server inst.ks=ftp://server/dir/file.cfg
NFS share inst.ks=nfs:server:/dir/file.cfg
Local disk partition inst.ks=hd:device:/dir/file.cfg
CD/DVD/USB media inst.ks=cdrom:device
Adding inst.ks= to the GRUB Boot Entry
# At the GRUB menu, highlight "Install Red Hat Enterprise Linux 10" # Press 'e' to edit. Find the line beginning with 'linux' # Append inst.ks= at the end of that line: linux /images/pxeboot/vmlinuz inst.stage2=hd:LABEL=RHEL-10 quiet \ inst.ks=http://classroom.example.com/ks/serverb.cfg # Press Ctrl+X to boot

If the Kickstart file contains all required commands, Anaconda performs the entire installation without prompting. You can still use the virtual console shortcuts (Ctrl+Alt+F1 through Ctrl+Alt+F6) to monitor progress or troubleshoot.

Hands-On Lab: Automate RHEL Installation with Kickstart

Scenario: You are a system administrator at Example Corp. Your team frequently deploys new RHEL 10 servers for application development. You have been asked to create a Kickstart file that standardizes server deployment, validate it, and publish it on the classroom HTTP server so it can be used to provision serverb.lab.example.com.

⚠ Warning

Kickstart deployments erase all data on the target disks. In lab environments, always verify you are targeting the correct virtual machine. Never run a Kickstart installation against a production system without a verified backup.

Objective: Install the pykickstart package and review the auto-generated Kickstart file from the workstation's own installation as a starting reference.

Task 1: Install Tools and Review Existing KS File
# Step 1: Install pykickstart (provides ksvalidator and ksverdiff) [student@workstation ~]$ sudo dnf install pykickstart -y ...output omitted... Installed: pykickstart-3.50-1.el10.noarch Complete! # Step 2: View the existing anaconda-ks.cfg from this system's install [student@workstation ~]$ sudo cat /root/anaconda-ks.cfg # Generated by Anaconda 34.25.0.58 # version=RHEL10 graphical ...output omitted... user --name=student --groups=wheel ... # Step 3: Check RHEL 9 to RHEL 10 command differences [student@workstation ~]$ ksverdiff -f RHEL9 -t RHEL10 The following commands were removed in RHEL10: auth authconfig autostep btrfs method nvdimm The following commands were deprecated in RHEL10: vnc

Objective: Create a Kickstart file for deploying serverb.lab.example.com with a locked root account, a wheel group user, and a minimal software environment.

Task 2: Create /root/serverb.cfg
[student@workstation ~]$ sudo vim /root/serverb.cfg # --- Begin file content --- # version=RHEL10 graphical keyboard --vckeymap=us --xlayouts='us' lang en_US.UTF-8 timezone America/New_York --utc network --device=enp1s0 --bootproto=dhcp --activate network --hostname=serverb.lab.example.com url --url="http://content.example.com/rhel10.0/x86_64/dvd/" zerombr clearpart --all --drives=vda autopart ignoredisk --only-use=vda rootpw --lock selinux --enforcing firewall --enabled --service=ssh firstboot --disabled reboot user --name=student --groups=wheel --gecos="student" \ --password=$6$abc123... --iscrypted %packages @^minimal-environment vim-enhanced %end %addon com_redhat_kdump --enable --reserve-mb='auto' %end %post echo "Deployed via Kickstart: $(date)" > /etc/motd %end # --- End file content --- save and exit vim with :wq

Objective: Use ksvalidator to confirm there are no syntax errors in your new Kickstart file. Intentionally introduce an error, observe the output, then fix it.

Chapter Summary

This chapter covered two methods of installing Red Hat Enterprise Linux 10: interactive installation through the Anaconda graphical installer, and fully automated deployment via Kickstart files.

Key Concepts

  • RHEL 10 is distributed as a binary DVD ISO (self-contained) or a boot ISO (requires network access during install). Choose based on connectivity and storage constraints.
  • Minimum requirements: 10 GiB disk space; 1.5–4 GiB RAM depending on architecture and install method.
  • Anaconda's Installation Summary window provides a centralized dashboard for all pre-installation configuration. Mandatory items are marked with orange triangle warnings.
  • Starting in RHEL 10, the root account is disabled by default. Creating a wheel group user is mandatory when root remains locked.
  • RDP mode (inst.rdp) replaces the deprecated VNC mode for remote installation sessions.
  • Virtual consoles and tmux windows provide log files and a root shell for troubleshooting active installations.
  • Kickstart files automate unattended installations. They contain a command section, optional %packages, and optional %pre/%post scripts.
  • The %pre section runs before partitioning (outside chroot); %post runs after installation (inside chroot).
  • ksvalidator checks Kickstart syntax; ksverdiff shows changes between RHEL versions.
  • Pass inst.ks=LOCATION as a kernel argument at the GRUB boot menu to start a Kickstart installation.

Quick Command Reference

ksvalidator /path/file.cfg

Validate Kickstart file syntax for keyword and option errors.

ksverdiff -f RHEL9 -t RHEL10

Show Kickstart command differences between RHEL versions.

cat /root/anaconda-ks.cfg

View the auto-generated Kickstart file from this system's installation.

inst.ks=http://server/ks.cfg

Kernel argument at GRUB to specify Kickstart file location.

inst.rdp inst.rdp.username=user

Enable RDP mode for remote graphical installation (RHEL 10).

Ctrl+Alt+F1 → Ctrl+B 2

Access a root shell in tmux window 2 during installation.
