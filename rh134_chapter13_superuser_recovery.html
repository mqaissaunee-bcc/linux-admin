2

Overview: When Superuser Access Is Lost

Every system administrator will encounter a scenario at some point where access to the root account is unavailable. This might happen because the password was forgotten, accidentally changed, or the account was locked during security hardening. Regardless of the cause, a locked-out root account on a production system is a critical situation that requires immediate resolution. The root account is the superuser account on Linux systems with UID 0. It has unrestricted access to all system resources, files, and commands.

On RHEL 10, the root password is stored as a salted hash in the /etc/shadow file. Without knowing the plaintext password, there is no way to reverse the hash — however, there are legitimate methods to replace the root password by booting the system into a special recovery environment where you have access to the file system without normal authentication.

Recovery Methods at a Glance

Red Hat Enterprise Linux 10 supports two approaches to password recovery. Which one you use depends on whether you have installation media available:

💿
With Rescue Media (Recommended)
  • Uses a RHEL 10 Boot ISO
  • Boots a minimal RHEL environment
  • Mounts system under /mnt/sysroot
  • Safer and more reliable
  • Works even if GRUB is damaged
  • Requires physical or virtual media access
⚠️
Without Rescue Media (Higher Risk)
  • Modifies the GRUB2 boot entry
  • Passes init=/bin/bash to kernel
  • Boots to a minimal bash shell
  • No media required
  • Requires manual remount of root filesystem
  • Must be performed carefully — risk of data corruption
💡Tip

Both methods require physical or console access to the system. Remote recovery over SSH is not possible when the root password is unknown. This is by design — it is a security feature that prevents remote attackers from resetting your root password.

Security Implications

The ability to reset a root password by booting into rescue mode or modifying kernel arguments is a fundamental characteristic of Linux systems that have physical access available. This is why physical security of servers is so critical. Anyone with physical access to a server running RHEL (or most Linux distributions) can reset the root password unless additional protections are in place:

  • GRUB2 password protection — Requiring a password to edit GRUB2 entries prevents unauthorized modification of kernel arguments. GRUB2 can be configured to require a username and password before allowing users to edit boot entries or access the GRUB2 command line, adding a layer of security against local attacks.
  • Full disk encryption (LUKS) — Encrypting the disk prevents access to the file system without the decryption key.
  • BIOS/UEFI passwords — Preventing changes to boot order stops unauthorized booting from external media.

For this chapter, we focus on the recovery procedures themselves — skills you will need for the RHCSA exam and for real-world system administration.

3

Resetting the Root Password with Rescue Media

The recommended approach for resetting a lost root password is to use rescue media. This method boots the system from an external RHEL 10 installation image rather than from the hard drive, giving you access to a special recovery environment. Rescue media is an installation ISO image (such as the RHEL 10 Boot ISO) used to boot a system into a minimal recovery environment when the installed system cannot boot normally or when administrative access is unavailable.

What You Need

You need the Red Hat Enterprise Linux 10.0 Boot ISO image. This is the same image used for network installations and is available from the Red Hat Customer Portal. For the rescue operation to work correctly, the ISO version must match the version of RHEL installed on the system (do not use a RHEL 9 ISO to rescue a RHEL 10 system).

  • Physical machine: Boot from a CD-ROM or USB flash drive containing the ISO
  • Virtual machine: Attach the ISO file as a virtual CD-ROM/DVD device

Step-by-Step: Rescue Media Method

Follow these steps carefully in sequence:

1

Reboot & Select Boot Media

Reboot the system and configure it to boot from the rescue ISO. During boot, select the ISO image as the boot device. The exact method varies by hardware/hypervisor.

↓
2

Navigate to Troubleshooting in GRUB2

From the GRUB2 boot menu, use arrow keys to select Troubleshooting → Rescue a Red Hat Enterprise Linux system.

↓
3

Select "Continue" (Option 1)

In the rescue environment menu, press 1 and Enter to mount the system normally in read/write mode under /mnt/sysroot.

↓
4

Change Root Directory (chroot)

Run chroot /mnt/sysroot to make the installed system's root directory the active root. This makes all installed commands and configuration files accessible.

↓
5

Reset the Root Password

Run passwd root and enter the new password twice when prompted. The password hash will be updated in /etc/shadow.

↓
6

Trigger SELinux Relabeling

Run touch /.autorelabel to create a flag file that forces SELinux to relabel all files on the next boot, including the modified /etc/shadow.

↓
7

Exit Twice to Reboot

Type exit once to leave the chroot environment, then type exit again to exit rescue mode. The system reboots, performs SELinux relabeling, and reboots once more.

GRUB2 Menu Navigation

When the system boots from the ISO, you will see the GRUB2 boot menu. Use the arrow keys to navigate and Enter to select. GRUB2 (GRand Unified Bootloader version 2) is the default boot loader for RHEL 10. It presents a menu of bootable entries and passes kernel parameters to the Linux kernel at boot time.

🔴Important

At the rescue environment menu, you will see four options: (1) Continue, (2) Read-only mount, (3) Skip to shell, (4) Quit/Reboot. You must choose option 1 (Continue) so the file system is mounted read/write. Option 2 mounts read-only, which will prevent password changes.

Command Sequence

Rescue Environment — Complete Command Sequence
# Step 4: Change root directory to the installed system bash-5.2# chroot /mnt/sysroot # Step 5: Set a new root password bash-5.2# passwd root New password: your_new_password Retype new password: your_new_password passwd: password updated successfully # Step 6: Create autorelabel flag for SELinux bash-5.2# touch /.autorelabel # Step 7a: Exit the chroot environment bash-5.2# exit exit # Step 7b: Exit rescue mode (triggers reboot) bash-5.2# exit # System reboots, performs full SELinux relabeling, then reboots again

Understanding the chroot Command

The chroot command is critical to the rescue media process. Without it, running passwd would modify the password for the rescue environment's temporary system — not your installed system's /etc/shadow. chroot (change root) changes the apparent root directory for the current process and its children. Commands run after chroot see the specified directory as /, making them operate on the installed system rather than the rescue environment.

When the rescue environment mounts your installed system under /mnt/sysroot, running chroot /mnt/sysroot makes that directory appear as / to all subsequent commands. So when you run passwd root, it reads and writes /etc/shadow — which in the chrooted context is actually /mnt/sysroot/etc/shadow on the real file system.

💡Tip

After running chroot /mnt/sysroot, your shell prompt changes from bash-5.2# to a standard root prompt, confirming you are now operating inside the installed system's context.

4

Resetting the Root Password Without Rescue Media

When rescue media is not available, you can reset the root password by intercepting the normal boot process and passing a special argument to the kernel through GRUB2's boot editor. This method instructs the kernel to run /bin/bash as the first process instead of systemd, dropping you into a root shell before any authentication. GRUB2 allows you to edit boot entries before booting by pressing 'e' at the boot menu. Changes made here are temporary and apply only to the current boot.

⚠️Warning

This method is riskier than using rescue media. If the system loses power or is interrupted during this process — particularly before you properly remount the filesystem read-only before rebooting — you risk file system corruption. Proceed carefully and do not skip any steps.

How This Method Works

Normally, the Linux kernel launches init (which on RHEL 10 is systemd) as the first process (PID 1). By passing init=/bin/bash as a kernel parameter, you replace systemd with bash, resulting in a root shell with no password authentication. init is the first process started by the kernel after boot (PID 1). On RHEL 10, systemd serves as init, managing all services and system state. The init= kernel parameter can override which program is launched as PID 1.

However, at this stage the root file system (/) is mounted read-only, which prevents any changes from being saved. Before you can reset the password, you must remount it as read-write.

Step-by-Step: No-Media Method

Editing the GRUB2 Boot Entry

  1. Reboot the system.
  2. When the GRUB2 countdown begins, press Esc to interrupt it and display the menu.
  3. Use the arrow keys to highlight the kernel entry you want to boot.
  4. Press e to edit the selected entry.
  5. Use the arrow keys to find the line beginning with linux — this is the kernel command line.
  6. Remove any console= parameters from the line (these redirect output to serial console and can interfere).
  7. Press Ctrl+E (or End) to move to the end of the linux line.
  8. Add a space, then type: init=/bin/bash
  9. Press Ctrl+X to boot with these temporary changes.
GRUB2 Kernel Line — Before and After
# BEFORE: Original kernel line (simplified) linux /vmlinuz-5.14.0-503.el10.x86_64 root=/dev/mapper/rhel-root ro crashkernel=1G-4G:192M,4G-64G:256M,64G-:512M resume=/dev/mapper/rhel-swap rd.lvm.lv=rhel/root rd.lvm.lv=rhel/swap console=tty0 console=ttyS0,115200n8 # AFTER: Remove console= args, add init=/bin/bash at end linux /vmlinuz-5.14.0-503.el10.x86_64 root=/dev/mapper/rhel-root ro crashkernel=1G-4G:192M,4G-64G:256M,64G-:512M resume=/dev/mapper/rhel-swap rd.lvm.lv=rhel/root rd.lvm.lv=rhel/swap init=/bin/bash

Resetting the Password in the Bash Shell

After booting with the modified kernel line, you will land at a minimal bash shell. The root file system is mounted read-only at this point.

1

Wait for boot to complete

The system will pause briefly at the bash prompt. You are now root with no password required.

2

Remount root filesystem read/write

The root filesystem is currently read-only. Remount it to allow changes:

Remount /
bash-5.2# mount -o remount,rw /
3

Set a new root password

Reset Root Password
bash-5.2# passwd New password: your_new_password Retype new password: your_new_password passwd: password updated successfully
4

Create the SELinux autorelabel file

Flag SELinux Relabeling
bash-5.2# touch /.autorelabel

Rebooting After the No-Media Method

Unlike the rescue media method (where you type exit twice), the no-media method requires a specific command to reboot because there is no init system running:

Reboot the System
bash-5.2# exec /sbin/init # This launches systemd as PID 1 and continues the normal boot process # The system performs SELinux relabeling and then reboots again automatically
⚠️Warning

Do not press the power button or use reboot directly after setting the password. Using exec /sbin/init allows a clean handoff to systemd, which then handles the SELinux relabeling and shutdown properly. A hard power-off at this point risks file system corruption.

Comparing Both Methods

Aspect With Rescue Media Without Rescue Media
Media required RHEL 10 Boot ISO None
Boot environment External rescue environment Minimal bash shell (no systemd)
Root directory /mnt/sysroot (requires chroot) / (direct, after remount)
Filesystem mount Automatically mounted r/w by rescue tool Must manually remount: mount -o remount,rw /
Reboot command exit twice exec /sbin/init

5

SELinux Relabeling After Password Recovery

One of the most critical steps in both recovery methods — and one that is frequently overlooked — is triggering an SELinux relabeling of the file system. Skipping this step can prevent the system from booting normally or cause authentication to fail even after the password is correctly changed. SELinux relabeling is the process of scanning all files on the system and assigning correct SELinux security context labels. It is required when files are created or modified outside the normal SELinux-protected environment.

Why Relabeling Is Required

Both recovery methods operate in environments where SELinux is not fully active. When SELinux is not running, any file that is created or modified does not receive the proper SELinux security context label. SELinux (Security-Enhanced Linux) is a Linux kernel security module that provides mandatory access control. It labels every file and process with a security context and enforces access rules based on those contexts.

The passwd command works by creating a new temporary file, writing the updated password hash into it, and then replacing the original file with the new one. This replacement creates a new file — and in a non-SELinux environment, that new file has no SELinux context label. When the system boots normally and SELinux is enforcing, it will deny access to /etc/shadow because it lacks the expected shadow_t context. This means login will fail even though the password was correctly set.

The .autorelabel Mechanism

The /.autorelabel file is a simple flag recognized by the SELinux subsystem during early boot. The /.autorelabel file is an empty flag file in the root directory. When SELinux detects this file at boot, it performs a complete file system relabeling (applying correct SELinux contexts to all files) before starting normal services. The file is deleted after relabeling completes.

Creating the SELinux Autorelabel Flag
# Create the flag file that triggers full SELinux relabeling on next boot bash-5.2# touch /.autorelabel # Verify the file was created bash-5.2# ls -la /.autorelabel -rw-r--r--. 1 root root 0 Mar 15 10:32 /.autorelabel

What Happens During Relabeling

After you reboot following password recovery, here is what happens:

  1. The system begins its normal boot sequence.
  2. SELinux detects the /.autorelabel file in the root directory.
  3. SELinux pauses the normal boot process and begins scanning every file on every mounted filesystem.
  4. Each file is assigned its correct SELinux context based on the system's policy rules.
  5. The /.autorelabel file is deleted.
  6. The system reboots automatically one more time.
  7. On this second reboot, the system starts normally with all files correctly labeled.
🔴Important

On systems with large filesystems (many terabytes of data), the SELinux relabeling process can take a significant amount of time — potentially many minutes or even hours. This is expected behavior. Do not power off the system during relabeling, as this can leave the filesystem in an inconsistent state.

Verifying SELinux Context After Recovery

Once the system has fully rebooted after relabeling, you can verify that /etc/shadow has the correct SELinux context:

Verifying /etc/shadow SELinux Context
# Check the SELinux context of the shadow file (should show shadow_t) [root@server ~]# ls -Z /etc/shadow system_u:object_r:shadow_t:s0 /etc/shadow # If the context was missing (before relabeling), it would appear as: # system_u:object_r:unlabeled_t:s0 /etc/shadow # This would cause authentication failures even with the correct password
📝Note

The -Z flag on ls displays the SELinux security context of files. The context format is user:role:type:sensitivity. For /etc/shadow, the expected type is shadow_t, which grants access only to privileged authentication processes.

Hands-On Lab: Recovering the Root Password

These procedures require console or physical access to the server. They will cause a system reboot. Ensure no critical services are running or that any outage has been properly coordinated before proceeding. The SELinux relabeling step will cause an additional automatic reboot — this is expected.

Objective: Use the RHEL 10 Boot ISO in rescue mode to reset the root password on serverb.lab.example.com.

Prerequisites: Attach the RHEL 10 Boot ISO to the virtual machine as a CD-ROM/DVD device and configure the VM to boot from it first.

  1. Reboot serverb and boot from the RHEL 10 Boot ISO.
  2. At the GRUB2 menu, use arrow keys to select Troubleshooting.
  3. Select Rescue a Red Hat Enterprise Linux system.
  4. At the rescue menu, press 1 then Enter to select Continue (mounts the system r/w at /mnt/sysroot).
  5. At the bash prompt, change root to the installed system:
Task 1 — Rescue Media Password Reset
# Change root directory to the installed system bash-5.2# chroot /mnt/sysroot # Set the new root password (use: redhat123 for lab purposes) bash-5.2# passwd root New password: Retype new password: passwd: password updated successfully # Create the SELinux autorelabel flag bash-5.2# touch /.autorelabel # Verify the autorelabel file exists bash-5.2# ls -la /.autorelabel -rw-r--r--. 1 root root 0 Mar 15 10:32 /.autorelabel # Exit chroot, then exit rescue mode to reboot bash-5.2# exit exit bash-5.2# exit # System reboots, relabels, reboots again...

Expected result: The system boots normally after the two-reboot SELinux relabeling cycle. You can now log in as root using the new password.

Objective: Reset the root password on serverb.lab.example.com by modifying the GRUB2 boot entry — without using any external media.

  1. Reboot serverb (remove the ISO if it is still attached).
  2. When the GRUB2 countdown appears, press Esc to stop it.
  3. Highlight the default kernel entry and press e to edit.
  4. Locate the line beginning with linux.
  5. Remove any console=ttyS0... or similar console= parameters.
  6. Press Ctrl+E to go to the end of the linux line.
  7. Add: init=/bin/bash (with a space before init)
  8. Press Ctrl+X to boot.

Objective: Understand common failure scenarios and how to diagnose them.

Scenario A: Login still fails after recovery (SELinux context issue)

Fix Missing SELinux Context Manually
# If /.autorelabel was forgotten, shadow file may have wrong context [root@serverb ~]# ls -Z /etc/shadow system_u:object_r:unlabeled_t:s0 /etc/shadow # Fix it manually with restorecon [root@serverb ~]# restorecon -v /etc/shadow Relabeled /etc/shadow from system_u:object_r:unlabeled_t:s0 to system_u:object_r:shadow_t:s0 # Or trigger a full relabeling via autorelabel (requires reboot) [root@serverb ~]# touch /.autorelabel && reboot

Scenario B: No-media method — root filesystem remount failed

Check for Filesystem Errors Before Remounting
# If remount fails, check dmesg for errors bash-5.2# dmesg | tail -20 # Look for filesystem errors or device issues # Try remount with explicit device bash-5.2# mount -o remount,rw /dev/mapper/rhel-root /

Chapter Summary

In this chapter, you learned how to recover administrative access to a RHEL 10 system when the root password is unknown or the account is locked. You practiced both recovery methods and understood the critical role of SELinux relabeling in the recovery process.

Key Concepts

  • Root password recovery requires physical or console access — it cannot be done remotely via SSH when locked out.
  • The rescue media method (using a RHEL 10 Boot ISO) is the recommended approach: boot the ISO, choose Troubleshooting → Rescue, select option 1, then chroot /mnt/sysroot before running passwd root.
  • The no-media method uses GRUB2 to pass init=/bin/bash to the kernel, bypassing systemd entirely and dropping directly into a root shell. The root filesystem must then be manually remounted read/write before changing the password.
  • SELinux relabeling is always required: the passwd command creates files without SELinux context labels when SELinux is inactive. Without touch /.autorelabel, the new /etc/shadow will have an incorrect context and logins will fail.
  • After the no-media method, use exec /sbin/init (not reboot) to hand off to systemd for a clean shutdown and relabeling cycle.

Command Reference

Command / Action Purpose Method
chroot /mnt/sysroot Change root to the installed system's filesystem within the rescue environment Rescue media
passwd root Set a new password for the root account Rescue media (after chroot)
passwd Set a new password for the current user (root in no-media shell) No-media
touch /.autorelabel Create a flag file that triggers full SELinux relabeling on next boot Both methods
exit (twice) Exit chroot, then exit rescue environment to trigger reboot Rescue media
mount -o remount,rw / Remount root filesystem as read/write (required before passwd in no-media method) No-media
exec /sbin/init Launch systemd as PID 1 for a clean boot/relabeling cycle No-media
ls -Z /etc/shadow Verify the SELinux context of /etc/shadow (should be shadow_t) Post-recovery verification
restorecon -v /etc/shadow Manually restore the correct SELinux context to /etc/shadow Troubleshooting
getenforce Verify SELinux is in Enforcing mode after recovery Post-recovery verification

RHCSA Exam Tips

🎯RHCSA Exam Focus

Resetting the root password is a commonly tested RHCSA skill. Key points to remember for the exam: (1) You must know both methods. (2) Never forget touch /.autorelabel — forgetting this step is the single most common reason the recovery appears to succeed but login still fails. (3) In the rescue media method, remember to chroot before running passwd. (4) In the no-media method, remember to remount the filesystem before running passwd.
