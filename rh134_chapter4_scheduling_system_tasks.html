Managing Repeating Jobs with Systemd Timer Units

Starting from Red Hat Enterprise Linux 10, the systemdThe system and service manager for Linux, responsible for initializing the system, managing services, and providing dependency tracking between units. daemon and service manager replaces the traditional cron daemon for most scheduled system tasks via the systemd timer units.

A systemd timer unit is defined by a unit configuration file whose name ends in .timer. Timer units activate a unit of another type — usually a service unitA systemd unit type that describes how to start, stop, and manage a background process or application.. By default, the timer unit activates the service with the matching base name.

For example, logrotate.timer triggers logrotate.service. The timer unit describes when to run; the service unit describes what to run.

An important advantage of systemd timer units over cron is that timer events are logged in the system journalThe systemd logging subsystem (journald) that captures structured log data from the kernel, services, and applications, queryable with journalctl., making debugging significantly easier.

Working with Systemd Timer Commands

All timer units are managed using the systemctl command, just like other systemd units. The key distinction is that you use the -t timer filter to work specifically with timer units.

Use systemctl list-units to see timers that are currently loaded in memory (active, pending, or failed). Add --all to include inactive timers.

List active/loaded timer units
[root@host ~]# systemctl list-units -t timer UNIT LOAD ACTIVE SUB DESCRIPTION dnf-makecache.timer loaded active waiting dnf makecache --timer fstrim.timer loaded active waiting Discard unused blocks once a week fwupd-refresh.timer loaded active waiting Refresh fwupd metadata and update motd logrotate.timer loaded active waiting Daily rotation of log files plocate-updatedb.timer loaded active waiting Update the plocate database daily raid-check.timer loaded active waiting Weekly RAID setup health check systemd-tmpfiles-clean.timer loaded active waiting Daily Cleanup of Temporary Directories ...output omitted...

Use systemctl list-unit-files to see all installed timer unit files, including templates and disabled units:

List all installed timer unit files
[root@host ~]# systemctl list-unit-files -t timer UNIT FILE STATE PRESET dnf-makecache.timer enabled enabled fstrim.timer enabled enabled fwupd-refresh.timer enabled enabled insights-client.timer disabled disabled logrotate.timer enabled enabled mdadm-last-resort@.timer static - podman-auto-update.timer disabled disabled systemd-tmpfiles-clean.timer static - 16 unit files listed.
ColumnMeaning
UNIT FILEFull name of the timer unit file
STATECurrent enablement: enabled (starts at boot), disabled, static (no [Install] section; activated by other units)
PRESETWhether the vendor default enables or disables this unit

Starting or stopping a timer unit affects its current session only. The change does not survive a reboot unless the unit is also enabled.

Start and stop a timer unit
# Start a timer unit immediately (current session only) [root@host ~]# systemctl start logrotate.timer # Stop a running timer unit (current session only) [root@host ~]# systemctl stop logrotate.timer
💡Tip

Starting a timer unit manually is useful to test that it functions correctly before enabling it for persistent use at boot time.

Enabling a timer unit ensures it starts automatically at system boot. Disabling removes that automatic start.

Enable and disable timer units
# Enable at boot (does NOT start immediately) [root@host ~]# systemctl enable logrotate.timer # Enable AND start immediately with --now [root@host ~]# systemctl enable --now logrotate.timer # Disable at boot (does NOT stop immediately) [root@host ~]# systemctl disable logrotate.timer # Disable AND stop immediately with --now [root@host ~]# systemctl disable --now logrotate.timer
🔴Important

The systemctl enable command alone does not start the timer unit. Use --now to both enable and start in a single command. The same applies to disable — it does not stop the running unit without --now.

The systemctl status command displays comprehensive information about a timer unit, including its next scheduled trigger time and recent activity.

View timer unit status
[root@host ~]# systemctl status logrotate.timer ● logrotate.timer - Daily rotation of log files Loaded: loaded (/usr/lib/.../logrotate.timer; enabled; preset: enabled) Active: active (waiting) since Wed 2025-05-28 23:28:22 UTC; 54min ago Invocation: 3a95de54d4c04c539d43b3808c27e99a Trigger: Fri 2025-05-30 00:48:43 UTC; 24h left Triggers: ● logrotate.service Docs: man:logrotate(8) man:logrotate.conf(5) May 28 ... workstation systemd[1]: Started logrotate.timer - Daily rotation...

Key fields in the status output:

  • Active: Shows whether the timer is currently waiting (active), stopped, or failed
  • Trigger: The date and time of the next scheduled activation
  • Triggers: The service unit that will be activated when the timer fires

Timer Unit Configuration Files

Understanding how timer unit configuration files are structured gives you the knowledge to interpret existing timers and create custom ones.

Structure of a Timer Unit File

Like all systemd unit files, timer units contain sections delimited by section headers in brackets. The timer-specific options live in the [Timer] section:

/usr/lib/systemd/system/sysstat-collect.timer
# ...output omitted... [Unit] Description=Run system activity accounting tool every 10 minutes [Timer] OnCalendar=*:00/10 [Install] WantedBy=sysstat.service

The OnCalendar=*:00/10 option means: every hour (*), starting at minute 0, repeating every 10 minutes (00/10). The resulting trigger times within any given hour are 00, 10, 20, 30, 40, and 50 minutes past.

Timer Unit Calendar Expressions

The OnCalendar option accepts flexible calendar expressions. Here are examples ranging from simple to complex:

ExpressionMeaning
*:00/10Every 10 minutes
dailyEvery day at midnight (shorthand)
weeklyEvery Monday at midnight
monthlyFirst day of each month at midnight
Mon..Fri 09:00Weekdays at 09:00
2025-07-* 12:35,37,39:16Jul 2025, at 12:35:16, 12:37:16, 12:39:16
*-*-1 00:00:00First day of every month at midnight

Relative Timers with OnUnitActiveSec

Instead of calendar-based scheduling, you can trigger timers relative to when the service unit was last active. This is useful for maintenance tasks that should run a set interval after previous execution completes:

Relative timer example (fires 15 minutes after last activation)
[Timer] OnUnitActiveSec=15min

Where Timer Configuration Files Live

🔴Important — Never Edit Files in /usr/lib/systemd/system/

Do not modify unit configuration files in the /usr/lib/systemd/system/ directory. Package updates will overwrite your changes. To customize a systemd unit, copy its file to /etc/systemd/system/ and edit the copy. When two files share the same name, systemd uses the one in /etc/systemd/system/.

The correct workflow for customizing a timer unit:

Safely customize a timer unit
# 1. Copy the vendor file to the administrator directory [root@host ~]# cp /usr/lib/systemd/system/logrotate.timer \ /etc/systemd/system/logrotate.timer # 2. Edit the copy to make your changes [root@host ~]# vim /etc/systemd/system/logrotate.timer # 3. Reload systemd to apply the new configuration [root@host ~]# systemctl daemon-reload # 4. Enable and start the updated timer [root@host ~]# systemctl enable --now logrotate.timer
💡Tip

Use systemctl cat logrotate.timer to display the current effective configuration of a unit, including which file is being read. This is a quick way to confirm that your override in /etc/systemd/system/ is taking effect.

References

For complete documentation on timer unit options, consult the following man pages: systemctl(1), systemd.directives(7), systemd.time(7), systemd.timer(5), systemd.syntax(7), and systemd.unit(5).

Managing Temporary Files with systemd-tmpfiles

Most critical applications and services rely on temporary files and directories. Some use the /tmp directory for transient working data; others use task-specific locations such as daemon-specific directories under /run, which reside only in memory and are self-cleaning after a reboot.

However, temporary files on persistent storage can accumulate indefinitely, causing disk space issues or leaving stale data that can disrupt service operation. Likewise, daemons and scripts often require their expected temporary files and directories to exist before they can function correctly.

The systemd-tmpfiles Tool

Red Hat Enterprise Linux includes the systemd-tmpfilesA utility that provides structured, configurable management of temporary files and directories: creating, deleting, or setting permissions based on configuration files. tool, which provides a structured and configurable method to manage temporary directories and files. It reads instructions from configuration files and acts on them during system startup and on a schedule.

At system boot, one of the first systemd service units to launch is systemd-tmpfiles-setup.service. This service runs the systemd-tmpfiles --create command, which reads configuration files from three locations:

  • /usr/lib/tmpfiles.d/*.conf — Vendor-provided configurations (do not edit)
  • /run/tmpfiles.d/*.conf — Runtime configurations managed by daemons
  • /etc/tmpfiles.d/*.conf — Administrator-defined configurations (edit here)

Cleaning Temporary Files on a Schedule

To prevent long-running systems from filling up with stale data, a systemd timer unit called systemd-tmpfiles-clean.timer periodically triggers systemd-tmpfiles-clean.service, which runs systemd-tmpfiles --clean.

View the timer unit configuration with systemctl cat:

systemctl cat systemd-tmpfiles-clean.timer
[user@host ~]$ systemctl cat systemd-tmpfiles-clean.timer # /usr/lib/systemd/system/systemd-tmpfiles-clean.timer # SPDX-License-Identifier: LGPL-2.1-or-later ... [Unit] Description=Daily Cleanup of Temporary Directories Documentation=man:tmpfiles.d(5) man:systemd-tmpfiles(8) ConditionPathExists=!/etc/initrd-release [Timer] OnBootSec=15min OnUnitActiveSec=1d

Interpreting these timer parameters:

  • OnBootSec=15min — Triggers the cleanup service 15 minutes after the system boots
  • OnUnitActiveSec=1d — Triggers again 24 hours after the service was last activated

After checking the current timer status, you can view when the next cleanup is scheduled:

Check timer status
[root@host ~]# systemctl status systemd-tmpfiles-clean.timer ● systemd-tmpfiles-clean.timer - Daily Cleanup of Temporary Directories Loaded: loaded (/usr/lib/.../systemd-tmpfiles-clean.timer; static) Active: active (waiting) since Fri 2025-05-30 13:59:31 UTC; 19min ago Trigger: Sat 2025-05-31 14:14:31 UTC; 23h left Triggers: ● systemd-tmpfiles-clean.service

Cleaning Temporary Files Manually

You can also run the cleanup manually without waiting for the timer to fire. The systemd-tmpfiles --clean command purges all files that were not accessed, changed, or modified more recently than the maximum age defined in the configuration files:

Manual tmpfiles operations
# Create files and directories as defined in config files [root@host ~]# systemd-tmpfiles --create # Clean (purge aged-out) files as defined in config files [root@host ~]# systemd-tmpfiles --clean # Apply only a specific configuration file (for testing) [root@host ~]# systemd-tmpfiles --clean /etc/tmpfiles.d/myapp.conf
💡Tip: Testing Configurations

When testing new or modified configurations, apply commands from a single configuration file at a time by specifying its full path on the command line. This avoids unexpected side effects from other configuration files.

The tmpfiles.d Configuration File Syntax

Configuration files for systemd-tmpfiles follow a specific columnar syntax. Each non-comment line specifies an action to take on a path. Understanding this syntax allows you to create custom configurations for your applications.

Column Format

Each line in a tmpfiles.d configuration file contains seven space-separated columns:

TypeRequired
Action to take
PathRequired
File/directory path
ModeOptional
Permissions (octal)
UIDOptional
Owner user
GIDOptional
Owner group
AgeOptional
Max file age
ArgumentOptional
Extra data

Common Type Values

TypeAction
dCreate directory if it does not exist; does not purge contents automatically
DCreate directory if it does not exist; remove contents when cleaned
fCreate a file if it does not exist
LCreate a symbolic link
zSet ownership and permissions on existing files
ZRecursively set ownership, permissions, and restore SELinux contexts
rRemove a path if it exists (without recursion)
RRecursively remove a path if it exists

Configuration Examples

Create /run/systemd/seats (no auto-cleanup)
d /run/systemd/seats 0755 root root -
  • Creates /run/systemd/seats if it does not exist
  • If the directory already exists, takes no action
  • Sets root user and root group as owners
  • Sets permissions to 0755 (rwxr-xr-x)
  • The - age value means this directory is never automatically purged
Create /home/student and purge files older than 1 day
D /home/student 0700 student student 1d
  • Creates the /home/student directory if it does not exist
  • If the directory exists, removes its contents on cleanup
  • Sets student user and student group as owners
  • Sets permissions to 0700 (rwx------)
  • Age is 1d: files not accessed, changed, or modified within the past 1 day will be removed when systemd-tmpfiles --clean runs
Create a symlink from /run/fstablink to /etc/fstab
L /run/fstablink - root root - /etc/fstab
  • Creates the /run/fstablink symbolic link pointing to /etc/fstab
  • Sets root user and root group as owners
  • The - in the Mode and Age fields means use defaults and never purge
  • The seventh column (Argument) specifies the symlink target: /etc/fstab

Configuration File Precedence

When the same configuration file name exists in multiple directories, systemd-tmpfiles applies a strict precedence rule:

/etc/tmpfiles.d/*.conf Highest priority — admin settings
/run/tmpfiles.d/*.conf Runtime (daemon-managed)
/usr/lib/tmpfiles.d/*.conf Lowest — vendor defaults (do not edit)

This precedence hierarchy means you can safely override vendor settings by copying a file from /usr/lib/tmpfiles.d/ to /etc/tmpfiles.d/ and editing the copy. Package updates will never overwrite your administrator-managed settings in /etc/tmpfiles.d/.

References

For complete syntax documentation, see: systemd-tmpfiles(8), tmpfiles.d(5), stat(1), stat(2), and systemd.timer(5) man pages.

Scheduling Recurring System Tasks with Cron

While systemd timer units are now the preferred mechanism for recurring system tasks in RHEL 10, the traditional cronA time-based job scheduler. The crond daemon reads crontab files that specify commands and their schedules, running them at the specified times. system remains available and is still appropriate for time-sensitive tasks on always-running systems.

Recurring System Jobs vs User Jobs

System administrators often need recurring jobs that run from system accounts rather than user accounts. These are configured using system-wide crontab filesConfiguration files read by the crond daemon that specify when to run commands. System-wide crontab files include an extra field for the username that runs the command. instead of the user-level crontab command. System-wide crontab entries have an extra field: the username to run the command as.

The /etc/crontab File Structure

The /etc/crontab file includes a helpful comment block explaining the format:

/etc/crontab
SHELL=/bin/bash PATH=/sbin:/bin:/usr/sbin:/usr/bin MAILTO=root # For details see man 4 crontabs # Example of job definition: # .------------- minute (0 - 59) # | .---------- hour (0 - 23) # | | .------- day of month (1 - 31) # | | | .---- month (1 - 12) OR jan,feb,mar,apr ... # | | | | .- day of week (0 - 6) (Sunday=0 or 7) # | | | | | # * * * * * user-name command to be executed
🔴Important

Apart from comments, the /etc/crontab file is otherwise empty by default. Never edit the /etc/crontab file directly. Use it solely as a reference. Place individual crontab files in /etc/cron.d/ instead, to prevent package updates from overwriting your entries.

The Cron Directory Structure

The cron system uses a structured set of directories for different job types. Always create custom crontab files in /etc/cron.d/:

LocationContentsNotes
/etc/crontabReference format fileDo not edit; reference only
/etc/cron.d/Custom system crontab filesCreate your files here
/etc/cron.hourly/Executable scripts run every hourScripts, not crontab files
/etc/cron.daily/Executable scripts run dailyScripts, not crontab files
/etc/cron.weekly/Executable scripts run weeklyScripts, not crontab files
/etc/cron.monthly/Executable scripts run monthlyScripts, not crontab files

Creating a System Cron Job

To create a system cron job, add a new crontab file in the /etc/cron.d/ directory. The following example appends a timestamp to a file every two minutes:

/etc/cron.d/timestamp_example
# Example timestamp cronjob SHELL=/bin/bash PATH=/sbin:/bin:/usr/sbin:/usr/bin MAILTO=root # Every 2 minutes, as root, append date to file */2 * * * * root date >> /tmp/timestamp_example.txt

The */2 notation means "every 2 units" — in the minute field, it runs every two minutes. After several minutes, verify the output:

Verify the job is running
[root@host ~]# cat /tmp/timestamp_example.txt Fri May 30 15:48:01 UTC 2025 Fri May 30 15:50:01 UTC 2025 Fri May 30 15:52:01 UTC 2025 Fri May 30 15:54:01 UTC 2025 ...output omitted...

The /etc/cron.d/0hourly File

Every script placed in /etc/cron.hourly/ is executed via the /etc/cron.d/0hourly secondary crontab file. This file instructs crond to run the run-partsA script that runs every executable file found in a specified directory, used by cron to execute all scripts in the hourly, daily, weekly, and monthly directories. command on the /etc/cron.hourly directory as root at one minute past every hour:

/etc/cron.d/0hourly
# Run the hourly jobs SHELL=/bin/bash PATH=/sbin:/bin:/usr/sbin:/usr/bin MAILTO=root 01 * * * * root run-parts /etc/cron.hourly
💡Tip

Use chmod +x scriptname to make a script executable before placing it in /etc/cron.hourly, /etc/cron.daily, or any cron directory. The run-parts command only runs files that are executable.

Running Periodic Commands with Anacron

The crond daemon assumes the system is always running. If a system is powered off or hibernated when a cron job was scheduled, that job is simply skipped — it will not run when the system comes back online. This is where anacronA job scheduler that fills the gap left by crond: it ensures daily, weekly, and monthly jobs run even if the system was offline at the scheduled time, running them when the system becomes ready. fills an important gap.

The anacron command uses the run-parts script to execute daily, weekly, or monthly jobs from the /etc/anacrontab configuration file. It ensures scheduled jobs always run and are not accidentally skipped due to the system being turned off or hibernated.

How Anacron Works

Anacron tracks the last time each job ran by storing timestamps in the /var/spool/anacron/ directory. When the system starts or anacron is invoked, it compares the current date against those timestamps. If a job is overdue, anacron runs it (after a configurable delay).

The /etc/anacrontab Configuration File

The /etc/anacrontab file has a different syntax from regular crontab files, using four fields per job line:

/etc/anacrontab
# /etc/anacrontab: configuration file for anacron # See anacron(8) and anacrontab(5) for details. SHELL=/bin/sh PATH=/sbin:/bin:/usr/sbin:/usr/bin MAILTO=root # the maximal random delay added to the base delay RANDOM_DELAY=45 # jobs start only during these hours START_HOURS_RANGE=3-22 #period-in-days delay-minutes job-id command 1 5 cron.daily nice run-parts /etc/cron.daily 7 25 cron.weekly nice run-parts /etc/cron.weekly @monthly 45 cron.monthly nice run-parts /etc/cron.monthly

Anacrontab Field Descriptions

FieldDescriptionExample
Period in days How often to run the job (in days). Also accepts macros: @daily (=1), @weekly (=7), @monthly. 1, 7, @monthly
Delay in minutes How long the crond daemon waits before starting the job after anacron is invoked. Prevents all jobs from running simultaneously at startup. 5, 25, 45
Job identifier A unique name that identifies this job in log messages and the /var/spool/anacron/ timestamp files. cron.daily
Command The command to execute. Typically nice run-parts /etc/cron.daily to run all scripts in the directory at reduced priority. nice run-parts /etc/cron.daily

Important Anacrontab Variables

  • RANDOM_DELAY — A random number of minutes (up to this value) added to the base delay for each job. This staggers job execution to avoid resource spikes when multiple jobs become due simultaneously.
  • START_HOURS_RANGE — Jobs only start during this time window. If a job is due but the system time is outside this range, the job will wait until the next day's window. In the example above, jobs only run between 03:00 and 22:00.

The Relationship Between Cron and Anacron

The crond service and anacron work together in a carefully orchestrated chain:

crond
Runs every minute
→
/etc/cron.d/0hourly
Runs run-parts at :01
→
/etc/cron.hourly/0anacron
Invokes anacron -s
→
anacron reads
/etc/anacrontab
Runs daily/weekly/monthly
  1. crond reads /etc/cron.d/0hourly and at one minute past every hour executes run-parts /etc/cron.hourly as root.
  2. The /etc/cron.hourly/ directory contains a shell script named 0anacron that conditionally runs /usr/sbin/anacron -s.
  3. anacron -s reads /etc/anacrontab and executes nice run-parts on the cron.daily, cron.weekly, and cron.monthly directories (with a random delay of up to 45 minutes).
📋Note

The crontabs command is a legacy name for the run-parts script and the system crontab. You may encounter it in older documentation.

ℹCron vs Systemd Timers: When to Use Each

Use systemd timer units for new scheduled system tasks on RHEL 10. They integrate with journald for logging, support richer calendar expressions, and are the preferred method going forward. Use cron/anacron when managing legacy scripts already placed in /etc/cron.d/ or the periodic directories, or when time-sensitive precision is required.

References

For complete documentation, see: crontab(5), crontabs(4), anacron(8), anacrontab(5), and crond(8) man pages.

Objective: Create a system cron job that compresses the access log every Sunday at 02:00 as the root user.

Create /etc/cron.d/archive_access_log
# Create the crontab file in /etc/cron.d/ [root@serverb ~]# cat > /etc/cron.d/archive_access_log <<'EOF' # Archive access log every Sunday at 02:00 SHELL=/bin/bash PATH=/sbin:/bin:/usr/sbin:/usr/bin MAILTO=root 0 2 * * 0 root gzip -c /var/log/httpd/access_log >> /var/log/httpd/access_log_archive.gz EOF # Verify crond picks up the file (no restart needed) [root@serverb ~]# ls -la /etc/cron.d/archive_access_log -rw-r--r-- 1 root root 198 May 30 16:15 /etc/cron.d/archive_access_log # crond reads /etc/cron.d/ automatically; no reload required

Objective: Change the systemd-tmpfiles-clean.timer to run every 30 minutes instead of daily, without editing the vendor-provided file.

Safely override the vendor timer configuration
# 1. Copy vendor file to administrator directory [root@serverb ~]# cp /usr/lib/systemd/system/systemd-tmpfiles-clean.timer \ /etc/systemd/system/systemd-tmpfiles-clean.timer # 2. Edit the copy — change OnUnitActiveSec to 30min [root@serverb ~]# sed -i 's/OnUnitActiveSec=1d/OnUnitActiveSec=30min/' \ /etc/systemd/system/systemd-tmpfiles-clean.timer # 3. Reload systemd daemon to apply the new configuration [root@serverb ~]# systemctl daemon-reload # 4. Verify the change took effect [root@serverb ~]# systemctl cat systemd-tmpfiles-clean.timer | grep OnUnit OnUnitActiveSec=30min

Chapter Summary

In this chapter, you learned about the methods used to run recurring system tasks automatically in Red Hat Enterprise Linux 10.

Key Concepts

  • Systemd timer units are the preferred mechanism for scheduling recurring system tasks in RHEL 10. A .timer file defines the schedule; a matching .service file performs the work. Timer events are logged in the system journal for easy debugging.
  • Timer management uses standard systemctl commands: list-units -t timer, start, stop, enable --now, and status. Always run systemctl daemon-reload after modifying unit files.
  • Safe customization of vendor-provided unit files requires copying from /usr/lib/systemd/system/ to /etc/systemd/system/ and editing the copy. Files in /etc/systemd/system/ take precedence.
  • systemd-tmpfiles provides structured management of temporary files and directories using configuration files in /etc/tmpfiles.d/, /run/tmpfiles.d/, and /usr/lib/tmpfiles.d/. The systemd-tmpfiles-clean.timer runs periodic cleanups automatically.
  • The cron system (crond + anacron) remains important for time-sensitive jobs and legacy scripts. Always create custom crontab files in /etc/cron.d/, never edit /etc/crontab directly.
  • Anacron ensures that daily, weekly, and monthly jobs in the cron directories are not skipped due to the system being offline, making it ideal for workstations and systems that are not always running.
