Learning Objectives

After completing this chapter, you will be able to:

  • Explain the difference between MBR and GPT partitioning schemes and when to use each.

Disk Partitioning Concepts

Disk partitioningThe process of dividing a physical disk drive into multiple independent logical storage units, each treated by the OS as a separate device. divides a hard drive into multiple logical storage areas. Administrators use partitions to organize storage based on different requirements.

Why Partition Disks?

  • Limit available space to specific applications or users, preventing runaway writes from filling the entire disk.
  • Separate operating system and program files from user data for easier management and backup.
  • Create a dedicated area for swap spaceDisk space used as an extension of RAM. When physical memory is full, the kernel moves inactive memory pages to swap to free up RAM for active processes..
  • Limit disk space usage to improve performance of diagnostic tools and backup imaging.

MBR vs. GPT Partitioning Schemes

On x86 systems, the two primary partitioning schemes are Master Boot Record (MBR)A legacy partitioning scheme stored in the first 512 bytes of a disk. Supports up to 4 primary partitions and disk sizes up to 2 TiB. and GUID Partition Table (GPT)The modern partitioning standard used with UEFI firmware. Supports up to 128 partitions, disk sizes up to 8 ZiB, and provides redundant partition tables for fault tolerance..

The Master Boot Record (MBR) partitioning scheme is the standard for systems running BIOS firmwareBasic Input/Output System — legacy firmware that initializes hardware at boot time. Being replaced by UEFI in modern systems..

  • Maximum of 4 primary partitions (or 3 primary + 1 extended)
  • Extended partition acts as a container for multiple logical partitionsPartitions created inside an extended partition on an MBR disk. Linux can number them from sdb5 onward, allowing more than 4 partitions.
  • Up to 15 total partitions (3 primary + 1 extended + 11 logical)
  • Maximum disk and partition size: 2 TiB (32-bit addressing)
  • Single partition table — no redundancy

⚠️MBR Limitation

The 2 TiB size limit is now commonly encountered with modern storage. If your disk exceeds 2 TiB, you must use GPT. MBR is considered a legacy scheme.

The GUID Partition Table (GPT) is the modern standard for systems running UEFI firmwareUnified Extensible Firmware Interface — modern replacement for BIOS. Provides faster boot, larger disk support, and a standardized interface between the OS and firmware.. It addresses all of the MBR limitations.

  • Maximum of 128 partitions (no primary/extended distinction)
  • Each partition gets a unique GUIDGlobally Unique Identifier — a 128-bit number used to uniquely identify disks and partitions in GPT. Eliminates naming collisions across systems. identifier
  • Supports disks and partitions up to 8 ZiB (64-bit addressing)
  • Redundant partition tables: primary at the start of the disk, backup at the end
  • Uses a checksumA calculated value used to detect data corruption. GPT stores checksums of both the partition table header and partition entries to detect errors. to detect errors in the GPT header and partition table
  • Required for disks larger than 2 TiB

Managing Partitions with parted

Red Hat Enterprise Linux provides the parted commandThe standard command-line partition editor in RHEL. It supports both MBR and GPT schemes and can be used interactively or with single non-interactive commands. as the standard partition editor. It supports both MBR and GPT schemes and takes a disk device name as its first argument.

Viewing the Partition Table

The print subcommand displays the partition table for a disk. You can pass it non-interactively:

View partition table
[root@host ~]# parted /dev/sda print Model: QEMU QEMU HARDDISK (scsi) Disk /dev/sda: 53.7GB Sector size (logical/physical): 512B/512B Partition Table: msdos Disk Flags: Number Start End Size Type File system Flags 1 1049kB 10.7GB 10.7GB primary xfs boot 2 10.7GB 53.7GB 42.9GB primary xfs

To use parted interactively, run it with just the device name. Use quit to exit:

Interactive parted session
[root@host ~]# parted /dev/sda GNU Parted 3.6 Using /dev/sda Welcome to GNU Parted! Type 'help' to view a list of commands. (parted) print ...output... (parted) quit [root@host ~]#

By default, parted displays sizes in powers of 10 (MB, GB). Change units with the unit parameter (s for sectors, MiB/GiB for binary, MB/GB for decimal).

Writing a Disk Label on a New Disk

Before creating partitions on a new disk, write a disk labelA data structure written at the beginning of a disk that identifies the partitioning scheme in use (MBR or GPT). Also called the partition table type. to indicate which partitioning scheme to use.

Create MBR or GPT disk label
# Write an MBR (msdos) disk label [root@host ~]# parted /dev/sdb mklabel msdos # Write a GPT disk label (preferred) [root@host ~]# parted /dev/sdb mklabel gpt
⚠️Warning: Data Destructive

The mklabel subcommand wipes the existing partition table. Any data stored in existing partitions becomes inaccessible if partition boundaries move. Always verify the device name before running this command.

Creating MBR Partitions

Use mkpart to create a new partition. For MBR disks, you specify a partition type (primary, extended, or logical).

Create MBR partition (interactive)
[root@host ~]# parted /dev/sdb (parted) mkpart Partition type? primary/extended? primary File system type? [ext2]? xfs Start? 2048s End? 1000MB (parted) quit Information: You may need to update /etc/fstab. # Run udevadm to register the new partition device file [root@host ~]# udevadm settle

As an alternative, create the partition in a single non-interactive command:

Non-interactive MBR partition creation
[root@host ~]# parted /dev/sdb mkpart primary xfs 2048s 1000MB [root@host ~]# udevadm settle
💡Partition Alignment

Starting a partition at sector 2048 (1 MiB from the disk start) is the standard alignment for most modern disks. This ensures the partition is aligned with the disk's physical block size for optimal performance. The parted command warns you if the start position causes misalignment.

Creating GPT Partitions

GPT partitions work similarly, but instead of a partition type, you specify a partition name (a descriptive label).

Create GPT partition (interactive and non-interactive)
# Interactive approach [root@host ~]# parted /dev/sdb (parted) mkpart Partition name? []? userdata File system type? [ext2]? xfs Start? 2048s End? 1000MB (parted) quit # Non-interactive equivalent [root@host ~]# parted /dev/sdb mkpart userdata xfs 2048s 1000MB [root@host ~]# udevadm settle

Deleting Partitions

Use the rm subcommand followed by the partition number. This works for both MBR and GPT schemes.

Delete partition number 1
# Interactive [root@host ~]# parted /dev/sdb (parted) print <-- identify the partition number first (parted) rm 1 (parted) quit # Non-interactive [root@host ~]# parted /dev/sdb rm 1
⚠️Immediate Effect

The rm subcommand immediately removes the partition from the partition table on disk. There is no confirmation prompt. Always run print first to confirm you have the correct partition number.

Creating File Systems

After creating a partition, the next step is to add a file systemA method for organizing and storing files on a storage device. The file system defines how data is laid out on disk, how files are named, and how metadata like permissions and timestamps are stored. to it. The file system defines how data is organized on disk. Without a file system, a partition is just raw disk space.

XFS — The Recommended Default

XFSA high-performance 64-bit journaling file system developed by Silicon Graphics. It is the default and recommended file system for RHEL 10, known for scalability with large files and parallel I/O. is the default and recommended file system for RHEL 10. It is a high-performance, journalingA file system feature that records changes in a transaction log (journal) before applying them to the main file system. This allows recovery after a crash without a full disk scan. file system that handles large files and parallel I/O efficiently. Use mkfs.xfs to format a partition:

Format partition with XFS
[root@host ~]# mkfs.xfs /dev/sdb1 meta-data=/dev/sdb1 isize=512 agcount=4, agsize=60992 blks = sectsz=512 attr=2, projid32bit=1 = crc=1 finobt=1, sparse=1, rmapbt=1 = reflink=1 bigtime=1 inobtcount=1 nrext64=1 = exchange=0 data = bsize=4096 blocks=243968, imaxpct=25 = sunit=0 swidth=0 blks naming =version 2 bsize=4096 ascii-ci=0, ftype=1, parent=0 log =internal log bsize=4096 blocks=1566, version=2 = sectsz=512 sunit=0 blks, lazy-count=1 realtime =none extsz=4096 blocks=0, rtextents=0 Discarding blocks...Done.

ext4 — The Extended File System

ext4The fourth extended file system. A journaling file system widely used on Linux systems. Supports volumes up to 1 EiB and files up to 16 TiB. Generally compatible with ext2/ext3 tools. is a widely-used Linux journaling file system. While XFS is the default in RHEL 10, ext4 remains supported. Use mkfs.ext4 to format:

Format partition with ext4
[root@host ~]# mkfs.ext4 /dev/sdb1

Mounting File Systems & /etc/fstab

Manual Mounting

Use the mount commandAttaches a formatted file system to a directory in the Linux directory hierarchy. Once mounted, files on the device are accessible at that directory path. to attach a formatted partition to a mount pointA directory in the existing file system hierarchy where a new file system is attached. The directory must already exist before mounting. directory:

Manual mount and verification
# Mount /dev/sdb1 to /mnt [root@host ~]# mount /dev/sdb1 /mnt # Verify the mount [root@host ~]# mount | grep sdb1 /dev/sdb1 on /mnt type xfs (rw,relatime,seclabel,attr2,inode64,logbufs=8,logbsize=32k,noquota)

Manual mounts are temporary — they disappear on reboot. To make a mount persist across reboots, add it to /etc/fstab.

Persistent Mounts with /etc/fstab

The /etc/fstab fileThe file system table. A configuration file that lists all file systems to mount at system boot. Each line describes one file system with device, mount point, type, options, dump flag, and fsck order. lists file systems to mount automatically at boot. Each line has six whitespace-separated fields.

Example /etc/fstab entries
[root@host ~]# cat /etc/fstab UUID=a8063676-44dd-409a-b584-68be2c9f5570 / xfs defaults 0 0 UUID=7a20315d-ed8b-4e75-a5b6-24ff9e1f9838 /dbdata xfs defaults 0 0

Each field explained:

  • Field 1 (Device): Use the partition UUIDUniversally Unique Identifier — a 128-bit value assigned to a file system when it is created and stored in the superblock. Unlike device names, UUIDs never change even if disks are reordered.. Red Hat recommends UUIDs because device names like /dev/sdb1 can change if disks are reordered or a cloud provider changes storage.
  • Field 2 (Mount Point): The directory where the file system is mounted. Must already exist.
  • Field 3 (FS Type): The file system type (xfs, ext4, swap, etc.).
  • Field 4 (Options): Mount options. defaults enables the standard set of options (rw, suid, dev, exec, auto, nouser, async).
  • Field 5 (Dump): Used by the legacy dump backup tool. Set to 0 for most systems.
  • Field 6 (fsck order): For XFS, set to 0 (XFS does not use fsck). For ext4, use 1 for root and 2 for others.

Finding the UUID

Use lsblk --fs to display UUIDs for all block devices:

List block device UUIDs
[root@host ~]# lsblk --fs NAME FSTYPE FSVER LABEL UUID FSAVAIL FSUSE% MOUNTPOINTS sda ├sda1 ├sda2 xfs boot 49dd...75fdf 312M 37% /boot └sda3 xfs root 8a90...cedda 4.8G 48% / sdb └sdb1 xfs d85f5387-9791-400a-9a74-f637e7ee68aa

Applying fstab Changes

After editing /etc/fstab, run systemctl daemon-reload so systemd registers the new configuration. You can then mount using only the mount point (systemd resolves the rest):

Apply fstab and test mount
[root@host ~]# systemctl daemon-reload [root@host ~]# mount /backup # systemd resolves /backup via /etc/fstab
🔴Critical: Validate Before Rebooting

An incorrect /etc/fstab entry can render the system unbootable. Always verify new entries with mount -a (mount all fstab entries) and check for errors before rebooting. Alternatively, use findmnt --verify to parse and validate the file.

Managing Swap Space

What Is Swap Space?

A swap spaceDisk space managed by the Linux kernel's memory management subsystem. Used to hold inactive memory pages when physical RAM is under pressure. System virtual memory = RAM + swap. is an area of disk under the control of the Linux kernel memory management subsystem. The kernel uses swap to supplement system RAM by holding inactive memory pages on disk.

When memory usage exceeds a defined limit, the kernel scans RAM for idle pages assigned to processes, writes them to swap, and reassigns that RAM to active processes. If a program needs a page currently in swap, the kernel swaps out another idle page to make room, then recalls the needed page.

⚠️Swap is Slow

Swap space is on disk — dramatically slower than RAM. While swap prevents out-of-memory crashes, excessive swapping (thrashingA condition where a system spends more time swapping pages in and out of memory than doing useful work. Indicates the system has insufficient RAM for its workload.) severely degrades performance. Swap augments RAM but does not replace it — if your system constantly swaps, add more RAM.

Swap Space Sizing

Red Hat provides guidance for sizing swap space based on total system RAM:

Total RAM Recommended Swap Swap for Hibernation
2 GB or lessTwice the RAMThree times the RAM
2 GB – 8 GBSame as RAMTwice the RAM
8 GB – 64 GBAt least 4 GB1.5 times the RAM
More than 64 GBAt least 4 GBHibernation not recommended

HibernationA power-saving feature on laptops and desktops that writes the entire RAM contents to swap space before powering off. On resume, the kernel restores RAM from swap, avoiding a full boot. saves RAM contents to swap before powering off. For systems that hibernate, swap must be larger than RAM.

Creating Swap Space

Creating swap space involves two steps: creating a partition with the linux-swap type, then formatting it with mkswap.

Step 1: Create a swap partition

Create swap partition on /dev/sdb
# Create a 256 MB swap partition named "swap1" [root@host ~]# parted /dev/sdb mkpart swap1 linux-swap 1001MB 1257MB [root@host ~]# udevadm settle

Step 2: Format with mkswap

The mkswap commandFormats a partition as swap space by writing a swap signature block at the beginning. Unlike file systems, most of the partition is left unformatted for the kernel to manage as memory pages. applies a swap signature to the partition, enabling the kernel to use it as swap:

Format partition as swap
[root@host ~]# mkswap /dev/sdb2 Setting up swapspace version 1, size = 244 MiB (255848448 bytes) no label, UUID=39e2667a-9458-42fe-9665-c5c854605881

Activating and Deactivating Swap

Use swapon to activate swap space and swapoff to deactivate it. These changes are temporary — they do not survive a reboot.

Activate, verify, and deactivate swap
# Activate a swap partition [root@host ~]# swapon /dev/sdb2 # Verify swap spaces and usage [root@host ~]# swapon --show NAME TYPE SIZE USED PRIO /dev/sdb2 partition 244M 0B -2 [root@host ~]# free total used free shared buff/cache available Mem: 1873036 135044 1536040 16748 201952 1575680 Swap: 249852 0 249852 # Deactivate swap (moves pages back to RAM if possible) [root@host ~]# swapoff /dev/sdb2

Persistent Swap in /etc/fstab

To activate swap automatically at boot, add an entry to /etc/fstab. Use the UUID from the mkswap output, and use swap as both the mount point placeholder and file system type:

Swap entry in /etc/fstab
# Format: UUID swap swap options dump fsck UUID=39e2667a-9458-42fe-9665-c5c854605881 swap swap defaults 0 0 # After editing fstab, reload systemd [root@host ~]# systemctl daemon-reload

Swap Space Priorities

When multiple swap spaces exist, the kernel uses them in priority order — highest priority first. If two swap spaces share the same priority, the kernel writes to them in a round-robinA scheduling method that distributes work evenly across multiple resources by cycling through them in order. Useful for load balancing swap across multiple disks. fashion.

Set the priority with the pri=N option in /etc/fstab. The default priority is -2. Higher numbers mean higher priority.

Three swap spaces with different priorities
# This swap has default priority (-2) — used last UUID=af30cbb0-3866-466a-825a-58889a49ef33 swap swap defaults 0 0 # Priority 4 — used second UUID=39e2667a-9458-42fe-9665-c5c854605881 swap swap pri=4 0 0 # Priority 10 — used first (highest priority) UUID=fbd7fa60-b781-44a8-961b-37ac3ef572bf swap swap pri=10 0 0

Verify priorities with swapon --show:

View active swap spaces and priorities
[root@host ~]# swapon --show NAME TYPE SIZE USED PRIO /dev/sdb2 partition 489M 0B 10 /dev/sdb3 partition 488M 0B 20

Chapter Summary

In this chapter, you learned to manage local storage devices from the command line:

  • MBR vs. GPT: MBR is for legacy BIOS systems (max 4 primary, 2 TiB). GPT is for UEFI systems (max 128 partitions, 8 ZiB), with redundant tables and checksums.
  • parted: The standard RHEL partition editor. Supports both schemes with interactive and non-interactive modes.
  • XFS: The default and recommended RHEL 10 file system. Use mkfs.xfs to format.
  • Mounting: Use mount for temporary mounts. Add UUID-based entries to /etc/fstab for persistent mounts. Run systemctl daemon-reload after editing fstab.
  • Swap: Disk space used as virtual memory extension. Size based on RAM. Create with parted, format with mkswap, activate with swapon. Set priorities with pri=N in fstab.
