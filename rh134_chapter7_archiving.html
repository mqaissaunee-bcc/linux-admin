Introduction to Archives

An archiveA single file or device that bundles multiple files together, preserving their structure, metadata, and permissions. is a single regular file or device file that contains multiple files and directories. On Linux systems, archiving is the standard mechanism for bundling files before backup, distribution, or transfer.

Archive files serve two primary purposes in system administration:

  • Backup: Collecting files into a single manageable unit that can be stored, versioned, and restored.
  • Transfer: Simplifying movement of directory trees across a network when other methods such as rsync are unavailable or overly complex.

Archives can be created with or without compressionThe process of encoding data using fewer bits to reduce file size, making storage and transfers more efficient. to reduce file size. On RHEL 10, the primary archiving tool is the tar command, which supports multiple embedded compression algorithms.

📘Historical Context: zip vs. tar

The original zipA file format and utility (PKZIP) that both archives and compresses files; widely used on Windows and supported on RHEL via the zip and unzip commands. utility uses the PKZIP algorithm (Phil Katz's ZIP for MS-DOS). It remains available on RHEL 10 via the zip and unzip commands. However, Linux administrators overwhelmingly prefer tar with gzip, bzip2, or xz compression because these algorithms generally achieve better compression ratios, and tar preserves Linux file metadata (ownership, permissions, symbolic links, SELinux contexts) more faithfully than the zip format.

When to Use Archives vs. Other Methods

Understanding when archiving is the right tool helps you choose efficiently:

  • Use tar when you need to preserve complete directory trees with permissions, when working with backup jobs, or when packaging software for distribution.
  • Use rsync for incremental synchronization of live directories between systems — rsync is more efficient for ongoing syncs because it only transfers changed files.
  • Use scp or sftp for quick one-off file copies between systems where directory preservation matters less.
  • Use cp -a for local copies where you want to preserve all file attributes without creating an archive bundle.
💡Tip: Archive First, Then Transfer

When transferring a large directory over a network, first create a compressed tar archive, then transfer the single archive file. This reduces transfer overhead and ensures the entire directory arrives atomically (all-or-nothing), making it easier to verify the transfer was complete.

The tar Command: Options and Syntax

The tarTape Archive utility. The standard Linux command for creating, listing, and extracting archive files. Predates most modern compression tools and originated for writing to tape drives. (Tape Archive) utility is the common command on Linux for creating, managing, and extracting archives. A tar archiveA file created by the tar command containing a structured sequence of file metadata and data blocks, with an index allowing individual file extraction. is a structured sequence of file metadata and data with an internal index, allowing individual files to be extracted without processing the entire archive.

Every tar invocation requires exactly one action option:

📘Legacy Option Style

The tar command supports a legacy option style that omits the leading dash (-), such as tar cvf archive.tar . instead of tar -cvf archive.tar . You may encounter this in older scripts or documentation. The behavior is identical, but Red Hat recommends using the standard short-option or long-option style for consistency and readability.

General Syntax Pattern

tar Command Syntax
# Create: tar -c[options]f ARCHIVE_NAME SOURCE [SOURCE...] [student@workstation ~]$ tar -czf backup.tar.gz /source/directory # List: tar -t[options]f ARCHIVE_NAME [student@workstation ~]$ tar -tf backup.tar.gz # Extract: tar -x[options]f ARCHIVE_NAME [-C DESTINATION] [student@workstation ~]$ tar -xf backup.tar.gz -C /destination/
💡Tip: -f Must Come Last in Option Groups

When combining options (e.g., -czf), the -f flag must be last because it takes the archive filename as its immediate argument. Writing tar -cfz backup.tar.gz will fail because tar tries to use z as the filename. Always write tar -czf archive.tar.gz.

Creating Archives

To create an archive with tar, use the -c (create) and -f (file) options, specifying the archive filename as the first argument, followed by the files and directories to include.

Absolute vs. Relative Path Names

The tar command recognizes both absolute path namesA path starting with / that specifies a file location from the root of the filesystem, e.g. /etc/hosts. and relative path namesA path that specifies a file location relative to the current working directory, e.g. etc/hosts or ./etc/hosts.. By default, tar strips the leading forward slash (/) from absolute paths, storing files internally with relative path names.

This is an important safety feature. When you later extract the archive, the files are extracted relative to your current directory rather than overwriting the original absolute-path locations. For example, /etc/hosts is stored as etc/hosts and extracted to ./etc/hosts in your current directory.

⚠Why Path Stripping Matters

If tar did not strip the leading slash, extracting an archive containing /etc/passwd would overwrite the live system's password file. The stripping behavior protects administrators from accidental overwrites. The warning message tar: Removing leading '/' from member names is normal and expected when archiving absolute paths.

Creating a Basic Archive (User Files)

Create Archive from Individual Files
# Create mybackup.tar containing three log files [student@workstation ~]$ tar -cf mybackup.tar myapp1.log myapp2.log myapp3.log # The -v (verbose) flag shows each file as it is added [student@workstation ~]$ tar -cvf mybackup.tar myapp1.log myapp2.log myapp3.log myapp1.log myapp2.log myapp3.log
📘Overwrite Behavior

If a file with the same name as the requested archive already exists in the target directory, the tar command overwrites it without prompting. Always verify your archive name before running the command, especially in automated scripts.

Creating a System Archive (Root Required)

Creating an archive of system directories such as /etc requires root privileges because only privileged users can read all files in those directories. An unprivileged user can attempt to archive /etc, but the resulting archive will silently exclude any files the user lacks permission to read.

Archive /etc as Root
# Root creates a complete archive of /etc [root@host ~]# tar -cf /root/etc-backup.tar /etc tar: Removing leading '/' from member names # Verify the archive was created and check its size [root@host ~]# ls -lh /root/etc-backup.tar -rw-r--r--. 1 root root 12M Mar 16 10:22 /root/etc-backup.tar
🔴Important: Extended Attributes Are Not Preserved by Default

By default, tar does not preserve extended file attributes. This means:

  • ACLsAccess Control Lists — extended permission rules attached to files that go beyond the standard owner/group/other permission model. are lost unless you use --acls
  • SELinux file contextsLabels attached to files that the SELinux mandatory access control system uses to determine what processes can access them. are lost unless you use --selinux
  • Other extended attributesArbitrary key-value metadata attached to files beyond standard permissions, used by SELinux, ACLs, and other subsystems. are lost unless you use --xattrs

For system backups on RHEL 10, always include all three flags:

tar -cf /root/etc-backup.tar --xattrs --selinux --acls /etc

Listing and Extracting Archives

Listing Archive Contents

Use the -t (list) option to display the contents of an archive without extracting anything. This is a safe, non-destructive operation that helps you verify an archive's contents before restoration.

List Archive Contents with -t
# List contents of an uncompressed archive [root@host ~]# tar -tf /root/etc.tar etc/ etc/fstab etc/crypttab etc/mtab ...output omitted... # -v gives a long listing format (like ls -l) [root@host ~]# tar -tvf /root/etc.tar drwxr-xr-x root/root 0 2025-03-01 08:00 etc/ -rw-r--r-- root/root 653 2025-03-01 07:45 etc/fstab -rw-r--r-- root/root 0 2025-01-10 12:00 etc/crypttab ...output omitted...

Notice that the paths are listed with relative syntax (etc/fstab rather than /etc/fstab) because the leading slash was stripped when the archive was created.

Extracting Archive Contents

Use the -x (extract) option to restore files from an archive. Best practice is to extract into an empty or dedicated directory to avoid overwriting existing files.

Extract to a Dedicated Directory
# Step 1: Create destination directory [root@host ~]# mkdir /root/etcbackup # Step 2: Verify contents before extracting (optional but recommended) [root@host ~]# tar -tf /root/etc.tar | head -5 etc/ etc/fstab etc/crypttab etc/mtab etc/hostname # Step 3: Change to destination and extract [root@host ~]# cd /root/etcbackup [root@host etcbackup]# tar -xf /root/etc.tar # Step 4: Verify extraction [root@host etcbackup]# ls etc

Extracting to a Specific Directory with -C

Instead of changing directories first, you can use the -C option to specify the destination directory directly. This is the preferred method in scripts.

Extract with -C (Recommended in Scripts)
# Extract directly to /root/etcbackup without cd'ing first [root@host ~]# tar -xf /root/etc.tar -C /root/etcbackup/ # Extract with verbose output to see every file [root@host ~]# tar -xvf /root/etc.tar -C /root/etcbackup/ etc/ etc/fstab etc/crypttab ...output omitted...

Preserving Permissions on Extraction

When extracting files, the system's umaskA value that masks (removes) certain permission bits from newly created files. For example, umask 022 results in files created with 644 permissions. is applied to each extracted file's permissions by default. To restore the exact original permissions instead, use the -p (preserve-permissions) option. For the root user, -p is enabled by default.

Preserve Original Permissions on Extract
# Regular user extracts with original permissions preserved [student@workstation ~/scripts]$ tar -xpf /home/student/myscripts.tar # Root user: -p is already the default behavior [root@host etcbackup]# tar -xpf /root/etc.tar -C /root/etcbackup/
💡Tip: Ownership on Extraction

When the root user extracts an archive, extracted files preserve the original user and group ownership from the archive. When a regular user extracts an archive, that user becomes the owner of all extracted files, regardless of what ownership is stored in the archive. This is a security feature preventing non-root users from creating files owned by other users.

Extracting Only Specific Files

You can extract individual files from an archive by appending the member names to the extract command. Use the exact path as stored in the archive (as shown by -t).

Extract Specific Files from an Archive
# Extract only etc/hosts from the archive [root@host ~]# tar -xf /root/etc.tar etc/hosts # Extract multiple specific files [root@host ~]# tar -xf /root/etc.tar etc/hosts etc/resolv.conf # Extract using a wildcard pattern (note: requires --wildcards) [root@host ~]# tar -xf /root/etc.tar --wildcards 'etc/ssh/*'

Compression Algorithms

The tar command supports three primary compression methods. Each makes a different trade-off between compression speed (how fast the archive is created) and compression ratio (how small the resulting file is). Understanding these trade-offs helps you choose the right algorithm for each use case.

📘When Compression Doesn't Help

The effectiveness of any compression algorithm depends entirely on the type of data being compressed. Already-compressed files — such as JPEG images, MP3 audio, RPM packages, or ZIP files — typically see little to no additional size reduction, regardless of which algorithm you choose. Attempting to compress them may even increase file size slightly. Plain text files, source code, and log files, by contrast, compress very well.

Creating Compressed Archives

Compressed Archive Creation Examples
# gzip compression — fastest, use for quick backups or interactive use [root@host ~]# tar -czf /root/etcbackup.tar.gz /etc tar: Removing leading '/' from member names # bzip2 compression — smaller files, moderate speed [root@host ~]# tar -cjf /root/logbackup.tar.bz2 /var/log tar: Removing leading '/' from member names # xz compression — best ratio, use for archival/distribution [root@host ~]# tar -cJf /root/sshconfig.tar.xz /etc/ssh tar: Removing leading '/' from member names # Compare resulting file sizes [root@host ~]# ls -lh /root/*.tar.* -rw-r--r--. 1 root root 3.1M /root/etcbackup.tar.gz -rw-r--r--. 1 root root 2.6M /root/logbackup.tar.bz2 -rw-r--r--. 1 root root 1.8M /root/sshconfig.tar.xz

Listing Compressed Archive Contents

After creating a compressed archive, you can still use tar -tf to list its contents. You do not need to specify the compression algorithm when listing — tar reads the file headerThe first few bytes of a file that identify its format (also called "magic bytes"). The tar command reads these to auto-detect compression type. to auto-detect the compression type.

List Compressed Archive — No Compression Flag Needed
# tar auto-detects compression from the file header [root@host ~]# tar -tf /root/etcbackup.tar.gz etc/ etc/fstab etc/crypttab etc/mtab ...output omitted... # Same command works identically for bzip2 and xz [root@host ~]# tar -tf /root/sshconfig.tar.xz etc/ssh/ etc/ssh/ssh_config etc/ssh/sshd_config ...output omitted...

Extracting Compressed Archives

Similarly, extracting compressed archives requires no compression flag. The tar -xf command auto-detects the compression algorithm from the file header. If you specify an incorrect compression flag, tar reports an error.

Extract Compressed Archives
# Correct: auto-detect compression (no flag needed) [root@host ~]# tar -xf /root/etcbackup.tar.gz -C /root/etcbackup/ # ERROR: Specifying wrong compression type causes failure [root@host ~]# tar -xzf /root/etcbackup.tar.xz gzip: stdin: not in gzip format tar: Child returned status 1 tar: Error is not recoverable: exiting now # Best practice: always use -xf without compression flag [root@host ~]# tar -xvf /root/etcbackup.tar.gz -C /root/etcbackup/ etc/ etc/fstab ...output omitted...

Standalone Compression Tools

The gzip, bzip2, and xz algorithms are also available as standalone commands that compress or decompress individual files without creating a tar archive. These tools are useful when you need to compress a single large file, such as a database dump or a single log file.

🔴Important: Standalone Tools Cannot Archive Directories

The standalone compression commands (gzip, bzip2, xz) can only compress individual files. They cannot create a single compressed file from a directory containing multiple files. To compress an entire directory, you must first use tar to bundle it into a single archive file, then (optionally) compress that archive.

Compressing Individual Files

Standalone gzip, bzip2, and xz Usage
# gzip: compresses file.log in place, creating file.log.gz [student@workstation ~]$ gzip largefile.log # Original file is replaced by largefile.log.gz # bzip2: creates largefile.log.bz2, removes original [student@workstation ~]$ bzip2 largefile.log # xz: creates largefile.log.xz, removes original [student@workstation ~]$ xz largefile.log # Use -k (keep) to retain the original file [student@workstation ~]$ gzip -k largefile.log [student@workstation ~]$ ls largefile.log largefile.log.gz

Decompressing Files

Each compression tool has a corresponding decompression command. You can also use the -d (decompress) flag with the original tool.

Decompression Commands
# gunzip decompresses a .gz file (equivalent to gzip -d) [student@workstation ~]$ gunzip largefile.log.gz # bunzip2 decompresses a .bz2 file (equivalent to bzip2 -d) [student@workstation ~]$ bunzip2 largefile.log.bz2 # unxz decompresses a .xz file (equivalent to xz -d) [student@workstation ~]$ unxz largefile.log.xz # Decompress a compressed tar archive without extracting (rarely needed) [student@workstation ~]$ gunzip etc-backup.tar.gz # Leaves etc-backup.tar (uncompressed, contents intact)

Checking Archive Size Before Extraction

Before extracting a large archive, it is good practice to verify the uncompressed size to ensure you have sufficient disk space. The gzip -l and xz -l options report this without extracting anything.

Check Uncompressed Size Before Extraction
# gzip -l shows compressed and uncompressed sizes [student@workstation ~]$ gzip -l file.tar.gz compressed uncompressed ratio uncompressed_name 221603125 303841280 27.1% file.tar # xz -l shows detailed compression information [student@workstation ~]$ xz -l file.tar.xz Strms Blocks Compressed Uncompressed Ratio Check Filename 1 1 195.7 MiB 289.8 MiB 0.675 CRC64 file.xz # Before extracting, verify enough free space exists [student@workstation ~]$ df -h /root Filesystem Size Used Avail Use% Mounted on /dev/sda1 50G 12G 38G 24% /
💡Tip: bzip2 -l is Not Available

Unlike gzip and xz, the bzip2 command does not have a -l (list) option to show the uncompressed size without decompressing. For bzip2 archives, you can estimate size based on typical compression ratios, or decompress to a temporary location to check.

Compression Tool Reference Summary

Algorithm Compress Decompress Extension View Size
gzip gzip file gunzip file.gz .gz gzip -l file.gz
bzip2 bzip2 file bunzip2 file.bz2 .bz2 Not available
xz xz file unxz file.xz .xz xz -l file.xz

Chapter Summary

In this chapter, you learned how to archive and compress files using the tools available in RHEL 10. These skills are foundational for system administration tasks such as backup, disaster recovery, and software packaging.

Key Concepts

  • An archive bundles multiple files and directories into a single file, preserving metadata, permissions, and directory structure.
  • The tar utility is the standard Linux tool for creating, listing, and extracting archives. It accepts both absolute and relative paths, and strips the leading / from absolute paths for safety.
  • Extraction should always be done into a dedicated empty directory to avoid overwriting live system files.
  • Extended attributes (SELinux contexts, ACLs) are not preserved by default; use --selinux, --acls, and --xattrs for complete system backups.
  • The three primary compression algorithms — gzip, bzip2, and xz — trade off speed against compression ratio. gzip is fastest; xz achieves the best ratio.
  • When extracting, no compression flag is needed — tar auto-detects the algorithm from the file header. Specifying the wrong flag causes an error.
  • Standalone tools (gzip, bzip2, xz) compress individual files and cannot bundle directories; always use tar for multi-file archives.
  • Use gzip -l or xz -l to check uncompressed size before extraction to avoid running out of disk space.
